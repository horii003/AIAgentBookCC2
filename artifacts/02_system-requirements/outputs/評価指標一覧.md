## 3. 評価の実施方法

### 3.1 開発段階における評価

| フェーズ | 実施方法 | 対象観点 |
|---------|---------|---------|
| 開発初期 | 人手評価 | 妥当性・有用性 |
| 開発中期 | 自動評価の段階的導入（MET-001, MET-003） | 妥当性・有用性 |
| リリース前 | 全テストケースの自動実行（MET-001, MET-003） | 妥当性・有用性（必須ゲート） |

### 3.2 回帰評価

| トリガー | 実施内容 | 合否基準 |
|---------|---------|---------|
| プロンプト変更時 | 全テストケースを再実行 | MET-001・MET-003の閾値を満たすこと |
| ナレッジ変更時（DATA-001更新時） | 全テストケースを再実行 | MET-001・MET-003の閾値を満たすこと |
| モデルバージョン変更時 | 全テストケースを再実行 | MET-001・MET-003の閾値を満たすこと |

### 3.3 運用段階における継続的評価

- リリース後も品質は変動し続けるため、評価を運用サイクルに組み込む
- MET-003（業務ゴール達成率）を運用中も定期的に評価する
- 定期的（要件上未定義の頻度）に全テストケースの回帰評価を実施する

# 評価指標定義書

## 評価指標ID：MET-001
## 評価指標名：申請種別判断正確率

### 1. 基本情報

| 項目 | 内容 |
|---|---|
| 評価観点 | 妥当性 |
| 対象エージェント | AG-001（申請受付窓口） |
| 対象会話フロー | CF-001 |

### 2. 指標の目的・意味

- この指標で **何を良し悪しと判断したいか**：社員の申請内容テキストから正しい申請種別（交通費精算申請 または 経費精算申請）を判断できているか（BRL-01・JD-01の遵守）
- 業務ゴールとの関係：申請種別判断の誤りはその後のフロー全体（情報収集・申請書生成）に波及するため、最優先で高い精度が必要
- 改善アクションにどう使うか：スコア低下時はAG-001のシステムプロンプトまたはKN-001の申請種別判定ルールを見直す

### 3. 指標定義

| 項目 | 内容 |
|---|---|
| 指標種別 | LLM-as-Judge（二値） |
| 定義式 | 正しく判断したケース数 ÷ 全テストケース数 |
| 評価単位 | テストケース |
| スコア範囲 | 0.0〜1.0 |
| 高いほど良い | Yes |

### 4. 評価方法・測定手順

| 項目 | 内容 |
|---|---|
| 評価方法 | テスト入力（申請内容テキスト）に対するAG-001の申請種別判断結果を、期待値（正解ラベル）とLLM-as-Judgeで二値（達成/未達成）で評価する |
| 実施手順 | 1. 交通費精算申請・経費精算申請・判断不能ケースのテストデータを準備する；2. AG-001に入力し申請種別判断結果（D-003）を取得する；3. LLM-as-Judgeで判断結果と正解ラベルを比較して二値評価する；4. 全テストケースの達成率を算出する |
| 評価者 | システム（LLM）|

### 5. 合格基準・閾値

| 項目 | 内容 |
|---|---|
| 目標値 | 0.97以上 |
| 最低許容値 | 0.95未満の場合は不合格 |
| 不合格時対応 | AG-001のシステムプロンプトまたはKN-001（申請種別判定ルール）を見直す |

### 6. 評価頻度

| フェーズ | 実施有無 |
|---|---|
| 開発中 | ○ |
| リリース前 | ○（必須ゲート） |
| 運用中 | △（モデル変更時のみ） |

## 評価指標ID：MET-003
## 評価指標名：業務ゴール達成率（申請書作成完了率）

### 1. 基本情報

| 項目 | 内容 |
|---|---|
| 評価観点 | 有用性 |
| 対象エージェント | AG-001（申請受付窓口）, AG-002（交通費精算申請）, AG-003（経費精算申請） |
| 対象会話フロー | CF-001〜CF-005（申請フロー全体） |

### 2. 指標の目的・意味

- この指標で **何を良し悪しと判断したいか**：社員が申請フロー（申請種別判断→情報収集→HitL承認→申請書Excel生成→チェック）を完了し、申請書ドラフトと事前チェック結果を受け取れたか
- 業務ゴールとの関係：このシステムの核心的な業務ゴール（申請書作成支援）の達成を測る最重要指標
- 改善アクションにどう使うか：スコア低下時は低下したフローステップ（CF-001〜CF-005）を特定し、該当エージェントのプロンプトや業務ルール定義を見直す

### 3. 指標定義

| 項目 | 内容 |
|---|---|
| 指標種別 | LLM-as-Judge（二値） |
| 定義式 | 申請書ドラフト（OUT-003）生成かつチェック結果（OUT-004）が提示されたセッション数 ÷ 全テストセッション数 |
| 評価単位 | セッション |
| スコア範囲 | 0.0〜1.0 |
| 高いほど良い | Yes |

### 4. 評価方法・測定手順

| 項目 | 内容 |
|---|---|
| 評価方法 | エンドツーエンドのマルチターン会話シナリオ（交通費精算申請・経費精算申請）を実行し、申請書ドラフトとチェック結果が正しく生成されたかLLM-as-Judgeで評価する |
| 実施手順 | 1. 代表的な申請シナリオ（交通費・経費・判断不能含む）のテストケースを準備する；2. 申請フロー全体（CF-001〜CF-005）をマルチターンで実行する；3. OUT-003（申請書ドラフト・Excelファイル）とOUT-004（チェック結果）の生成有無をLLM-as-Judgeで確認する；4. 全テストセッションの達成率を算出する |
| 評価者 | システム（LLM） |

### 5. 合格基準・閾値

| 項目 | 内容 |
|---|---|
| 目標値 | 0.93以上 |
| 最低許容値 | 0.90未満の場合は不合格 |
| 不合格時対応 | 失敗したフローステップを特定し、該当エージェントのプロンプトまたは業務ルール定義を見直す |

### 6. 評価頻度

| フェーズ | 実施有無 |
|---|---|
| 開発中 | ○ |
| リリース前 | ○（必須ゲート） |
| 運用中 | ○（定期評価） |
