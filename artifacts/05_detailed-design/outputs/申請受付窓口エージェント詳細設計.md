## 1. 概要

### 1.1 コンポーネントの目的

社員が入力した申請者名と申請内容テキストから申請種別（交通費精算申請 / 経費精算申請）を判断し、申請書名・申請先・判断根拠を社員に提示した後、専門エージェント（AG-002 または AG-003）へ申請フロー全体を委譲する。システム全体のエントリーポイントとして機能する。

### 1.2 主要な責務

1. **申請者名収集**: 社員から申請者名を受け取り、バリデーション（空入力・500文字超）を適用する
2. **申請内容テキスト収集**: 社員から申請内容テキストを受け取り、バリデーション（空入力・500文字超・30回上限）を適用する
3. **申請種別判断**: 社内申請ルール（DATA-001）を参照して LLM が申請種別を判断する（JD-01/JD-02）
4. **申請種別案内情報提示**: 申請種別・申請書名・申請先・判断根拠を社員に提示する（BRL-02準拠）
5. **専門エージェントへの委譲**: 申請種別確定後に AG-002/AG-003 へ申請フロー全体を委譲する（Agent As Tools方式）

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- なし（申請日は AG-001 が `datetime.date.today()` で自動取得する）

**対話ループでの入力**:
- 申請者名（必須・500文字以内・空不可）
- 申請内容テキスト（必須・500文字以内・空不可）
- 特殊コマンド:
  - `reset`, `リセット`, `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る

**パラメータ**:
- `query` (str): ユーザーからの申請者名・申請内容テキスト

**ToolContext経由で受け取る情報**:
- AG-001 はオーケストレーターのため ToolContext を受け取らない（エントリーポイント）

#### 2.2.2 出力

**出力先**: 社員（コンソール / CLI）

**出力メッセージ**:
- 起動時ウェルカムメッセージ:
  ```
  ============================================================
  こちらは申請受付窓口エージェントです
  社内の様々な申請作業をサポートします

  最初に申請者名を入力してください。その後、申請したい内容をお知らせください。キーワードでも構いません

  ※終了するには 'exit' または 'quit' と入力ください
  ※最初からやり直すには 'reset' と入力ください
  ============================================================
  ```
- 申請種別案内: `"申請種別：[申請種別名]、申請書：[申請書名]、申請先：[申請先]、判断根拠：[参照ルール名]（BRL-02準拠）"`
- 判断不能時選択肢提示: `"申請種別を判断できませんでした。以下からお選びください：1. 交通費精算申請 2. 経費精算申請"`
- 入力エラー（EX-01）: `"入力が長すぎます。500文字以内で入力してください"` 等

**戻り値の形式**: `str`（エージェントからの応答）

**出力内容**:
- 申請種別案内情報（D-004）：申請種別名・申請書名・申請先・判断根拠
- 専門エージェントからの最終応答（申請書ドラフトパス・チェック結果）
- エラーメッセージ

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要（AG-001 の処理フローに日付・期限を基準とした判断処理が含まれないため） |

##### プロンプト全文

```
あなたは「申請受付窓口エージェント」です。社員の申請種別を判断し、適切な専門エージェントに委譲することが主な役割です。

【役割】
1. 申請者名の受取・バリデーション
   - 申請者名を受け取り、空入力・500文字超の場合は再入力を促す（EX-01）

2. 申請内容テキストの受取・バリデーション
   - 社員から申請内容を受け取り、空入力・500文字超の場合は再入力を促す（EX-01）

3. 申請種別の判断（JD-01）
   - data/rules/ 配下の社内申請ルール（DATA-001）を参照し、申請内容テキストから申請種別を判断する
   - 「交通費精算申請」または「経費精算申請」に分類する

4. 申請種別案内情報の提示（BRL-02）
   - 申請種別名・申請書名・申請先・判断根拠（参照ルール名）を社員に提示する

5. 判断不能時の選択肢提示（JD-02）
   - 判断できない場合は「1. 交通費精算申請 2. 経費精算申請」の選択肢を提示する（BRL-06）

6. 専門エージェントへの委譲
   - 交通費精算申請: handle_transport_expense_application ツールを呼び出す
   - 経費精算申請: handle_general_expense_application ツールを呼び出す

【制約事項】
- 対話回数が30回を超えた場合はセッションを終了する（FR-012）
- 申請種別の判断・提示のみを担当する。情報収集・計算・生成・チェックは専門エージェントの責務
- 委譲後は申請フロー内部に介入しない

【処理フロー（CF-001）】
Step1: 「お名前をお聞かせください」と申請者名を確認する
Step2: 申請者名をバリデーションする（空・500文字超は EX-01 として再入力要求）
Step3: 「どのような申請をご希望ですか？」と申請内容テキストを受け取る
Step4: 申請内容テキストをバリデーションする（空・500文字超は EX-01 として再入力要求）
Step5: 社内申請ルール（DATA-001）を参照して申請種別を判断する（JD-01）
Step6: 申請種別案内情報を提示する（BRL-02：申請種別名・申請書名・申請先・判断根拠）
Step7: 社員の確認を受け、専門エージェント（AG-002/AG-003）に委譲する

【特殊コマンド】
- `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 役割定義 | エントリーポイントとして申請種別判断のみを担い、専門エージェントへ完全委譲する責務を明確にする |
| JD-01/JD-02 | LLM による intent 分類とルール参照を組み合わせて確実に分類し、判断不能時は社員選択にフォールバックする |
| BRL-02（根拠提示） | 判断根拠を常に提示することで社員がフローを信頼できるようにする |
| 処理フロー | CF-001 に準拠した逐次処理を保証する |

#### 2.3.2 申請種別振り分けルール

| ルール | 条件 | 対応 |
|--------|------|------|
| JD-01（交通費判断） | 申請内容テキストが交通費精算申請（電車代・タクシー代・バス代・飛行機代等）に合致 | handle_transport_expense_application ツールを呼び出す |
| JD-01（経費判断） | 申請内容テキストが経費精算申請（事務用品費・宿泊費・資格精算費等）に合致 | handle_general_expense_application ツールを呼び出す |
| JD-02（判断不能） | いずれにも分類できない | 「1. 交通費精算申請 2. 経費精算申請」の選択肢を提示する（BRL-06/CF-004） |

#### 2.3.3 処理フロー

```
開始
  ↓
申請者名の入力受取
  - 空入力・500文字超 → EX-01（再入力要求ループ）
  ↓
申請内容テキストの入力受取
  - 空入力・500文字超 → EX-01（再入力要求ループ）
  ↓
社内申請ルール（DATA-001）を参照して申請種別を判断（JD-01）
  ↓
判断結果
  - 判断可能 → 申請種別案内情報を提示（BRL-02）→ 社員確認へ
  - 判断不能 → 選択肢提示（JD-02/BRL-06/CF-004）→ 社員の選択を受け取る
  ↓
社員確認（Step7）
  ↓
委譲
  - 交通費精算申請 → handle_transport_expense_application を呼び出す
  - 経費精算申請 → handle_general_expense_application を呼び出す
  ↓
専門エージェントからの応答を社員へ返す
  ↓
終了
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindow(30) | オーケストレーターとして申請受付・種別判断・委譲を担当し会話コンテキストを十分に保持するため |
| リトライ戦略 | 標準設定（Bedrock APIのboto3組み込みリトライ） | 最大6回/初期遅延4秒/最大遅延240秒/指数バックオフ |
| フック | LoopControlHook(max_iterations=10) | 全エージェント共通。HumanApprovalHookは不要（TOOL-002を呼び出さないため） |
| セッション管理 | SessionManager | セッション状態（DATA-005）をファイル永続化してresume可能にするため |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| - | `handle_transport_expense_application` | AG-002への委譲（交通費精算申請フロー全体） | 申請種別が「交通費精算申請」に確定した場合 |
| - | `handle_general_expense_application` | AG-003への委譲（経費精算申請フロー全体） | 申請種別が「経費精算申請」に確定した場合 |

> AG-001 が呼び出すツールは機能ツール一覧（TOOL-XXX）ではなく、エージェント間連携設計（SYS-MA-001）に基づくエージェント委譲ツールである。

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

AG-001 は AG-002/AG-003 を Agent As Tools として登録して呼び出す。

**handle_transport_expense_application ツール説明**:
```
交通費精算申請エージェント（AG-002）への委譲ツール。
交通費精算申請フロー全体（情報収集・交通費計算・HitL承認・申請書生成・申請書チェック）を
自己完結で実行する。
```

**handle_general_expense_application ツール説明**:
```
経費精算申請エージェント（AG-003）への委譲ツール。
経費精算申請フロー全体（情報収集・領収書解析・HitL承認・申請書生成・申請書チェック）を
自己完結で実行する。
```

**ToolContextの使用**:
- AG-001 は ToolContext を通じて専門エージェントに invocation_state を渡す
- `@tool(context=True)` デコレータは AG-002/AG-003 側で定義される

#### 2.5.2 invocation_stateの受け渡し

```python
# AG-001 から AG-002/AG-003 へ渡す invocation_state
invocation_state = {
    "applicant_name": applicant_name,      # str: CF-001 Step1 で収集した申請者名
    "application_date": application_date,  # str: システム日付（YYYY-MM-DD形式）
    "session_id": session_id               # str: セッションID（タイムスタンプ_UUID8桁形式: 例 20260323153045_a1b2c3d4）
}
# Agent As Tools として登録された AG-002/AG-003 呼び出し時に自動的に渡される
```

#### 2.5.3 AG-002/AG-003への振り分けロジック

**振り分け判断基準**:

| 連携先 | キーワード例 | 特徴 |
|--------|-----------|------|
| AG-002（交通費精算） | 電車代・タクシー代・バス代・飛行機代・交通費 | 移動に関する経費精算 |
| AG-003（経費精算） | 事務用品費・宿泊費・資格精算費・その他経費 | 物品・サービスに関する経費精算 |

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 create_reception_agent関数

##### 説明
AG-001（申請受付窓口エージェント）インスタンスを生成して返す関数。

##### 引数
- `bedrock_model` (BedrockModel): Amazon Bedrock モデルインスタンス（必須）
- `transport_tool` (callable): AG-002 の `handle_transport_expense_application` ツール関数（必須）
- `expense_tool` (callable): AG-003 の `handle_general_expense_application` ツール関数（必須）

##### 戻り値
- `Agent`: 設定済みの申請受付窓口エージェントインスタンス

##### 処理内容
1. `RECEPTION_AGENT_SYSTEM_PROMPT`（静的定数）を参照する
2. `SlidingWindowConversationManager(window_size=30)` を生成する
3. `LoopControlHook(max_iterations=10)` を生成する
4. `Agent(model=bedrock_model, system_prompt=RECEPTION_AGENT_SYSTEM_PROMPT, tools=[transport_tool, expense_tool], conversation_manager=..., callback_handler=...)` を生成して返す

#### 3.1.2 reception_agent_loop関数

##### 説明
AG-001 の対話ループを実行する関数。セッション開始からフロー完了まで管理する。

##### 引数
- `agent` (Agent): `create_reception_agent` で生成したエージェントインスタンス（必須）
- `session_manager` (SessionManager): セッションマネージャインスタンス（必須）

##### 戻り値
- `None`

##### 処理内容
1. セッション状態を CREATED に設定する
2. 対話ループ開始前にウェルカムメッセージを `print` で出力する
3. 対話ループを開始する
4. ユーザー入力を `input("\n\n入力内容（終了時はquit）: ")` で受け取る
5. `agent(query)` を呼び出す
6. エージェントからの応答を表示する
7. 例外発生時はエラーハンドリングを実施する
8. セッション完了時はループを終了する

##### 例外処理

| 例外クラス | ログレベル | ログ付加情報 | ループ制御 | 処理内容 |
|-----------|----------|-----------|---------|--------|
| KeyboardInterrupt | INFO | session_id | break（ループ終了） | `ErrorHandler.handle_keyboard_interrupt(e)` の戻り値を `print` で表示 |
| LoopLimitError | WARNING | session_id | continue（ループ継続） | `ErrorHandler.handle_loop_limit_error(e)` の戻り値を `print` で表示 |
| ContextWindowOverflowException | WARNING | session_id | continue | `ErrorHandler.handle_context_window_error(e)` の戻り値を `print` で表示 |
| MaxTokensReachedException | WARNING | session_id | continue | `ErrorHandler.handle_max_tokens_error(e)` の戻り値を `print` で表示 |
| RuntimeError | ERROR (exc_info=True) | session_id | continue | `ErrorHandler.handle_runtime_error(e)` の戻り値を `print` で表示 |
| Exception | ERROR (exc_info=True) | session_id | continue | `ErrorHandler.handle_unexpected_error(e)` の戻り値を `print` で表示 |

- 戻り値なし（対話ループのため不要）
- ログにはセッションIDを付加情報として含める: `extra={"session_id": session_id}`
- `ErrorHandler` のメソッドの戻り値（str）を `print` で表示する
- ログ出力は `ErrorHandler` 呼び出し前に行う

### 3.2 エラーハンドリング

| 例外クラス | 条件 | ループ制御 | 対応 |
|-----------|------|---------|------|
| KeyboardInterrupt | Ctrl+C による割り込み | break（ループ終了） | `ErrorHandler.handle_keyboard_interrupt(e)` の戻り値を `print` で表示 |
| LoopLimitError | LoopControlHook によるループ上限超過 | continue（ループ継続） | `ErrorHandler.handle_loop_limit_error(e)` の戻り値を `print` で表示 |
| ContextWindowOverflowException | コンテキストウィンドウ超過 | continue | `ErrorHandler.handle_context_window_error(e)` の戻り値を `print` で表示 |
| MaxTokensReachedException | 最大トークン数超過 | continue | `ErrorHandler.handle_max_tokens_error(e)` の戻り値を `print` で表示 |
| RuntimeError | その他の RuntimeError | continue | `ErrorHandler.handle_runtime_error(e)` の戻り値を `print` で表示 |
| Exception | 予期しないエラー | continue | `ErrorHandler.handle_unexpected_error(e)` の戻り値を `print` で表示 |

| WARNING | LoopLimitError 発生時 | `"reception_agent: LoopLimitError 発生"` | `extra={"session_id": session_id}` |
| WARNING | ContextWindowOverflowException 発生時 | `"reception_agent: ContextWindowOverflowException 発生"` | `extra={"session_id": session_id}` |
| WARNING | MaxTokensReachedException 発生時 | `"reception_agent: MaxTokensReachedException 発生"` | `extra={"session_id": session_id}` |
| ERROR | RuntimeError 発生時 | `"reception_agent: RuntimeError 発生"` | `exc_info=True`, `extra={"session_id": session_id}` |
| ERROR | 予期しない例外発生時 | `"reception_agent: 予期しない例外発生"` | `exc_info=True`, `extra={"session_id": session_id}` |

## 4. データ設計

### 4.1 invocation_state

AG-001 はオーケストレーターのため invocation_state を受け取らない。

**AG-002/AG-003 に渡すデータ**:
```python
{
    "applicant_name": str,       # 申請者名（CF-001 Step1 で収集）
    "application_date": str,     # 申請日（システム日付: YYYY-MM-DD形式）
    "session_id": str            # セッションID（タイムスタンプ_UUID8桁形式: 例 20260323153045_a1b2c3d4、セッション開始時に AG-001 が生成）
}
```

## 5. 補足情報

### 5.1 実装上の注意点

1. **静的システムプロンプトの管理**
   - `RECEPTION_AGENT_SYSTEM_PROMPT` は `agents/reception_agent.py` 内でモジュールレベル定数として定義する

2. **AG-002/AG-003 の Agent As Tools 登録順序**
   - `tools=[transport_tool, expense_tool]` として登録する
   - AG-001 の `Agent` インスタンス生成前に AG-002/AG-003 の `@tool(context=True)` デコレータ付き関数を定義しておく必要がある

3. **session_id の生成**
   - タイムスタンプ（秒単位）+ UUID（8文字）形式で生成する: `datetime.now().strftime("%Y%m%d%H%M%S") + "_" + uuid.uuid4().hex[:8]`
   - 生成例: `20260323153045_a1b2c3d4`
   - セッション開始時に AG-001 が一度だけ生成し、invocation_state に含めて専門エージェントに渡す

4. **委譲後の非干渉**
   - AG-001 は AG-002/AG-003 の呼び出し後に結果を受け取り社員に提示するが、申請フロー内部には介入しない

---

### 5.2 パフォーマンス考慮事項

1. **DATA-001 の参照**
   - 申請種別判断はシステムプロンプトにルール参照指示を含めることで LLM が DATA-001 を参照する
   - ファイルの事前キャッシュは不要

---

### 5.3 セキュリティ考慮事項

1. **入力バリデーション**
   - 申請者名・申請内容テキストの 500 文字上限チェックにより過大な入力を防止する

2. **session_id の一意性**
   - UUID 生成により衝突確率を実質的にゼロにする
