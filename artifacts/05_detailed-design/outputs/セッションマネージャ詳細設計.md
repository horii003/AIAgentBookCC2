## 1. 概要

### 1.1 コンポーネントの目的

AG-001/AG-002/AG-003 全エージェントの申請フロー全体を通じた状態（DATA-005）を `data/sessions/` ディレクトリにJSON形式で永続化・管理するコンポーネント。セッション生成・状態更新・再開・対話回数管理のAPIを提供し、マルチエージェント連携での状態一貫性を保証する。Strands Agents フレームワークの `SlidingWindowConversationManager`（会話履歴管理）とは独立して動作する。

### 1.2 主要な責務

1. **セッション生成**: 新規セッションを生成し、session_id（`sess_UUID` 形式）を発行する。初期状態（CREATED）のセッション状態ファイル（DATA-005）を `data/sessions/` に作成する
2. **セッション状態の読み書き**: 全エージェントからセッション状態（ステータス・対話回数・業務コンテキスト・HitL承認結果等）の読み込みと書き込みAPIを提供する
3. **対話回数管理**: 対話ターンごとに `iteration` カウンタをインクリメントし、30回到達時に TERMINATED 遷移を実施する（FR-012）
4. **セッションライフサイクル管理**: CREATED → ACTIVE → WAITING → CLOSED/TERMINATED の状態遷移を制御する（SYS-SESSION-001 3節準拠）
5. **セッション再開（resume）**: セッション状態ファイルが存在し状態が ACTIVE または WAITING の場合に前回の中断箇所から再開できる情報を復元する

## 4. セッションライフサイクル設計

### 4.1 セッション生成

```
セッション生成要求（applicant_name, application_date）
  ↓
data/sessions/ ディレクトリの存在確認
  - 存在しない → os.makedirs(exist_ok=True) で自動作成
  ↓
初期 SessionState の構築
  - session_id: コンストラクタで設定済み
  - status: "CREATED"
  - iteration: 0
  - current_step: "CF-001-Step1"
  - application_type: None
  - hitl_result: None
  ↓
SessionState.model_dump_json() でシリアライズ
  ↓
session_file_path に UTF-8 で書き込む
  ↓
session_id を返却
```

### 4.2 セッション保存（更新）

```
セッション保存要求（field, value, agent_id）
  ↓
書き込み権限の検証（agent_id × field の組み合わせ確認）
  - 権限なし → ValueError（ErrorHandler.handle_system_failure に委譲）
  ↓
session_file_path からセッション状態を読み込む（load_session）
  ↓
指定フィールドを更新する
  ↓
更新後の SessionState をシリアライズ
  ↓
session_file_path に UTF-8 で書き込む
  - ファイルパス: data/sessions/{session_id}.json
  ↓
保存完了
```

### 4.3 セッション復元

```
セッション復元要求（resume_session）
  ↓
session_file_path の存在確認
  - 存在しない → None を返す（新規セッション開始を促す）
  ↓
ファイルを UTF-8 で読み込む
  ↓
SessionState.model_validate_json() でデシリアライズ
  - ValidationError → ErrorHandler.handle_system_failure に委譲、None を返す
  ↓
status の確認
  - "ACTIVE" または "WAITING" → SessionState を返す（再開可能）
  - "CLOSED" または "TERMINATED" → None を返す（再開不可）
  - "CREATED" → None を返す（未着手のため新規扱い）
  ↓
セッション復元完了
```

### 4.4 セッション削除

```
セッション削除要求（delete_session）
  ↓
session_file_path の存在確認
  ↓
存在する → os.remove(session_file_path) でファイル削除
  ↓
削除完了（ファイルが存在しない場合は正常完了として扱う）
```

## 5. 永続化設計

### 5.1 ファイルパス生成ルール

| 項目 | 値 |
|------|-----|
| 基底ディレクトリ | `data/sessions/`（コンストラクタの `storage_path` で設定） |
| ファイル名規則 | `{session_id}.json`（例: `sess_a1b2c3d4-e5f6-7890-abcd-ef1234567890.json`） |
| ファイル形式 | JSON |
| エンコード | UTF-8 |
| 自動作成 | `create_session` 時に `os.makedirs(storage_path, exist_ok=True)` で自動作成 |

### 5.2 永続化データ構造

```json
{
  "session_id": "sess_a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "ACTIVE",
  "iteration": 5,
  "current_step": "CF-002-Step4",
  "application_type": "交通費精算申請",
  "hitl_result": null
}
```

> `ApplicationContext`（申請者名・申請日・申請種別案内情報・収集済み申請情報等）は `SessionState` とは別に `application_context_{session_id}.json` としてではなく、同一 `{session_id}.json` ファイルの `application_context` フィールドに JSON 辞書として保存する。

## 6. Strands SDK連携設計

### 6.1 ConversationManager設定

`SessionManager` 自体は Strands SDK の `ConversationManager` を使用しない。会話履歴管理は各エージェントが独立して `SlidingWindowConversationManager` で行う。`SessionManager` は業務コンテキスト・セッション状態のみを管理する（SYS-SESSION-001 9.1節の役割分担に準拠）。

| エージェント種別 | クラス | 管理範囲 | window_size |
|--------------|-------|--------|------------|
| AG-001（オーケストレーター） | `SlidingWindowConversationManager` | 会話履歴のみ（SessionManager 非担当） | 30 |
| AG-002（交通費精算申請エージェント） | `SlidingWindowConversationManager` | 会話履歴のみ（SessionManager 非担当） | 20 |
| AG-003（経費精算申請エージェント） | `SlidingWindowConversationManager` | 会話履歴のみ（SessionManager 非担当） | 15 |

### 6.2 エージェントへの渡し方

```python
from strands.agent.conversation_manager import SlidingWindowConversationManager
from strands import Agent
from session.session_manager import SessionManager

# AG-001: セッション生成
session_manager = SessionManager(session_id=session_id, storage_path="data/sessions/")
session_manager.create_session(applicant_name=applicant_name, application_date=str(application_date))

# AG-002: セッション再開・利用（ToolContext から session_id を取得後）
session_manager = SessionManager(session_id=session_id, storage_path="data/sessions/")
existing_session = session_manager.resume_session()

# SlidingWindowConversationManager は SessionManager と独立して設定する
conversation_manager = SlidingWindowConversationManager(window_size=20)
agent = Agent(
    model=bedrock_model,
    system_prompt=system_prompt,
    tools=[...],
    conversation_manager=conversation_manager,
    # hooks=[...] は SessionManager とは別に設定
)
```

## 7. 主要メソッド設計

### 7.1 create_session

| 項目 | 値 |
|------|-----|
| 説明 | 新規セッションを生成し、初期状態（CREATED）のセッション状態ファイル（DATA-005）を作成する |
| 引数 | `applicant_name: str`（申請者名）、`application_date: str`（申請日 YYYY-MM-DD形式） |
| 戻り値 | `str`（発行した session_id） |

**処理フロー:**
```
storage_path ディレクトリを os.makedirs(exist_ok=True) で自動作成
  ↓
初期 SessionState を生成（status="CREATED", iteration=0）
  ↓
applicant_name と application_date を ApplicationContext として構築
  ↓
session_file_path に SessionState.model_dump_json() を UTF-8 で書き込む
  ↓
session_id を返す
```

**例外:**

| 例外クラス | 発生条件 |
|----------|---------|
| `OSError` | ディレクトリ作成失敗・書き込み権限なし（EX-07として ErrorHandler に委譲） |

---

### 7.2 load_session

| 項目 | 値 |
|------|-----|
| 説明 | セッション状態ファイルを読み込み、SessionState オブジェクトを返す |
| 引数 | なし（インスタンスの `session_file_path` を使用） |
| 戻り値 | `Optional[SessionState]`（ファイルが存在しない場合は `None`） |

**処理フロー:**
```
session_file_path の存在確認（os.path.exists）
  ↓
存在しない → None を返す
  ↓
存在する → ファイルを UTF-8 で読み込む
  ↓
SessionState.model_validate_json(content) でデシリアライズ
  ↓
SessionState を返す
```

**例外:**

| 例外クラス | 発生条件 |
|----------|---------|
| `json.JSONDecodeError` | ファイル内容が不正な JSON（EX-07として ErrorHandler に委譲） |
| `ValidationError` | SessionState の Pydantic バリデーション失敗（EX-07として ErrorHandler に委譲） |
| `IOError` | ファイル読み込みエラー（EX-07として ErrorHandler に委譲） |

---

### 7.3 update_session

| 項目 | 値 |
|------|-----|
| 説明 | セッション状態の特定フィールドを更新し、セッション状態ファイルに書き込む。エージェントIDに基づく書き込み権限を検証する |
| 引数 | `field: str`（更新フィールド名）、`value: Any`（更新値）、`agent_id: str`（更新元エージェントID） |
| 戻り値 | `None` |

**処理フロー:**
```
書き込み権限マッピングの確認（AGENT_FIELD_PERMISSIONS）
  - AG-001 の権限フィールド: ["application_type", "current_step"]
  - AG-002 の権限フィールド: ["hitl_result", "current_step", "iteration"]
  - AG-003 の権限フィールド: ["hitl_result", "current_step", "iteration"]
  - セッションマネージャ: ["status", "iteration", "current_step"]
  ↓
権限なし → ValueError（ErrorHandler.handle_system_failure に委譲）
  ↓
load_session() で既存セッション状態を読み込む
  ↓
指定フィールドを更新する（setattr または dict 更新）
  ↓
SessionState.model_dump_json() でシリアライズ
  ↓
session_file_path に UTF-8 で書き込む
```

**例外:**

| 例外クラス | 発生条件 |
|----------|---------|
| `ValueError` | 書き込み権限なし（上書き禁止フィールドへの書き込み試行） |
| `OSError` | ファイル書き込みエラー（EX-07として ErrorHandler に委譲） |

---

### 7.4 transition_status

| 項目 | 値 |
|------|-----|
| 説明 | セッション状態を指定の遷移先ステータスに変更する。遷移可否の検証を行う |
| 引数 | `new_status: Literal["CREATED", "ACTIVE", "WAITING", "CLOSED", "TERMINATED"]` |
| 戻り値 | `None` |

**処理フロー:**
```
load_session() で現在の状態を取得する
  ↓
遷移可否の確認（VALID_TRANSITIONS マッピング）
  - CREATED → ACTIVE
  - ACTIVE → WAITING, CLOSED, TERMINATED
  - WAITING → ACTIVE, CLOSED, TERMINATED
  - CLOSED → （遷移不可）
  - TERMINATED → （遷移不可）
  ↓
遷移不可 → ValueError（ErrorHandler.handle_system_failure に委譲）
  ↓
status を new_status に更新する
  ↓
update_session("status", new_status, agent_id="session_manager") を呼び出す
```

**例外:**

| 例外クラス | 発生条件 |
|----------|---------|
| `ValueError` | 遷移不可能な状態変更（例: CLOSED → ACTIVE） |

---

### 7.5 increment_iteration

| 項目 | 値 |
|------|-----|
| 説明 | 対話回数（`iteration`）をインクリメントし、30回到達時に TERMINATED 遷移を実施する |
| 引数 | なし |
| 戻り値 | `bool`（`True`: 継続可能、`False`: 上限到達で TERMINATED に遷移済み） |

**処理フロー:**
```
load_session() で現在の iteration を取得する
  ↓
iteration += 1
  ↓
update_session("iteration", new_iteration, agent_id="session_manager") を呼び出す
  ↓
new_iteration >= 30（FR-012上限）の確認
  - YES → transition_status("TERMINATED") を実行し、False を返す
  - NO  → True を返す
```

**例外:**

| 例外クラス | 発生条件 |
|----------|---------|
| `OSError` | ファイル書き込みエラー（EX-07として ErrorHandler に委譲） |

---

### 7.6 resume_session

| 項目 | 値 |
|------|-----|
| 説明 | セッション状態ファイルからセッション情報を復元し、再開可否を判定する |
| 引数 | なし |
| 戻り値 | `Optional[SessionState]`（復元成功: SessionState、再開不可: None） |

**処理フロー:**
```
load_session() を呼び出す
  ↓
None（ファイルなし）→ None を返す（新規セッション開始）
  ↓
SessionState が返った場合: status を確認する
  - "ACTIVE" または "WAITING" → SessionState を返す（再開可能）
  - "CLOSED" または "TERMINATED" → None を返す（再開不可）
  - "CREATED" → None を返す（未着手のため新規扱い）
```

---

### 7.7 delete_session

| 項目 | 値 |
|------|-----|
| 説明 | セッション状態ファイルを削除する（セッション終了時のクリーンアップ） |
| 引数 | なし |
| 戻り値 | `None` |

**処理フロー:**
```
session_file_path の存在確認
  ↓
存在する → os.remove(session_file_path) でファイル削除
存在しない → 正常完了として扱う（冪等性を保証する）
```

## 8. エラーハンドリング

### 8.1 エラー処理フロー

```
エラー発生
  ↓
エラー種別判定
  ↓
IOError / OSError（ファイルI/O失敗）
  → EX-07 として ErrorHandler.handle_system_failure(error, context) に委譲
  → 呼び出し元にエスカレーション案内メッセージを返す（TERMINATED）
  ↓
json.JSONDecodeError / ValidationError（ファイル内容破損）
  → EX-07 として ErrorHandler.handle_system_failure(error, context) に委譲
  → None を返す（セッション再開不可として新規セッション開始を促す）
  ↓
ValueError（書き込み権限エラー・不正状態遷移）
  → EX-05 として ErrorHandler.handle_execution_blocked(action) に委譲
  → 呼び出し元にエスカレーション案内メッセージを返す（TERMINATED）
```

### 8.2 エラーメッセージ

| エラー種別 | 発生条件 | ユーザー向けメッセージ |
|----------|---------|---------------------|
| EX-07（ファイル書き込みエラー） | `create_session` / `update_session` 時の IOError | `"申請処理中にシステムエラーが発生しました。しばらく経ってから再度お試しいただくか、担当部門にお問い合わせください。"` |
| EX-07（ファイル読み込みエラー） | `load_session` 時の IOError / 破損JSON | セッション再開不可として `None` を返す（ユーザーへのメッセージは呼び出し元が生成） |
| EX-05（書き込み権限エラー） | `update_session` での上書き禁止フィールドへの書き込み | `"申請処理中にエラーが発生しました。しばらく経ってから再度お試しいただくか、担当部門にお問い合わせください。"` |

## 10. 補足情報

### 10.1 実装上の注意点

1. **原子的書き込みの推奨**
   - ファイルの部分書き込みによる破損防止のため、tempファイルへの書き込み後にリネームする方式が望ましい: `tempfile.NamedTemporaryFile` + `os.replace()`

2. **ファイルロック不要**
   - AG-002 と AG-003 は同一セッション内で排他実行のため（SYS-MA-001 6.3節）、同時書き込みは発生しない

3. **AGENT_FIELD_PERMISSIONS 定数**
   - `update_session` の権限検証用定数を `session/session_manager.py` 内にモジュールレベル定数として定義する

4. **`data/sessions/` の存在確認**
   - `create_session` 時のみ `os.makedirs(exist_ok=True)` を実行し、他のメソッドでは実行しない

5. **整合性チェック**
   - `resume_session` 時に SessionState の Pydantic バリデーション（`model_validate_json`）を実施することで、ファイル内容の整合性を確認する

  - `ErrorHandler.handle_system_failure()`: EX-07（ファイルI/Oエラー）の処理

## 12. テスト観点

### 12.1 セッションライフサイクルテスト
- `create_session` 呼び出しで `data/sessions/{session_id}.json` が初期状態（CREATED）で作成されること
- `update_session` で `application_type` フィールドが正しく更新されること
- `increment_iteration` を29回呼び出した時点で `True` が返され、iteration=29、status=ACTIVE のままであること
- `increment_iteration` を30回呼び出した時点で `False` が返され、status=TERMINATED に遷移していること
- `resume_session` で status=ACTIVE のセッションが正しく復元されること
- `transition_status("CLOSED")` でセッション状態が CLOSED に変更されること
- `delete_session` でセッション状態ファイルが削除されること

### 12.2 永続化テスト
- `create_session` 後に `load_session` を呼び出すと初期 `SessionState` が返されること
- `update_session` 後にファイルを直接読み込むと更新値が反映されていること
- テスト用 `storage_path`（temporaryディレクトリ）を指定してファイルが正しいパスに作成されること

### 12.3 異常系テスト
- `load_session` でセッション状態ファイルが存在しない場合に `None` が返されること
- `resume_session` で status=TERMINATED のセッションに対して `None` が返されること
- `update_session` で AG-002 が `application_type`（AG-001 専用フィールド）を更新しようとした場合に `ValueError` が発生すること
- 破損した JSON ファイルに対して `load_session` を呼び出した場合に `None` が返されること（EX-07処理）
- `transition_status("ACTIVE")` を CLOSED ステータスのセッションに呼び出した場合に `ValueError` が発生すること

### 12.4 境界値テスト
- `increment_iteration` を29回呼び出した直後（iteration=29）で `True` が返されること
- `increment_iteration` を30回呼び出した時点（iteration=30）で `False` が返されること（FR-012境界）

## 13. 設定値

### 13.1 セッション設定
- ストレージパス: `data/sessions/`（コンストラクタのデフォルト値）
- セッションID形式: `sess_{uuid.uuid4()}`（例: `sess_a1b2c3d4-e5f6-7890-abcd-ef1234567890`）
- 対話回数上限: `30`（FR-012準拠）
- 遷移可能状態: `{"CREATED": ["ACTIVE"], "ACTIVE": ["WAITING", "CLOSED", "TERMINATED"], "WAITING": ["ACTIVE", "CLOSED", "TERMINATED"], "CLOSED": [], "TERMINATED": []}`
