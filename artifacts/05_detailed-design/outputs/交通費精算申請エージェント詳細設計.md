## 1. 概要

### 1.1 コンポーネントの目的

AG-001 から委譲された交通費精算申請フロー全体（情報収集・交通費計算・HitL承認・申請書生成・申請書チェック・申請期限チェック）を自己完結で実行する。AG-001 から渡された invocation_state（applicant_name, application_date）をもとに動的システムプロンプトを生成し、専門エージェントとして機能する。

### 1.2 主要な責務

1. **申請情報収集**: CF-002 の会話ステップに従って移動区間情報を一括確認・追加収集する（BRL-07/BRL-03）
2. **交通費自動計算**: TOOL-001 で各移動区間の交通費を計算する（BRL-08/BRL-10）
3. **上長承認要否判断**: 交通費合計が10,000円を超える場合に上長承認が必要な旨を通知する（BRL-09/JD-08）
4. **申請期限チェック**: 移動日から3ヶ月以内かどうかをチェックし、超過時は社員に通知する（BRL-14/JD-07）
5. **HitL承認**: Excel生成前に申請情報サマリーを提示し社員の承認（OK/修正/キャンセル）を確認する（GRD-009）
6. **申請書ドラフト生成**: HitL承認（OK）後に TOOL-002 で Excel 申請書ドラフトを生成する（BRL-04）
7. **申請書チェック**: 申請ルール（DATA-001）に基づいて申請書の不備を確認し結果を社員に提示する（BRL-05/JD-06）

### 2.2 インターフェース設計

#### 2.2.1 入力

**初期化時の入力**:
- `applicant_name` (str): 申請者名（invocation_state から取得）
- `application_date` (str): 申請日（invocation_state から取得、YYYY-MM-DD形式）

**対話ループでの入力**:
- 移動区間情報（移動日・出発地・目的地・交通手段・業務目的）（自然文）
- HitL承認結果（OK / 修正 / キャンセル）
- 特殊コマンド:
  - `reset`, `リセット`, `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る

**パラメータ**:
- `query` (str): AG-001 からの委譲リクエスト（申請内容テキスト）
- `tool_context` (ToolContext): `@tool(context=True)` デコレータを通じて受け取る

**ToolContext経由で受け取る情報**:
```python
{
    "applicant_name": str,       # 申請者名（AG-001 で収集済み）
    "application_date": str,     # 申請日（YYYY-MM-DD形式）
    "session_id": str            # セッションID（タイムスタンプ_UUID8桁形式。例: 20260323153045_a1b2c3d4）
}
```

#### 2.2.2 出力

**出力先**: 社員（コンソール / CLI）

**出力メッセージ**:
- 一括確認質問: `"交通費精算申請に必要な情報を確認します。[必須項目一覧]をお知らせください"`
- 上長承認要否通知: `"交通費合計が10,000円を超えるため、上長の事前承認が必要です（BRL-09）。合計：[金額]円"`
- HitL承認確認: `"申請情報を確認してください。[サマリー]。承認しますか？（OK / 修正 / キャンセル）"`
- 申請書チェック結果（合格）: `"申請書チェック：合格。申請書ドラフトが生成されました。ファイル：[パス]"`
- 申請書チェック結果（不備）: `"申請書チェック：不備あり。[不備箇所・理由・適用ルール名]"`

**戻り値の形式**: `str`（エージェントからの最終応答：申請書チェック結果・ファイルパスを含む）

**出力内容**:
- 申請書ドラフト（Excel）ファイルパス
- 申請書チェック結果
- 上長承認要否通知
- エラーメッセージ

### 2.3 ビジネスロジック

#### 2.3.1 システムプロンプト

##### システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-002 Step6 の申請期限チェック（BRL-14）で「移動日から3ヶ月以内」の判断をLLMが行うため、実行時の申請日（application_date）と3ヶ月前の基準日をプロンプトに埋め込む必要がある |

##### 動的に埋め込む情報
- 申請日（`application_date`）: AG-001 から invocation_state で渡された YYYY-MM-DD 形式の文字列
- 申請期限基準日（`application_date` から90日前）: 申請期限チェック（BRL-14）のLLM判断基準

##### 妥当性チェックルールの参照方式

妥当性チェックルール（交通費申請ポリシー）は `agent_knowledge/transportation_policies.py` を参照してプロンプトに組み込む。
ポリシー内容はシステムプロンプト生成時に動的に読み込まれ、LLMが判断基準として使用できるようにプロンプトへ注入される。

##### プロンプト全文（抜粋）

```
あなたは「交通費精算申請エージェント」です。交通費精算申請フロー全体（CF-002）を担当します。

【基本情報】
- 申請日: {application_date}
- 申請期限基準日（3ヶ月前）: {deadline_date}  ← 申請日から90日前

【役割】
1. 申請情報の一括確認収集（Step1-3: CF-002 BRL-07）
   - 移動区間ごとに必須項目（移動日・出発地・目的地・交通手段・業務目的）を一括で質問する
   - 不足情報がある場合は追加収集する（BRL-03）

2. 交通費自動計算（Step4）
   - 全移動区間の情報収集完了後に calculate_transport_fare ツールを呼び出す（BRL-08）
   - 駅名は末尾の「駅」を除去して正規化する（BRL-10）
   - ツール失敗時（EX-06）は社員に手動入力を促す

3. 上長承認要否判断（Step5: BRL-09/JD-08）
   - 交通費合計が10,000円を超える場合のみ通知する

4. 申請期限チェック（Step6: BRL-14/JD-07）
   - 各移動日が申請期限基準日（{deadline_date}）より後であることを確認する
   - 超過している場合は通知するが申請フローは継続可能

5. 申請書ドラフト提示（Step7: テキスト整理）
   - 収集済み情報をテキストで整理して提示する
   - この段階では generate_transport_expense_form ツールを呼び出してはならない

6. HitL承認確認（Step8）
   - 申請情報サマリーを提示して「OK / 修正 / キャンセル」の選択を求める
   - HumanApprovalHook が BeforeToolCallEvent で自動的に承認確認を実施する

7. 申請書生成（Step9: BRL-04）
   - HitL承認（OK）後のみ generate_transport_expense_form ツールを呼び出す
   - 収集済み申請情報のみを使用し、推測・補完を行わない（BRL-04）

8. 申請書チェック（Step10: BRL-05/JD-06）
   - 申請ルール（DATA-001）に基づいて不備を確認し結果を提示する

【制約事項】
- HitL承認（OK）なしに generate_transport_expense_form を呼び出してはならない（GRD-009）
- 収集済み情報に存在しないフィールドを推測・補完してはならない（BRL-04）
- 申請書の自動送信・実際の提出は禁止（GRD-010）
- 対話回数が30回を超えた場合はセッションを終了する（FR-012）
- システム系エラーが発生した場合は呼び出し元エージェント（AG-001）にエラー内容を要約して返す

【特殊コマンド】
- `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る
```

##### プロンプト設計の意図

| 要素 | 意図 |
|------|------|
| 動的日付埋め込み | BRL-14（申請期限チェック）の基準日をLLMに正確に判断させるため実行時に注入する |
| 一括確認方式（BRL-07） | 対話回数を削減するために全必須項目を1メッセージで確認する |
| ドラフト提示とツール実行の分離 | HumanApprovalHook が BeforeToolCallEvent でトリガーされる仕組みを保証するため明示的に記述する |
| GRD-009 の明示 | HitL承認前の TOOL-002 呼び出しを LLM レベルでも禁止して二重に保証する |

#### 2.3.2 申請期限チェックルール（BRL-14）

| ルール | 条件 | 対応 |
|--------|------|------|
| 申請期限内（JD-07） | 移動日 >= 申請期限基準日（application_date - 90日） | フローを継続する |
| 申請期限超過（JD-07） | 移動日 < 申請期限基準日（application_date - 90日） | 期限超過を通知するがフローは継続可能 |

#### 2.3.3 処理フロー

```
開始（invocation_state から applicant_name, application_date, session_id を取得）
  ↓
動的システムプロンプトを生成（application_date, deadline_date を埋め込む）
  ↓
SessionManager(session_id, storage_path="data/sessions/") を初期化する
  ↓
Agent インスタンスを生成する（SlidingWindow(20), HumanApprovalHook, LoopControlHook）
  ↓
Step1: 必須項目一括確認（CF-002 BRL-07）
  - 移動日・出発地・目的地・交通手段・業務目的を一括で質問する
  ↓
Step2: 申請情報回答収集
  ↓
Step3: 充足確認・不足情報追加収集（JD-03/JD-04）
  - 不足がある場合は Step2 に戻る
  ↓
Step4: 交通費自動計算（各区間ごとに TOOL-001 呼び出し）
  - calculate_transport_fare(departure, destination, transport_type, travel_date, purpose)
  - 成功: 計算結果を CollectedTransportInfo に格納する
  - EX-06（経路なし/I/Oエラー）: 手動入力代替へ
  - EX-07（ファイル不存在）: エスカレーション案内・TERMINATED
  ↓
Step5: 上長承認要否通知（合計 > 10,000円 の場合のみ: BRL-09/JD-08）
  ↓
Step6: 申請期限チェック（BRL-14/JD-07）
  - 移動日 < deadline_date の区間があれば通知（フローは継続可能）
  ↓
Step7: 申請書ドラフトをテキストで整理・提示（TOOL-002 は呼び出さない）
  ↓
Step8: HitL承認確認（GRD-009）
  - HumanApprovalHook が BeforeToolCallEvent でトリガー（TOOL-002 呼び出し直前）
  - OK     → Step9 へ
  - 修正   → Step2（情報収集）に戻る
  - キャンセル → フロー終了（CLOSED）
  ↓
Step9: 申請書生成（HitL承認 OK 後のみ）
  - generate_transport_expense_form(applicant_name, application_date, sections, total_fare)
  - 成功: file_path を取得する
  - EX-06（テンプレートなし/I/Oエラー）: エスカレーション案内・TERMINATED
  ↓
Step10: 申請書チェック（DATA-001 参照: BRL-05/JD-06）
  - 合格: `"申請書チェック：合格"` を通知して file_path を提示する
  - 不備: `"申請書チェック：不備あり"` + 不備箇所・理由・適用ルール名を提示する
  ↓
終了（フロー完了結果を AG-001 に返す）
```

### 2.4 設定・構成

#### 2.4.1 共通設定

| 設定項目 | 設定値 | 備考 |
|---------|--------|------|
| 会話管理 | SlidingWindow(20) | 交通費精算申請の必須項目（複数区間）を対話収集するため20件の会話履歴を保持 |
| リトライ戦略 | 標準設定 + TOOL-001ファイル参照1回のみ再試行 | Bedrock API は boto3/botocore 組み込みリトライ（最大6回）。TOOL-001 ファイル参照失敗は1回再試行 |
| フック | HumanApprovalHook + LoopControlHook(max_iterations=10) | TOOL-002 実行前の承認必須。ループ上限制御は全エージェント共通 |
| セッション管理 | SessionManager(session_id, storage_path="data/sessions/") | セッション状態（DATA-005）をファイル永続化してresume可能にするため |

#### 2.4.2 利用ツール

| ツールID | ツール名 | 利用目的 | 呼び出し条件 |
|----------|----------|----------|--------------|
| TOOL-001 | `calculate_transport_fare` | 各移動区間の交通費を計算する | Step4：全移動区間の情報収集完了後（1区間1呼び出し） |
| TOOL-002 | `generate_transport_expense_form` | 交通費精算申請書ドラフト（Excel）を生成する | Step9：HitL承認（OK）後のみ呼び出し可能（GRD-009） |

#### 2.4.3 エージェント固有の設定値

| 設定項目 | 設定値 | 説明 |
|---------|--------|------|
| 上長承認要否閾値 | 10,000円 | 交通費合計がこの金額を超える場合に上長承認要否を通知する（BRL-09） |
| 申請期限日数 | 90日 | 申請日から遡ってこの日数以内の移動日が申請可能（BRL-14） |

### 2.5 連携設計

#### 2.5.1 Agent as Tools設計

**ツール名**: `handle_transport_expense_application`

**ツール説明**:
```
交通費精算申請エージェント（AG-002）を呼び出す。
交通費精算申請フロー全体（情報収集・交通費計算・HitL承認・申請書生成・申請書チェック）を
自己完結で実行して結果を返す。
```

**ToolContextの使用**:
- `@tool(context=True)` デコレータを使用する
- `tool_context` パラメータで ToolContext を受け取る
- invocation_state から `applicant_name`・`application_date` を取得してエージェント初期化に使用する
- `session_id` は ToolContext から取得し、SessionManager の初期化に使用する（エージェントへの invocation_state には含めない）

#### 2.5.2 invocation_stateの受け渡し

```python
# AG-001 から受け取る（ToolContext 経由）
state = tool_context.invocation_state
applicant_name = state["applicant_name"]
application_date = state["application_date"]
session_id = state["session_id"]   # SessionManager に渡す。エージェントへは渡さない

# 動的プロンプト生成
system_prompt = build_transport_agent_prompt(application_date)

# SessionManager の初期化
session_manager = SessionManager(session_id=session_id, storage_path="data/sessions/")

# エージェントに渡す invocation_state（session_id は含めない）
invocation_state = {
    "applicant_name": applicant_name,
    "application_date": application_date
}
response = agent(query, invocation_state=invocation_state)
```

## 3. 実装詳細

### 3.1 クラス/関数設計

#### 3.1.1 build_transport_agent_prompt関数

##### 説明
動的システムプロンプトを生成する関数。application_date と deadline_date を埋め込む。

##### 引数
- `application_date` (str): 申請日（YYYY-MM-DD形式）（必須）

##### 戻り値
- `str`: 動的システムプロンプト文字列（application_date と deadline_date が埋め込まれた状態）

##### 処理内容
1. `application_date` を `datetime.date` にパースする
2. `deadline_date = application_date - datetime.timedelta(days=90)` を計算する
3. `agent_knowledge/transportation_policies.py` を読み込み、ポリシー内容をプロンプトに組み込む
4. プロンプトテンプレートに `application_date`・`deadline_date`・ポリシー内容を埋め込んで返す

#### 3.1.2 handle_transport_expense_application関数

##### 説明
AG-002 を Agent As Tools として AG-001 に公開するツール関数。

##### 引数
- `query` (str): AG-001 からの委譲リクエスト（申請内容テキスト）（必須）
- `tool_context` (ToolContext): ToolContext インスタンス（`@tool(context=True)` から自動渡し）（必須）

##### 戻り値
- `str`: エージェントからの最終応答（申請書チェック結果・ファイルパスを含む）

##### 処理内容
1. `tool_context.invocation_state` から `applicant_name`・`application_date`・`session_id` を取得する
2. `build_transport_agent_prompt(application_date)` で動的プロンプトを生成する
3. `SessionManager(session_id=session_id, storage_path="data/sessions/")` を初期化する
4. `HumanApprovalHook`・`LoopControlHook(max_iterations=10)` を生成する
5. `Agent(model=..., system_prompt=..., tools=[calculate_transport_fare, generate_transport_expense_form], conversation_manager=SlidingWindowConversationManager(window_size=20), callback_handler=...)` を生成する
6. `agent(query, invocation_state={"applicant_name": ..., "application_date": ...})` を呼び出す
7. 応答を返す

##### 例外処理

以下の例外クラスを個別に捕捉する。ログ出力は ErrorHandler 呼び出し前に行う。エラー時の戻り値はすべて `str`（呼び出し元 LLM コンテキストに渡されるため）。

| 例外クラス | ログレベル | ログ付加情報 | 処理内容 |
|-----------|-----------|------------|---------|
| `LoopLimitError` | WARNING | `extra={"query_preview": query[:50]}` | `ErrorHandler.handle_loop_limit_error(e)` の戻り値（str）を return する |
| `ContextWindowOverflowException` | WARNING | `extra={"query_preview": query[:50]}` | `ErrorHandler.handle_context_window_error(e)` の戻り値（str）を return する |

## 5. 補足情報

### 5.1 実装上の注意点

1. **動的プロンプトの deadline_date 計算**
   - `application_date` は YYYY-MM-DD 形式の文字列として渡されるため、`datetime.date.fromisoformat(application_date)` でパースしてから `timedelta(days=90)` を引く

2. **HumanApprovalHook と TOOL-002 の関係**
   - HumanApprovalHook は `BeforeToolCallEvent` で `generate_transport_expense_form` の実行直前に自動的に承認確認を実施する
   - エージェントのシステムプロンプトでも HitL承認前の TOOL-002 呼び出しを禁止しており、二重に保証する

3. **TOOL-001 のリトライ管理**
   - TOOL-001 の IOError 時の1回リトライはツール側（`tools/transport_calculator.py`）で実装される
   - AG-002 は TOOL-001 からエラーが返った場合に手動入力代替フローへ遷移する

4. **session_id の ToolContext からの取得**
   - `session_id` は invocation_state からではなく `tool_context.invocation_state["session_id"]` から取得する
   - エージェントに渡す invocation_state には `session_id` を含めない（SYS-MA-001 6.1節準拠）

---

### 5.2 パフォーマンス考慮事項

1. **SlidingWindow(20) によるコンテキスト制限**
   - 20件を超えた古い会話履歴は LLM に渡されないため、重要な収集済み申請情報は SessionManager（DATA-005）で永続化する

---

### 5.3 セキュリティ考慮事項

1. **BRL-04 の遵守**
   - 申請書生成時に収集済み申請情報に存在しないフィールドを推測・補完しない

2. **GRD-009 の遵守**
   - HumanApprovalHook によって TOOL-002 の実行前に必ず承認確認が実施される
