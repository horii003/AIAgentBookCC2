## 1. 概要

### 1.1 コンポーネントの目的

AG-002（交通費精算申請エージェント）が申告した移動区間情報（出発地・目的地・交通手段・移動日）をもとに、運賃データファイルを参照して正確な交通費を計算する。電車経路の場合は `./template/train_routes.json`（DATA-003）を検索し、バス・タクシー・飛行機の場合は `./template/fixed_fares.json`（DATA-004）の固定運賃を取得する。

### 1.2 主要な責務

1. **入力バリデーション**: `TransportSectionInput` Pydanticモデルによる出発地・目的地・交通手段・移動日・業務目的の型・形式チェックと正規化
2. **電車経路テーブル検索**: DATA-003（`./template/train_routes.json`）から出発地・目的地をキーに運賃を検索する
3. **固定運賃取得**: DATA-004（`./template/fixed_fares.json`）から交通手段をキーに固定金額を取得する
4. **計算根拠の返却**: 使用した経路テーブルまたは固定運賃の情報を出力に含め、エージェントが社員に根拠を提示できるようにする（BRL-02準拠）
5. **運賃データのキャッシュ**: アプリケーション起動時またはツール初回呼び出し時にDATA-003/004をメモリにロードし、都度ファイルI/Oを行わない
   - DATA-003: `./template/train_routes.json`
   - DATA-004: `./template/fixed_fares.json`

## 3. calculate_transport_fareツール設計

### 3.1 基本情報

#### ツール名（関数名）
`calculate_transport_fare`

#### 説明（docstring）
```
指定された移動区間の交通費運賃を計算して返す。

電車の場合は電車経路・運賃データ（DATA-003: ./template/train_routes.json）から
出発地と目的地をキーに運賃を検索する。
バス・タクシー・飛行機の場合は固定運賃データ（DATA-004: ./template/fixed_fares.json）
から交通手段をキーに固定金額を取得する。

申請者名・申請日はツール関数内部で invocation_state から取得する。

Args:
    departure (str): 出発地（駅名またはバス停名等）。末尾の「駅」は自動除去される。
    destination (str): 目的地（駅名またはバス停名等）。末尾の「駅」は自動除去される。
    transport_type (str): 交通手段。「電車」「バス」「タクシー」「飛行機」のいずれか。
        英語表記（"train", "bus", "taxi", "plane", "airplane"）も受け付け自動変換する。
    travel_date (str): 移動日。YYYY-MM-DD形式（例: "2026-07-03"）。
        YYYY/MM/DDおよびYYYY年MM月DD日形式も受け付け自動変換する。
    purpose (str): 業務目的。500文字以内の非空文字列（BRL-15必須）。

Returns:
    dict: 以下のキーを持つ辞書（成功時）:
        - fare (int): 交通費金額（円、非負整数）
        - calculation_basis (str): 計算根拠の説明文
        - transport_type (str): 正規化後の交通手段
        - departure (str): 正規化後の出発地
        - destination (str): 正規化後の目的地
        - travel_date (str): 移動日（YYYY-MM-DD形式）
    dict: エラー時: {"success": False, "message": "エラーメッセージ文字列"}

Note:
    例外を raise せず、エラー情報は戻り値として返す。
```

### 3.2 インターフェース設計

#### 3.2.1 入力

**パラメータ定義**:

| 引数名 | 型 | 説明 | デフォルト値 |
|--------|-----|------|--------------|
| `departure` | `str` | 出発地（駅名またはバス停名等。末尾「駅」は自動除去） | なし（必須） |
| `destination` | `str` | 目的地（駅名またはバス停名等。末尾「駅」は自動除去） | なし（必須） |
| `transport_type` | `str` | 交通手段（電車/バス/タクシー/飛行機、英語表記も可） | なし（必須） |
| `travel_date` | `str` | 移動日（YYYY-MM-DD、YYYY/MM/DD、YYYY年MM月DD日） | なし（必須） |
| `purpose` | `str` | 業務目的（BRL-15：非空・500文字以内） | なし（必須） |

##### バリデーションルール詳細

| フィールド | ルール | エラーメッセージ | 例 |
|-----------|--------|----------------|-----|
| `departure` | 空文字禁止（正規化後も非空） | `"駅名を入力してください"` | `""` → エラー |
| `destination` | 空文字禁止（正規化後も非空） | `"駅名を入力してください"` | `""` → エラー |
| `transport_type` | `["電車", "バス", "タクシー", "飛行機"]` または対応英語表記 | `"交通手段「{v}」は対象外です。電車・バス・タクシー・飛行機のいずれかを入力してください"` | `"自転車"` → エラー |
| `travel_date` | YYYY-MM-DD / YYYY/MM/DD / YYYY年MM月DD日のいずれか | `"日付の形式が不正です。YYYY-MM-DD形式で入力してください（例: 2026-07-03）"` | `"20260703"` → エラー |
| `purpose` | 非空かつ500文字以内 | `"この項目は必須です。"` / `"入力内容が長すぎます。500文字以内で入力してください"` | `""` → エラー |

##### 正規化処理

| 入力値 | 正規化後 | 処理内容 |
|--------|---------|---------|
| `"渋谷駅"` | `"渋谷"` | `removesuffix("駅").strip()` |
| `"train"` | `"電車"` | `normalize_transport_type` マッピング適用 |
| `"airplane"` | `"飛行機"` | `normalize_transport_type` マッピング適用 |
| `"2026/07/03"` | `date(2026, 7, 3)` | `python-dateutil` による自動パース |

**戻り値の型**: `dict[str, Any]`

**説明**: 交通費金額・計算根拠・正規化済みの入力情報を含む辞書

**構造**:
```python
{
    "fare": int,               # 交通費金額（円、非負整数）
    "calculation_basis": str,  # 計算根拠（例: "電車経路テーブル（渋谷→新宿）: 200円"）
    "transport_type": str,     # 正規化後の交通手段（例: "電車"）
    "departure": str,          # 正規化後の出発地（例: "渋谷"）
    "destination": str,        # 正規化後の目的地（例: "新宿"）
    "travel_date": str,        # 移動日（YYYY-MM-DD形式）
}
```

#### 3.2.3 エラー時の戻り値

| エラー種別 | 発生条件 | 戻り値形式 | 戻り値例 |
|-----------|---------|-----------|---------|

| ファイル不存在 | DATA-003またはDATA-004のファイルが存在しない（EX-07） | `dict` | `{"success": False, "message": "運賃データファイルが見つかりません: ./template/train_routes.json"}` |

### 3.3 ビジネスロジック

#### 3.3.1 処理フロー

```
開始
  ↓
TransportSectionInput Pydanticモデルで入力バリデーション・正規化
  - departure: normalize_station_name（末尾「駅」除去）
  - destination: normalize_station_name（末尾「駅」除去）
  - transport_type: normalize_transport_type（英語→日本語変換）
  - travel_date: parse_date（各種形式→datetime.date）
  - purpose: validate_non_empty_string（非空・500文字以内）
  ↓
バリデーション失敗?
  Yes → {"success": False, "message": "エラーメッセージ"} を返す（EX-01）
  No ↓
transport_type が「電車」か?
  Yes → _search_train_fare(departure, destination) を呼び出す
    ↓
    _load_fare_data()（初回のみファイルロード）
      ↓ ファイル不存在・I/Oエラーの場合 → {"success": False, "message": "..."} を返す（EX-07）
    ↓
    train_fares[departure][destination] を検索
    ↓
    経路が存在しない場合 → {"success": False, "message": "経路が見つかりません: {departure}→{destination}"} を返す（EX-06）
    存在する場合 → fare（int）と calculation_basis（str）を取得
  No → _get_fixed_fare(transport_type) を呼び出す
    ↓
    _load_fare_data()（初回のみファイルロード）
      ↓ ファイル不存在・I/Oエラーの場合 → {"success": False, "message": "..."} を返す（EX-07）
    ↓
    fixed_fares[transport_type] を検索
    ↓
    取得した fare（int）と calculation_basis（str）を設定
  ↓
結果辞書を構築して返す
```

#### 3.3.2 キャッシュ機構

**キャッシュ機構**:

**用途**: DATA-003（`./template/train_routes.json`）とDATA-004（`./template/fixed_fares.json`）をアプリケーション起動時またはツール初回呼び出し時にメモリ上にロードし、都度ファイルI/Oを行わない（バリデーション方針 9b節準拠）

**実装**: モジュールレベルのキャッシュ変数 `_train_fares_cache: Optional[dict]` および `_fixed_fares_cache: Optional[dict]` を `None` で初期化し、`_load_fare_data()` の初回呼び出し時にJSONファイルをロードして代入する。以降は既存のキャッシュ変数を参照する。

#### 3.3.3 データの検証方法（該当する場合のみ）

**Pydanticモデル**: `TransportSectionInput`

**検証内容**:
- `travel_date`: `parse_date` バリデータ（`mode="before"`）で `datetime.date` 型に変換
- `departure`, `destination`: `normalize_station_name` バリデータ（`mode="before"`）で末尾「駅」除去・空文字チェック
- `transport_type`: `normalize_transport_type` バリデータ（`mode="before"`）で英語表記を日本語列挙値に正規化
- `purpose`: `validate_non_empty_string` バリデータで非空・500文字以内を検証

### 3.4 エラーハンドリング

#### 3.4.1 エラー処理フロー

```
エラー発生
  ↓
エラー種別の判定
  ↓
ValidationError（入力バリデーション失敗）
  → EX-01として分類
  → 日本語エラーメッセージを生成
  → ErrorHandlerに渡す
  → セッション状態はWAITINGのまま（社員に再入力を促す）
  → {"success": False, "message": "エラーメッセージ"} を返す
  
ValueError（電車経路テーブルに経路なし）またはIOError（1回目）
  → EX-06として分類
  → 1回だけ再試行（IOErrorのみ）
  → 再試行も失敗またはValueError → 手動入力代替へ遷移
  → ユーザー向けメッセージ生成
  → ErrorHandlerに渡す
  → {"success": False, "message": "エラーメッセージ"} を返す
  
ファイル不存在（os.path.exists()チェック失敗）または json.JSONDecodeError
  → EX-07として分類
  → 再試行なし
  → エスカレーション案内・セッションTERMINATED
  → ErrorHandlerに渡す
  → {"success": False, "message": "エラーメッセージ"} を返す（WARNINGレベルでログ出力）
  
上記以外のエラー
  → EX-08として分類
  → 再試行なし
  → エスカレーション案内・セッションTERMINATED
  → ErrorHandlerに渡す
  → {"success": False, "message": "エラーメッセージ"} を返す
```

#### 3.4.2 エラーメッセージの詳細

| エラー種別 | 発生条件 | 対応 | ユーザー向けメッセージ | 戻り値形式 |
|-----------|---------|------|---------------------|-----------|
| EX-01（入力例外） | バリデーション失敗（交通手段不正等） | 再入力促し | `"入力内容に誤りがあります。{フィールド別メッセージ}"` | `{"success": False, "message": "..."}` |
| EX-06（電車経路なし） | DATA-003に対象経路が存在しない（ValueError） | 手動入力代替 | `"入力された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。"` | `{"success": False, "message": "..."}` |
| EX-06（ファイルI/Oエラー） | DATA-003/004 I/Oエラー（1回再試行後） | 手動入力代替 | `"運賃データの読み込みに失敗しました。交通費を手動で入力してください。"` | `{"success": False, "message": "..."}` |
| EX-07（システム障害） | DATA-003/004 ファイル不存在（os.path.exists()チェック失敗）・JSON不正 | エスカレーション案内・TERMINATED | `"申請処理中にエラーが発生しました。しばらく経ってから再度お試しいただくか、担当部門にお問い合わせください。"` | `{"success": False, "message": "..."}` ／ WARNINGレベルでログ出力 |
| EX-08（想定外） | 上記以外のエラー | エスカレーション案内・TERMINATED | `"申請処理中にエラーが発生しました。しばらく経ってから再度お試しいただくか、担当部門にお問い合わせください。"` | `{"success": False, "message": "..."}` |

## 5. 補助機能

### 5.1 _load_fare_data

**目的**: DATA-003（`./template/train_routes.json`）とDATA-004（`./template/fixed_fares.json`）を読み込み、モジュールレベルキャッシュ変数に格納する。キャッシュ済みの場合は再ロードしない。

**戻り値**: `tuple[bool, str]`
- 成功時: `(True, "")`
- エラー時: `(False, "エラーメッセージ文字列")`

**処理**:
1. `_train_fares_cache` が `None` の場合のみ以下を実施する:
   - `os.path.exists("./template/train_routes.json")` で事前チェックを行う
   - ファイルが存在しない場合: WARNINGレベルでログ出力し `(False, "運賃データファイルが見つかりません: ./template/train_routes.json")` を返す
   - ファイルが存在する場合: `json.load()` で読み込みキャッシュに代入する
2. `_fixed_fares_cache` が `None` の場合のみ以下を実施する:
   - `os.path.exists("./template/fixed_fares.json")` で事前チェックを行う
   - ファイルが存在しない場合: WARNINGレベルでログ出力し `(False, "運賃データファイルが見つかりません: ./template/fixed_fares.json")` を返す
   - ファイルが存在する場合: `json.load()` で読み込みキャッシュに代入する
3. JSON形式不正の場合: `(False, "運賃データファイルの形式が不正です: {path}")` を返す
4. すべて成功した場合: `(True, "")` を返す

**注意**: `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止。`os.path.exists()` による事前チェックのみを使用すること。例外を raise せず戻り値としてエラー情報を返すこと。

### 5.2 _search_train_fare

**目的**: `_train_fares_cache` から `departure` → `destination` のキーで運賃を検索して返す。

**戻り値**: `dict`
- 成功時: `{"fare": int, "calculation_basis": str}`
- エラー時: `{"success": False, "message": "エラーメッセージ文字列"}`

**処理**:
1. `_load_fare_data()` を呼び出してキャッシュを確保する
2. `_load_fare_data()` が `(False, ...)` を返した場合: `{"success": False, "message": "..."}` を返す
3. `_train_fares_cache.get(departure, {}).get(destination)` で運賃を取得する
4. `None` の場合: ValueError相当として `{"success": False, "message": "経路が見つかりません: {departure}→{destination}"}` を返す（EX-06）
5. 取得できた場合は `{"fare": fare, "calculation_basis": calculation_basis}` を返す

### 5.3 _get_fixed_fare

**目的**: `_fixed_fares_cache` から `transport_type` をキーに固定運賃を取得して返す。

**戻り値**: `dict`
- 成功時: `{"fare": int, "calculation_basis": str}`
- エラー時: `{"success": False, "message": "エラーメッセージ文字列"}`

**処理**:
1. `_load_fare_data()` を呼び出してキャッシュを確保する
2. `_load_fare_data()` が `(False, ...)` を返した場合: `{"success": False, "message": "..."}` を返す
3. `_fixed_fares_cache.get(transport_type)` で運賃を取得する
4. `None` の場合: ValueError相当として `{"success": False, "message": "固定運賃が見つかりません: {transport_type}"}` を返す（EX-06）
5. 取得できた場合は `{"fare": fare, "calculation_basis": calculation_basis}` を返す

## 6. 補足情報

### 6.1 実装上の注意点

1. **ファイルI/O禁止（都度）**
   - DATA-003/004 は起動時またはツール初回呼び出し時のみロードし、各呼び出し時のファイルI/Oを行わない

2. **1区間1呼び出し**
   - 1ツール呼び出しで処理できるのは1移動区間のみ。複数区間はエージェントがループで呼び出す

3. **再試行ルール**
   - IOError 発生時のみ1回だけ再試行する。2回失敗または ValueError（経路未登録）の場合は即座に手動入力代替へ遷移する（実行制御方針 7.1節準拠）

4. **`@tool` デコレータの使用**
   - strands-agents v1.25.0 の `@tool` デコレータを使用して関数を定義する

5. **例外を raise しない設計**
   - 本ツールは例外を raise せず、エラー情報を戻り値（`dict` または `tuple[bool, str]`）として返す
   - 計算系メソッドのエラー時戻り値: `{"success": False, "message": "エラーメッセージ文字列"}`
   - データ読み込み系メソッドのエラー時戻り値: `(False, "エラーメッセージ文字列")`

6. **ファイル存在チェック方式**
   - `_load_fare_data()` 内では `os.path.exists()` による事前チェックを実施する
   - `open()` 実行時の `FileNotFoundError` 捕捉との併用は禁止
   - ファイルが存在しない場合はWARNINGレベルでログ出力し `(False, エラーメッセージ)` を返す

7. **申請者名・申請日の取得**
   - 申請者名・申請日はツール関数内部で `invocation_state` から取得する（ツール引数として受け取らない）

---

### 6.2 パフォーマンス考慮事項

1. **キャッシュによるI/O最小化**
   - 運賃データをメモリキャッシュすることで、ツール呼び出しごとのファイルI/Oを排除し応答遅延を低減する

2. **辞書のネスト検索**
   - `train_fares[departure][destination]` の検索は O(1) であるため、データ量に依存しない一定時間で取得できる

---

### 6.3 セキュリティ考慮事項

1. **ファイル直接参照（外部API不使用）**
   - RAGを使用せず運賃データを直接ローカルファイルから参照することで、機微情報の外部送信リスクを排除する

2. **入力値の正規化**
   - Pydanticモデルでバリデーション・正規化を実施し、不正な入力値が後続処理に伝播することを防止する

## 8. テスト観点

### 8.1 機能テスト
- 電車の登録済み経路（例: 渋谷→新宿）でツールを呼び出した場合に、DATA-003の運賃と一致する金額が返されること（BRL-08準拠）
- バス・タクシー・飛行機の固定運賃（バス230円・タクシー10,000円・飛行機50,000円）が正しく返されること（DATA-004準拠）

- `travel_date` が戻り値に YYYY-MM-DD 形式で含まれること

### 8.2 異常系テスト
- 電車の未登録経路（例: 渋谷→大阪）でツールを呼び出した場合に `{"success": False, "message": "経路が見つかりません: 渋谷→大阪"}` が返されること（EX-06）

- `departure` に空文字を渡した場合に `{"success": False, "message": "..."}` が返されること（EX-01）
- `travel_date` に `"20260703"` を渡した場合に `{"success": False, "message": "..."}` が返されること（EX-01）
- いかなるエラーケースでも例外が raise されないこと（戻り値のみでエラーを通知すること）

### 8.3 性能テスト
- 初回呼び出し時の応答時間（ファイルロードを含む）が1秒以内であること
- キャッシュ済みの2回目以降の呼び出しが100ms以内であること

- `purpose` が500文字の入力でバリデーション通過すること
- `purpose` が501文字の入力で `ValueError` が発生すること
