## 1. 概要

### 1.1 コンポーネントの目的

AG-001/AG-002/AG-003 が申請種別判断・必須項目充足確認・申請書ルール準拠チェック・経費区分判断等の業務判断を行う際に参照するナレッジ（知識データ）と業務ルールを Python モジュールとして定義し、実装方法・システムプロンプトへの組み込み方式・メンテナンス手順を明確化する。

### 1.2 ナレッジ・ルールの分類

| 分類 | 説明 |
|------|------|
| 静的知識 | `data/rules/` 配下の社内申請ルールファイル（DATA-001）。LLM が直接参照するファイルベースのナレッジ。`knowledge/` Python モジュールではなく外部ファイルとして管理する |
| 動的ルール | 実行時に Python コードで判定する業務ルール（BRL-09/BRL-13: 上長承認通知閾値、BRL-14: 申請期限チェック、BRL-10: 駅名正規化等）。`models/data_models.py` のバリデーター・ツール実装・システムプロンプト内テキストとして実装する |
| 判断基準 | LLM が判断に使用するルール・定義（BRL-12: 経費区分定義、BRL-01: 申請種別判断基準等）。`knowledge/` Python モジュールにテキストデータとして定義し、システムプロンプトに組み込む |

### 2.2 KN-001: 申請種別判定ルール

#### 基本情報

| 項目 | 値 |
|------|-----|
| ナレッジID | KN-001 |
| ファイルパス | `knowledge/application_type_policies.py` |
| Pythonモジュール名 | `knowledge.application_type_policies` |
| 変数名 | `APPLICATION_TYPE_RULES` |
| 型 | `str`（テキスト形式の判定ルール説明） |

#### ナレッジ内容定義

```python
APPLICATION_TYPE_RULES: str = """
申請種別判定ルール（KN-001）

申請種別は以下の基準で判断する:

1. 交通費精算申請
   - 電車・バス・タクシー・飛行機等の交通手段を利用した移動に関する経費
   - 業務目的の移動（出張・顧客訪問・社内移動等）の交通費

2. 経費精算申請
   - 事務用品費: 文房具・コピー用紙・印刷用品等の事務用品購入
   - 宿泊費: 出張・研修等の宿泊費用
   - 資格精算費: 業務関連の資格取得・試験費用・研修費用
   - その他経費: 上記に分類できない業務上必要な経費

判断不能時（JD-02）: 上記いずれにも分類できない場合は社員に選択肢を提示する
"""
```

#### システムプロンプトへの組み込み

| 項目 | 値 |
|------|-----|
| 組み込み対象エージェント | AG-001（申請受付窓口エージェント） |
| 組み込み方式 | DATA-001 ファイル参照指示をシステムプロンプトに記述する方式。`APPLICATION_TYPE_RULES` は AG-001 の補完知識として参照用に保持するが、主要なルール参照は LLM が `data/rules/` を直接参照する |
| 挿入箇所 | システムプロンプト「役割」セクション Step5 の DATA-001 参照指示内 |

### 3.2 BRL-01: 申請種別判断

#### ルール定義

| 項目 | 値 |
|------|-----|
| ルールID | BRL-01 |
| 名称 | 申請種別判断 |
| 対象エージェント | AG-001 |
| 実装場所 | システムプロンプト内テキスト + LLM による `data/rules/`（DATA-001）参照 |
| 優先順位 | 1 |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|
| 申請内容テキストが交通費に関連する（電車代・タクシー代・バス代・飛行機代等） | 申請種別 = 「交通費精算申請」として判定し BRL-02 へ進む |
| 申請内容テキストが経費（事務用品・宿泊費・資格精算費等）に関連する | 申請種別 = 「経費精算申請」として判定し BRL-02 へ進む |
| 申請内容テキストがいずれにも分類できない（JD-02） | BRL-06 へ進み選択肢を提示する |

#### 例外・特殊ケース

| ケース | 対応 |
|--------|------|
| 申請内容テキストが空文字・500文字超 | EX-01 として再入力要求（BRL-01 適用前にバリデーション） |

### 3.7 BRL-14: 申請期限（3ヶ月）制約

#### ルール定義

| 項目 | 値 |
|------|-----|
| ルールID | BRL-14 |
| 名称 | 申請期限（3ヶ月）制約 |
| 対象エージェント | AG-002, AG-003 |
| 実装場所 | 動的システムプロンプト（`{deadline_date}` プレースホルダー）+ LLM による日付比較 |
| 優先順位 | 1 |

#### 条件・処理定義

| 条件（IF） | 処理（THEN） |
|-----------|------------|
| 移動日（AG-002）または購入日（AG-003）< 申請期限基準日（`deadline_date`）（JD-07） | 申請期限を超過している旨を社員に通知する（申請フローは継続可能） |
| 移動日または購入日 >= 申請期限基準日（`deadline_date`）（JD-07） | 期限内として通知せずに処理を継続する |

#### 例外・特殊ケース

| ケース | 対応 |
|--------|------|
| 複数区間・複数品目がある場合 | 各移動日・購入日を個別にチェックする（1件でも超過があれば通知する） |
| 申請期限超過時もフロー継続 | 超過通知後も申請フローを中断しない（BRL-14 は強制中断ルールではない） |

## 5. メンテナンス設計

### 5.1 ナレッジ更新手順

1. 更新対象の `knowledge/` 配下 Python ファイルを特定する
2. 該当変数（`APPLICATION_TYPE_RULES`・`APPLICATION_DESTINATION`・`REQUIRED_FIELDS`・`EXPENSE_CATEGORY_RULES`）のテキストまたは辞書値を編集する
3. 関連するシステムプロンプトへの影響を確認する（システムプロンプト詳細設計書を参照）
4. テスト観点（8章）に従って動作確認を実施する
5. 変更内容とバージョンを変更履歴に記録する

### 5.2 ルール更新手順

1. 変更対象の業務ルール（BRL-XX）を特定する
2. 閾値変更の場合（BRL-09/BRL-13）: システムプロンプト定数の数値を変更する（`agents/transport_expense_agent.py` / `agents/expense_agent.py`）
3. 申請期限変更の場合（BRL-14）: `build_transport_agent_prompt` / `build_expense_agent_prompt` 関数の `timedelta(days=90)` の日数を変更する
4. 必須項目変更の場合（BRL-15/KN-003）: `knowledge/required_fields_policies.py` の `REQUIRED_FIELDS` を更新し、`models/data_models.py` の対応モデルクラスも同期して更新する
5. テスト観点（8章）に従って境界値テストを再実施する

## 6. 補足情報

### 6.1 実装上の注意点

1. **DATA-001 との役割分担**
   - `knowledge/` Python モジュールは LLM の判断支援用テキストデータを管理する。申請ルールの詳細（申請書フォーマット・申請先組織・具体的なルール文言）は `data/rules/` 配下の DATA-001 ファイルで管理し、Python モジュールに転記しない

2. **`REQUIRED_FIELDS` と Pydantic モデルの同期**
   - `knowledge/required_fields_policies.py` の `REQUIRED_FIELDS` と `models/data_models.py` の `CollectedTransportInfo`・`CollectedExpenseInfo` の必須フィールド定義は常に同期させること。一方を変更した場合は他方も同時に更新する

3. **経費区分 Literal 型との同期**
   - `knowledge/expense_category_policies.py` の経費区分定義（事務用品費・宿泊費・資格精算費・その他経費）と `models/data_models.py` の `ExpenseItem.expense_category` の `Literal[...]` 定義は常に同期させること

4. **KN-002 申請先情報の未定義**
   - `APPLICATION_DESTINATION` の値は現時点で要件上未定義。申請先が定義された場合は辞書の値を更新する

## 8. テスト観点

### 8.1 ナレッジ参照テスト
- `from knowledge.application_type_policies import APPLICATION_TYPE_RULES, APPLICATION_DESTINATION` が正常にインポートされること
- `from knowledge.required_fields_policies import REQUIRED_FIELDS` が正常にインポートされること

- `REQUIRED_FIELDS["経費精算申請"]` に `"purchase_date"` / `"store_name"` / `"item_name"` / `"expense_category"` / `"amount"` / `"purpose"` が含まれること

### 8.2 ルール適用テスト
- BRL-09: 交通費合計 10,001円で上長承認通知が表示されること
- BRL-09: 交通費合計 10,000円で上長承認通知が表示されないこと（10,000円ちょうどは通知不要）
- BRL-13: 経費合計 5,001円で上長承認通知が表示されること
- BRL-13: 経費合計 5,000円で上長承認通知が表示されないこと（5,000円ちょうどは通知不要）
- BRL-14: 申請日 2026-07-03 の場合、deadline_date = 2026-04-04 となること（90日前）
- BRL-12: 品目「ボールペン」から「事務用品費」が自動判断されること
- BRL-12: 品目「不明な商品」から「その他経費」が候補として提示されること（COND-002）

### 8.3 境界値テスト
- BRL-09 境界値: 交通費合計ちょうど 10,000円では通知なし、10,001円では通知あり
- BRL-13 境界値: 経費合計ちょうど 5,000円では通知なし、5,001円では通知あり
- BRL-14 境界値: 申請期限基準日ちょうどの移動日（`deadline_date` と同日）は期限内として処理されること
- BRL-14 閏年: `application_date = "2024-03-01"` で `deadline_date = "2023-12-01"` となること（90日前）
