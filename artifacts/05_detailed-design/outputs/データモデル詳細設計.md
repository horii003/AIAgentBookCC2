## 1. 概要

### 1.1 コンポーネントの目的

全エージェント（AG-001/AG-002/AG-003）とツール（TOOL-001/TOOL-002）が使用する入出力データ・状態管理データの型を `models/data_models.py` に一元定義する。Pydantic v2 モデルによるバリデーション・正規化を集約し、エージェント間・ツール間のデータ受け渡しに型安全性を提供する。

### 1.2 モデル分類

| 分類 | 説明 |
|------|------|
| 入力モデル | ツールパラメータ（TOOL-001/TOOL-002入力値）の型・バリデーションを定義する |
| 出力モデル | ツール出力（TOOL-001/TOOL-002返却値）の型を定義する |
| 内部モデル | エージェントが収集した申請情報・エージェント間委譲メッセージを定義する |
| 状態管理モデル | セッション状態（DATA-005）・申請フローコンテキストを定義する |
| invocation_stateモデル | AG-001→AG-002/AG-003への委譲時に渡す辞書リテラル（Pydanticモデルは定義しない） |

## 3. 各モデルの詳細設計

### 3.1 TransportSectionInputモデル

#### 基本情報

| 項目 | 値 |
|------|-----|
| クラス名 | `TransportSectionInput` |
| 基底クラス | `BaseModel` |
| 分類 | 入力モデル |
| 説明 | TOOL-001（交通費計算ツール）への入力パラメータ（1移動区間）。AG-002 がエージェントのターンごとに生成してTOOL-001に渡す |

#### フィールド定義

| フィールド名 | 型 | 必須/任意 | デフォルト値 | 説明 | バリデーション制約 |
|------------|-----|-----------|------------|------|-----------------|
| `travel_date` | `date` | 必須 | なし | 移動日（YYYY-MM-DD形式）。DATA-003の検索キー | `parse_date`（mode="before"）。未来日付は許容しない |
| `departure` | `str` | 必須 | なし | 出発地（BRL-10正規化済み）。DATA-003の検索キー | `normalize_station_name`（mode="before"）。空文字は ValueError |
| `destination` | `str` | 必須 | なし | 目的地（BRL-10正規化済み）。DATA-003の検索キー | `normalize_station_name`（mode="before"）。空文字は ValueError |
| `transport_type` | `Literal["電車", "バス", "タクシー", "飛行機"]` | 必須 | なし | 交通手段（BRL-07/BRL-08） | `normalize_transport_type`（mode="before"）。範囲外は ValueError |
| `purpose` | `str` | 必須 | なし | 業務目的（BRL-15必須） | `validate_non_empty_string`。空・500文字超は ValueError |

#### バリデーター

| バリデーター名 | 種別 | 対象フィールド | 処理内容 |
|-------------|------|-------------|---------|
| `parse_date` | `@field_validator(mode="before")` | `travel_date` | 文字列（YYYY-MM-DD / YYYY/MM/DD / YYYY年MM月DD日）を `datetime.date` に変換する。`python-dateutil` を使用する |
| `normalize_station_name` | `@field_validator(mode="before")` | `departure`, `destination` | `removesuffix("駅").strip()` で末尾「駅」を除去する。空文字は `ValueError("駅名を入力してください")` |
| `normalize_transport_type` | `@field_validator(mode="before")` | `transport_type` | `{"train": "電車", "鉄道": "電車", "bus": "バス", "taxi": "タクシー", "plane": "飛行機", "airplane": "飛行機", "飛行": "飛行機"}` で正規化する。範囲外は `ValueError("交通手段は電車・バス・タクシー・飛行機のいずれかを入力してください")` |
| `validate_non_empty_string` | `@field_validator` | `purpose` | `strip()` 後に空文字 → `ValueError("この項目は必須です。入力内容を確認してください")`。501文字以上 → `ValueError("入力内容が長すぎます。500文字以内で入力してください")` |

#### シリアライズ仕様

| メソッド | 用途 | 注意事項 |
|--------|------|---------|
| `model_dump()` | TOOL-001 呼び出し引数への変換 | `travel_date` は `datetime.date` 型として返される。TOOL-001に渡す際は文字列化不要 |
| `model_dump_json()` | デバッグ・ログ出力 | `travel_date` は `"YYYY-MM-DD"` 形式の文字列として出力される |

### 3.2 TransportExpenseFormInputモデル

#### 基本情報

| 項目 | 値 |

#### フィールド定義

| フィールド名 | 型 | 必須/任意 | デフォルト値 | 説明 | バリデーション制約 |
|------------|-----|-----------|------------|------|-----------------|
| `applicant_name` | `str` | 必須 | なし | 申請者名（CF-001 Step1 で収集） | `validate_non_empty_string` |
| `application_date` | `date` | 必須 | なし | 申請日（システム日付） | `parse_date`（mode="before"） |
| `sections` | `list[TransportSection]` | 必須 | なし | 移動区間リスト（1件以上） | `model_validator`（mode="after"）で1件以上を確認する |
| `total_fare` | `int` | 必須 | なし | 交通費合計（円） | `parse_amount`（mode="before"）で非負整数チェック |

#### バリデーター

| バリデーター名 | 種別 | 対象フィールド | 処理内容 |
|-------------|------|-------------|---------|
| `validate_non_empty_string` | `@field_validator` | `applicant_name` | 空・500文字超チェック |

| `parse_amount` | `@field_validator(mode="before")` | `total_fare` | カンマ・「円」除去後に非負整数変換 |
| `validate_sections_not_empty` | `@model_validator(mode="after")` | `sections` | `len(self.sections) < 1` の場合 `ValueError("移動区間は1件以上入力してください")` |

#### シリアライズ仕様

| メソッド | 用途 | 注意事項 |
|--------|------|---------|
| `model_dump()` | TOOL-002 呼び出し引数への変換 | `sections` はネストされた辞書リストとして返される |
| `model_dump_json()` | ログ出力 | `application_date` は `"YYYY-MM-DD"` 形式で出力される |

## 4. invocation_state設計

### 4.1 受け渡しデータ定義

invocation_state は辞書リテラルで受け渡す（SYS-MA-001 6.1節準拠）。専用の Pydantic モデルは定義しない。

| キー名 | 型 | 送信元エージェント | 受信先エージェント | 説明 |
|--------|-----|----------------|----------------|------|
| `applicant_name` | `str` | AG-001 | AG-002, AG-003 | CF-001 Step1 で収集した申請者名 |
| `application_date` | `str` | AG-001 | AG-002, AG-003 | システム日付（`YYYY-MM-DD` 形式） |
| `session_id` | `str` | AG-001 | AG-002, AG-003 | セッションID（`sess_UUID` 形式）。AG-002/AG-003 は SessionManager 初期化にのみ使用する |

#### 送信側の構築例（AG-001）

```python
invocation_state = {
    "applicant_name": applicant_name,        # str
    "application_date": str(application_date),  # str（YYYY-MM-DD形式）
    "session_id": session_id                 # str（sess_UUID形式）
}
```

#### 受信側の参照例（AG-002/AG-003 の `handle_*_application` 関数）

```python
state = tool_context.invocation_state
applicant_name = state["applicant_name"]     # str
application_date = state["application_date"] # str（YYYY-MM-DD形式）
session_id = state["session_id"]             # str（SessionManagerのみに渡す）
```

## 6. エラーハンドリング

### 6.1 ValidationError処理フロー

```
ValidationError 発生（Pydantic v2）
  ↓
呼び出し元（エージェントまたはツール）で except ValidationError でキャッチ
  ↓
ErrorHandler.handle_input_error(message=e.errors()[0]["msg"]) に委譲
  ↓
日本語ユーザー向けエラーメッセージ生成
  ↓
社員へのメッセージ提示（再入力を促す）
  ↓

```

### 6.2 バリデーションエラーメッセージ

## 7. 補足情報

### 7.1 実装上の注意点

1. **Pydantic v2 専用実装**
   - `field_validator` に `mode="before"` を指定して入力値の前処理（正規化）を実施する
   - v1 の `@validator` は使用しない

2. **date 型のシリアライズ**
   - `model_dump()` で `datetime.date` 型が返る場合、JSON保存時は `.isoformat()` または `model_dump_json()` を使用して文字列化する

3. **経費区分の ValueError 非送出**
   - `normalize_expense_category` は判断不能時に `"その他経費"` を返し、`ValueError` を送出しない（BRL-12準拠）

4. **invocation_state は辞書リテラル**
   - invocation_state 専用の Pydantic モデルを定義しない（SYS-MA-001 6.1節準拠）

---

### 7.2 パフォーマンス考慮事項

1. **モデルのインスタンス化コスト**
   - ツール呼び出しのたびに Pydantic モデルがインスタンス化されるが、モデルのサイズが小さいため問題にならない
   - `CollectedTransportInfo.sections` に多数の区間が含まれる場合は `model_dump_json()` のシリアライズコストがわずかに増加する

## 9. テスト観点

### 9.1 正常系テスト
- `TransportSectionInput` が有効な引数（移動日・駅名・交通手段・業務目的）でインスタンス化されること
- `"渋谷駅"` が `normalize_station_name` により `"渋谷"` に正規化されること
- `"train"` が `normalize_transport_type` により `"電車"` に正規化されること
- `"10,000円"` が `parse_amount` により `10000`（int）に変換されること
- `"2026/07/03"` が `parse_date` により `date(2026, 7, 3)` に変換されること
- `ExpenseItem` の `expense_category="文房具"` が `"事務用品費"` に正規化されること
- `ExpenseItem` の `expense_category="雑費"` が `"その他経費"` にフォールバックすること（ValueError 非送出）
- `SessionState` の `hitl_result="ok"` が `"OK"` に正規化されること
- `CollectedTransportInfo` が `sections` 1件以上でインスタンス化されること

### 9.2 バリデーション異常系テスト
- `applicant_name=""` で `ValueError` が発生すること
- `applicant_name="a" * 501` で `ValueError`（500文字超）が発生すること
- `transport_type="自転車"` で `ValueError`（対象外交通手段）が発生すること
- `fare=-100` で `ValueError`（負数）が発生すること
- `travel_date="20260703"` で `ValueError`（形式不正）が発生すること（`python-dateutil` での解釈に依存）
- `CollectedTransportInfo(sections=[])` で `ValueError`（1件以上必須）が発生すること
- `SessionState(hitl_result="yes please")` で `ValueError`（無効な選択）が発生すること
- `departure="駅"` で `normalize_station_name` により空文字になり `ValueError` が発生すること

### 9.3 境界値テスト
- `applicant_name="a" * 500` でバリデーションが通過すること（500文字が上限）
- `fare=0` でバリデーションが通過すること（0円は許容）
- `sections=[TransportSection(...)]` の1件でバリデーションが通過すること
- `hitl_result=None` で `SessionState` がインスタンス化できること

### 9.4 シリアライズ・デシリアライズテスト
- `TransportSection.model_dump_json()` で `travel_date` が `"YYYY-MM-DD"` 形式の文字列として出力されること
- `SessionState.model_dump_json()` で `hitl_result=None` が `null` として出力されること
- JSON 文字列を `SessionState.model_validate_json()` でデシリアライズできること
