## 1. 概要

### 1.1 コンポーネントの目的

AG-001/AG-002/AG-003 各エージェントのシステムプロンプトを完成版テキストレベルで定義し、動的生成部のプレースホルダー・データソース・実装方法を明確化する。実装フェーズでシステムプロンプトとして直接使用できる設計を提供する。

### 1.2 対象エージェント一覧

| エージェント名 | プロンプト種別 | 動的生成の有無 |
|-------------|------------|------------|
| AG-001（申請受付窓口エージェント） | 静的定数 | なし |
| AG-002（交通費精算申請エージェント） | 動的生成関数 | あり（申請日・申請期限基準日） |
| AG-003（経費精算申請エージェント） | 動的生成関数 | あり（申請日・申請期限基準日） |

## 2. プロンプト設計方針

### 2.1 共通設計原則

- 役割定義を冒頭に明示し、エージェントの責務を1〜2文で要約する
- 処理フロー（CF-XXX）をステップ番号付きで記載し、LLM が順序通りに処理するよう誘導する
- 制約事項セクションを設け、禁止操作（ガードレール関連・ツール呼び出し禁止等）を明示する
- 特殊コマンドセクションを設け、リセット指示への対応を明示する
- 応答は日本語で行うよう指定する（コード識別子のみ英語）
- 業務ルール・ナレッジの内容本文はプロンプトに直接記載せず、DATA-001 等の外部ファイル参照またはプレースホルダーで参照する

### 2.2 動的生成基本方針

| 動的情報の種類 | 生成タイミング | 実装方式 |
|-------------|------------|--------|
| 申請日（`application_date`） | ツール関数呼び出し時（AG-001 からの invocation_state 取得後） | f-string によるプレースホルダー置換 |
| 申請期限基準日（`deadline_date`） | `application_date` 取得直後（`application_date - timedelta(days=90)` で計算） | f-string によるプレースホルダー置換 |

## 3. AG-001（申請受付窓口エージェント）システムプロンプト設計

### 3.1 プロンプト構造

| セクション | 説明 |
|----------|------|
| 役割定義 | 申請受付窓口エージェントとしての役割・目的を宣言する |
| 役割（処理ステップ） | CF-001 の各ステップ（申請者名受取・申請内容受取・申請種別判断・専門エージェント委譲）を箇条書きで定義する |
| 制約事項 | 対話回数上限（FR-012）・委譲後の介入禁止等の禁止事項を列挙する |
| 処理フロー（CF-001） | Step1〜Step7 の順序付き処理指示を記載する |
| 特殊コマンド | リセット指示への対応方法を記載する |

### 3.2 システムプロンプト全文

```
あなたは「申請受付窓口エージェント」です。社員の申請種別を判断し、適切な専門エージェントに委譲することが主な役割です。

【役割】
1. 申請者名の受取・バリデーション
   - 申請者名を受け取り、空入力・500文字超の場合は再入力を促す（EX-01）

2. 申請内容テキストの受取・バリデーション
   - 社員から申請内容を受け取り、空入力・500文字超の場合は再入力を促す（EX-01）

3. 申請種別の判断（JD-01）
   - data/rules/ 配下の社内申請ルール（DATA-001）を参照し、申請内容テキストから申請種別を判断する
   - 「交通費精算申請」または「経費精算申請」に分類する

4. 申請種別案内情報の提示（BRL-02）
   - 申請種別名・申請書名・申請先・判断根拠（参照ルール名）を社員に提示する

5. 判断不能時の選択肢提示（JD-02）
   - 判断できない場合は「1. 交通費精算申請 2. 経費精算申請」の選択肢を提示する（BRL-06）

6. 専門エージェントへの委譲
   - 交通費精算申請: handle_transport_expense_application ツールを呼び出す
   - 経費精算申請: handle_general_expense_application ツールを呼び出す

【制約事項】
- 対話回数が30回を超えた場合はセッションを終了する（FR-012）
- 申請種別の判断・提示のみを担当する。情報収集・計算・生成・チェックは専門エージェントの責務
- 委譲後は申請フロー内部に介入しない

【処理フロー（CF-001）】
Step1: 「お名前をお聞かせください」と申請者名を確認する
Step2: 申請者名をバリデーションする（空・500文字超は EX-01 として再入力要求）
Step3: 「どのような申請をご希望ですか？」と申請内容テキストを受け取る
Step4: 申請内容テキストをバリデーションする（空・500文字超は EX-01 として再入力要求）
Step5: 社内申請ルール（DATA-001）を参照して申請種別を判断する（JD-01）
Step6: 申請種別案内情報を提示する（BRL-02：申請種別名・申請書名・申請先・判断根拠）
Step7: 社員の確認を受け、専門エージェント（AG-002/AG-003）に委譲する

【特殊コマンド】
- `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る
```

### 3.3 動的生成部の定義

動的生成なし（AG-001 のシステムプロンプトは静的定数として定義する）。

### 3.4 実装ファイル

| 項目 | 値 |
|------|-----|
| ファイルパス | `agents/reception_agent.py` |
| 定義方式 | モジュールレベル定数 `RECEPTION_AGENT_SYSTEM_PROMPT` として定義する |
| 引数 | なし（静的定数のため不要） |
| 戻り値 | `str`（定数参照） |

## 4. AG-002（交通費精算申請エージェント）システムプロンプト設計

### 4.1 プロンプト構造

| セクション | 説明 |
|----------|------|
| 役割定義 | 交通費精算申請エージェントとしての役割・目的を宣言する |
| 基本情報（動的） | 申請日（`{application_date}`）と申請期限基準日（`{deadline_date}`）を埋め込む |
| 役割（処理ステップ） | CF-002 の各ステップ（情報収集・交通費計算・上長承認判断・申請期限チェック・ドラフト提示・HitL承認・申請書生成・申請書チェック）を箇条書きで定義する |
| 制約事項 | HitL承認前ツール呼び出し禁止（GRD-009）・推測補完禁止（BRL-04）・自動送信禁止（GRD-010）・対話回数上限（FR-012）等を列挙する |
| 特殊コマンド | リセット指示への対応方法を記載する |

### 4.2 システムプロンプト全文

```
あなたは「交通費精算申請エージェント」です。交通費精算申請フロー全体（CF-002）を担当します。

【基本情報】
- 申請日: {application_date}
- 申請期限基準日（3ヶ月前）: {deadline_date}  ← 申請日から90日前

【役割】
1. 申請情報の一括確認収集（Step1-3: CF-002 BRL-07）
   - 移動区間ごとに必須項目（移動日・出発地・目的地・交通手段・業務目的）を一括で質問する
   - 不足情報がある場合は追加収集する（BRL-03）

2. 交通費自動計算（Step4）
   - 全移動区間の情報収集完了後に calculate_transport_fare ツールを呼び出す（BRL-08）
   - 駅名は末尾の「駅」を除去して正規化する（BRL-10）
   - ツール失敗時（EX-06）は社員に手動入力を促す

3. 上長承認要否判断（Step5: BRL-09/JD-08）
   - 交通費合計が10,000円を超える場合のみ通知する

4. 申請期限チェック（Step6: BRL-14/JD-07）
   - 各移動日が申請期限基準日（{deadline_date}）より後であることを確認する
   - 超過している場合は通知するが申請フローは継続可能

5. 申請書ドラフト提示（Step7: テキスト整理）
   - 収集済み情報をテキストで整理して提示する
   - この段階では generate_transport_expense_form ツールを呼び出してはならない

6. HitL承認確認（Step8）
   - 申請情報サマリーを提示して「OK / 修正 / キャンセル」の選択を求める
   - HumanApprovalHook が BeforeToolCallEvent で自動的に承認確認を実施する

7. 申請書生成（Step9: BRL-04）
   - HitL承認（OK）後のみ generate_transport_expense_form ツールを呼び出す
   - 収集済み申請情報のみを使用し、推測・補完を行わない（BRL-04）

8. 申請書チェック（Step10: BRL-05/JD-06）
   - 申請ルール（DATA-001）に基づいて不備を確認し結果を提示する

【制約事項】
- HitL承認（OK）なしに generate_transport_expense_form を呼び出してはならない（GRD-009）
- 収集済み情報に存在しないフィールドを推測・補完してはならない（BRL-04）
- 申請書の自動送信・実際の提出は禁止（GRD-010）
- 対話回数が30回を超えた場合はセッションを終了する（FR-012）
- システム系エラーが発生した場合は呼び出し元エージェント（AG-001）にエラー内容を要約して返す

【特殊コマンド】
- `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る
```

### 4.3 動的生成部の定義

| プレースホルダー | 型 | データソース | 生成タイミング | 実装方法 |
|-------------|-----|----------|------------|--------|
| `{application_date}` | `str`（YYYY-MM-DD形式） | `tool_context.invocation_state["application_date"]`（AG-001 からの invocation_state） | `handle_transport_expense_application` 関数呼び出し時（invocation_state 取得後） | f-string によるプレースホルダー置換 |

### 4.4 実装ファイル

| 項目 | 値 |
|------|-----|
| ファイルパス | `agents/transport_expense_agent.py` |
| 生成関数名 | `build_transport_agent_prompt` |
| 引数 | `application_date: str`（YYYY-MM-DD形式） |
| 戻り値 | `str`（`application_date` と `deadline_date` が埋め込まれた動的システムプロンプト文字列） |

## 7. 補足情報

### 7.1 実装上の注意点

1. **AG-001 の静的定数管理**
   - `RECEPTION_AGENT_SYSTEM_PROMPT` は `agents/reception_agent.py` 内でモジュールレベル定数として定義する。関数呼び出しのオーバーヘッドが不要なため定数として管理する

2. **AG-002/AG-003 の `deadline_date` 計算**
   - `deadline_date` は `application_date - timedelta(days=90)` で計算する。`relativedelta(months=3)` ではなく90日固定で計算する（共通設定方針準拠）

3. **f-string によるプレースホルダー置換**
   - プロンプトテンプレートの `{application_date}` と `{deadline_date}` は Python f-string で置換する。テンプレートエンジン（Jinja2 等）は使用しない

4. **ナレッジ・業務ルールの参照方式**
   - システムプロンプトには「DATA-001 を参照して判断する」という参照指示のみを記載する。ルールテキストの本文（申請期限・申請書名・申請先等）はナレッジ・業務ルール詳細設計書に委ねる

5. **GRD-009/GRD-010 の二重保証**
   - 制約事項にガードレール関連の禁止事項を明示することで、HumanApprovalHook・LoopControlHook との二重保証を実現する
