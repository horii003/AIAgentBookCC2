## 1. ファイル概要

評価テスト方式設計で定義された評価戦略に基づき、実行可能な評価テストスクリプトを生成するために必要な詳細仕様を定義する。

| 評価レベル | 出力レベル | セッションレベル |
| 評価器種別 | LLM-as-Judge（二値）+ 決定的マッチ | LLM-as-Judge（二値） |
| 実行方式 | シングルターン | マルチターン |
| 対応指標 | MET-001 | MET-003 |
| スコア形式 | 二値（1.0/0.0） | 二値（1.0/0.0） |

## 2. 評価基盤構成

### 2.1 ファイル構成

```
evals/
├── helpers.py                          # 共通ヘルパー（テレメトリ管理・エージェント生成・シミュレーター等）
├── eval_tool_selection.py              # MET-001 評価スクリプト（申請種別判断正確率）
├── eval_goal_success_rate.py           # MET-003 評価スクリプト（業務ゴール達成率）
├── cases/
│   ├── tool_selection_cases.py         # MET-001 テストケース定義
│   └── goal_success_rate_cases.py      # MET-003 テストケース定義
└── logs/
    ├── eval_tool_selection_{timestamp}.log
    ├── eval_tool_selection_{timestamp}_report.json
    ├── eval_goal_success_rate_{timestamp}.log
    └── eval_goal_success_rate_{timestamp}_report.json
```

### 2.2 共通ヘルパーの機能一覧

| 機能カテゴリ | 目的 | 入力 | 出力 |
|------------|------|------|------|
| テレメトリ管理 | `strands_evals.telemetry.StrandsEvalsTelemetry` によるエージェント実行軌跡（ツール呼び出し・エージェント委譲）の記録・取得 | - | 軌跡データ（ツール呼び出し順序・スパン情報） |
| エージェント生成 | 本番同一構成（モデルID・システムプロンプト・ツール定義・ループ制御設定）での評価対象エージェント初期化 | セッション識別子・エージェント種別 | エージェントインスタンス |
| 状態管理 | テストケースごとの新規セッション生成・セッション状態初期化（`strands_evals.mappers.StrandsInMemorySessionMapper` を使用） | セッション識別子 | `SessionState` オブジェクト |
| フック制御 | `HumanApprovalHook` のブロッキング動作を自動承認（OK）に差し替えるパッチの適用 | - | - |
| 会話シミュレーション | テストケースのゴールに基づくマルチターン会話ループ実行（`strands_evals.ActorSimulator` による動的発話生成） | エージェントインスタンス・テストケース・セッション状態 | 会話履歴（ターンリスト） |

### 2.3 評価フレームワークから利用する概念

| 抽象概念 | 用途 | 採用フレームワークでの実装 |
|---------|------|------------------------|
| テストケース | テスト入力・メタデータの定義 | `strands_evals.Case`（`strands_evals` ライブラリ） |
| 評価器（有用性評価） | ゴール達成率のスコア算出 | `strands_evals.evaluators.GoalSuccessRateEvaluator`（`strands_evals` ライブラリ） |
| 評価器（ツール適切性評価） | ツール選択正確率のスコア算出 | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator`（`strands_evals` ライブラリ） |
| 実験 | バッチ実行単位（テストケース群 × 評価器群） | `strands_evals.Experiment`（`strands_evals` ライブラリ） |
| セッションマッパー | テレメトリ→評価可能構造への変換 | `strands_evals.mappers.StrandsInMemorySessionMapper`（`strands_evals` ライブラリ） |
| ユーザーシミュレーター | マルチターン会話の自動生成 | `strands_evals.ActorSimulator`（`strands_evals` ライブラリ） |
| テレメトリ | 実行軌跡の記録・取得 | `strands_evals.telemetry.StrandsEvalsTelemetry`（`strands_evals` ライブラリ） |

> **評価フレームワーク**: `strands_evals`（Strands Evals ライブラリ）
> **LLM-as-Judge 判定モデルID**: 要件上未定義（評価テスト方式設計 U-001）。評価対象（Claude Sonnet 4.5）とは別モデルを使用すること。

## 3. テストケース定義

### 3.1 テストケースの構成要素

| 要素 | 型 | 必須条件 | 説明 |
|------|---|---------|------|
| 識別名 | string | 必須 | テストケースの一意識別子（例: `TC_TOOL_001`） |
| 入力 | string | 必須 | 1ターン目のユーザー入力テキスト |
| タスク説明 | string | 必須 | テストの目的説明 |
| ゴール基準 | string | セッションレベル時に必須 | LLM-as-Judge が判定する達成基準 |
| 期待ツール | string | ツールレベル時に必須 | 期待されるツール関数名（エージェント識別子） |
| 期待軌跡 | list[string] | 軌跡レベル時に必須 | 期待されるツール呼び出し順序 |
| 期待出力 | string | 出力レベル時に必須 | 期待される最終応答（部分一致対象文字列） |
| メタデータ | dict | 任意 | 評価器が参照する追加情報（正解ラベル・シナリオ種別） |

## 4. 評価器設計

### 4.1 評価器一覧

| 評価スクリプト | 評価器種別 | 判定入力 | スコアリング方式 | 出力形式 |
|-------------|-----------|---------|---------------|---------|
| eval_tool_selection.py | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator`（ツール選択正確率評価） | AG-001 の応答・実行軌跡 + 期待ツール識別子 | 期待ツールが実際に呼び出された場合 1.0、呼び出されなかった場合 0.0 | `{score: float, pass: bool, reason: str}` |
| eval_goal_success_rate.py | `strands_evals.evaluators.GoalSuccessRateEvaluator`（ゴール達成率評価） | セッション全体の会話履歴 + ゴール基準テキスト | LLM が「達成/未達成」を判定し、達成 = 1.0、未達成 = 0.0 | `{score: float, pass: bool, reason: str}` |

### 4.2 LLM-as-Judge 評価器の設計（eval_goal_success_rate.py）

| 項目 | 値 |
|------|------|
| 評価器名 | `strands_evals.evaluators.GoalSuccessRateEvaluator` |
| 判定モデル | 要件上未定義（評価対象 Claude Sonnet 4.5 とは別モデルを使用する。評価テスト方式設計 U-001） |
| 判定観点 | テストケースのゴール基準（申請書ドラフト生成・チェック結果提示・通知メッセージ等）が会話履歴の中で達成されているかを判定する |
| スコア尺度 | 二値（達成: 1.0 / 未達成: 0.0） |
| 判定基準の参照元 | テストケースの `goal_criteria` フィールド |
| 判定根拠の出力 | `reason` 文字列として必ず出力する。スコアのみの判定は禁止 |
| temperature | 0（決定性を高めるため） |
| max_tokens | 要件上未定義 |

#### 判定プロンプト（eval_goal_success_rate.py）

```
以下の会話履歴と達成基準をもとに、申請フローのゴールが達成されたかを判定してください。
達成された場合は「ACHIEVED」、達成されていない場合は「NOT_ACHIEVED」とのみ答え、続けて判定理由を記述してください。

達成基準:
{goal_criteria}

会話履歴:
{conversation_history}

判定:
```

### 4.3 軌跡マッチ評価器の設計

eval_tool_selection.py および eval_goal_success_rate.py では独自の軌跡マッチ評価器は使用しない（MET-001: `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` によるツール選択正確率評価、MET-003: `strands_evals.evaluators.GoalSuccessRateEvaluator` によるセッションレベルのゴール達成判定）。

ツール利用の適切性（AG-001 委譲精度）の補助的確認は `strands_evals.mappers.StrandsInMemorySessionMapper` で実行軌跡からツール呼び出し列を取得し、`eval_goal_success_rate.py` のログに記録する。

### 4.4 決定的マッチ評価器の設計（eval_tool_selection.py）

| 項目 | 値 |
|------|------|
| 評価器名 | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` |
| マッチ方式 | ツール呼び出し軌跡から期待ツール識別子との一致を検証 |
| 比較対象 | AG-001 の実行軌跡（ツール呼び出し順序）および応答テキスト |
| スコア算出 | 期待ツール（`expected_tool`）が実際に呼び出された場合 1.0、呼び出されなかった場合 0.0 |
| reason 出力 | マッチしたツール識別子 or 不一致の理由を `reason` に記録する |

### 4.5 カスタム評価器の設計

eval_tool_selection.py および eval_goal_success_rate.py の現行スコープではカスタム評価器は使用しない。

> 申請書 Excel ファイルの生成有無確認・必須項目充足確認等のカスタム評価は、将来拡張として `eval_goal_success_rate.py` に追加する（評価テスト方式設計 3.3 参照）。

## 5. タスク実行設計

### 5.1 シングルターン実行フロー（eval_tool_selection.py）

| 順序 | 処理概要 | 入力 | 出力 |
|------|---------|------|------|
| [1] | テレメトリ初期化（前ケースの軌跡をクリア） | - | 初期化済み `strands_evals.telemetry.StrandsEvalsTelemetry` |
| [2] | セッション生成（テストケースごとに新規 session_id を発行） | テストケース識別名 | `session_id`（`sess_` + UUID形式） |
| [3] | 評価対象エージェント生成（AG-001 を本番同一構成で初期化） | `session_id` | AG-001 エージェントインスタンス |
| [4] | 申請者名の事前設定（会話履歴に名前収集ターンを事前注入） | 仮申請者名（`評価テスト太郎`） | 設定済みエージェント |
| [5] | エージェント呼び出し（テストケースの入力テキストを1ターン送信） | ユーザー入力テキスト | AG-001 の応答テキスト |
| [6] | 評価器スコアリング（`strands_evals.evaluators.ToolSelectionAccuracyEvaluator` でツール選択検証） | 応答テキスト・実行軌跡 + 期待ツール識別子 | `{score, pass, reason}` |
| [7] | 結果構築・ログ記録 | スコア・合否・判定理由 | ケース単位の評価結果 |

### 5.2 マルチターン実行フロー（eval_goal_success_rate.py）

| 順序 | 処理概要 | 入力 | 出力 |
|------|---------|------|------|
| [1] | テレメトリ初期化（前ケースの軌跡をクリア） | - | 初期化済み `strands_evals.telemetry.StrandsEvalsTelemetry` |
| [2] | セッション生成（テストケースごとに新規 session_id を発行） | テストケース識別名 | `session_id`（`sess_` + UUID形式） |
| [3] | 評価対象エージェント生成（AG-001 を本番同一構成で初期化。HumanApprovalHook は自動承認パッチ適用） | `session_id` | AG-001 エージェントインスタンス |
| [4] | ユーザーシミュレーター生成（テストケースのゴールと入力テキストを設定） | テストケース・最大ターン数（30） | `strands_evals.ActorSimulator` インスタンス |
| [5] | 会話ループ実行（会話終了条件を満たすまでシミュレーターが発話を生成） | AG-001 インスタンス・`strands_evals.ActorSimulator`・セッション状態 | 会話履歴（ターンリスト） |
| [6] | 実行軌跡の取得（`strands_evals.mappers.StrandsInMemorySessionMapper` でスパンからツール呼び出し列を構築） | テレメトリデータ | 軌跡構造（ツール呼び出し順序） |
| [7] | 評価器スコアリング（`strands_evals.evaluators.GoalSuccessRateEvaluator` で LLM-as-Judge 判定） | 会話履歴 + ゴール基準テキスト | `{score, pass, reason}` |
| [8] | 結果構築・ログ記録 | スコア・合否・判定理由・軌跡 | ケース単位の評価結果 |

### 5.3 ユーザーシミュレーター設計（マルチターン）

| 項目 | 値 |
|------|------|
| シミュレーターの役割 | テストケースのゴール基準と1ターン目の入力テキストに基づき、申請フローの各ステップに対応した動的なユーザー発話を生成する。AG-001 の応答に基づいて次ターンの発話を LLM で生成する |
| 使用クラス | `strands_evals.ActorSimulator` |
| 最大ターン数 | 30ターン（FR-012 のセッション対話回数上限に合わせる） |
| 会話終了条件 | ゴール達成判定（申請書ドラフト生成完了かつチェック結果受領を示す応答を検知）または最大ターン数（30）到達 |
| エージェント例外時の動作 | 例外を記録して会話ループを終了し、当該ケースを評価失敗（score=0.0）として処理する |
| シミュレーター例外時の動作 | 例外を記録して会話ループを終了し、当該ケースを評価失敗（score=0.0）として処理する |
| シミュレーターモデル | 要件上未定義（評価対象 Claude Sonnet 4.5 とは別モデルを推奨） |

### 5.4 エージェント生成方針

| 項目 | 方針 |
|------|------|
| 本番との同一性 | モデルID・システムプロンプト・ツール定義・ループ制御設定（LoopControlHook 含む）を本番と一致させる |
| テレメトリ属性 | セッション識別子（`session_id`）をスパン属性として付与し、ケース間の軌跡混入を防止する |
| ストリーミング | 無効（評価時は同期実行） |
| ブロッキング処理 | `HumanApprovalHook` のブロッキング動作を自動承認（OK）に差し替える評価用パッチを適用する（SD-08 7.2節準拠） |
| ログ設定 | UTF-8 対応・ファイル出力（`evals/logs/`）。Strands Agents / botocore 等の SDK ログは WARNING レベル以上のみ出力する |

## 6. 実験実行設計

### 6.1 実験構成

| 評価スクリプト | テストケース群 | 評価器群 | タスク関数 |
|-------------|-------------|---------|----------|
| `evals/eval_tool_selection.py` | `evals/cases/tool_selection_cases.py` の全ケース（`TC_TOOL_001〜003`、計3件） | `strands_evals.evaluators.ToolSelectionAccuracyEvaluator` | `run_single_turn_task` |
| `evals/eval_goal_success_rate.py` | `evals/cases/goal_success_rate_cases.py` の全ケース（`TC_GOAL_001〜003`、計3件） | `strands_evals.evaluators.GoalSuccessRateEvaluator` | `run_multi_turn_task` |

### 6.2 実行順序

| 順序 | 処理 | 説明 |
|------|------|------|
| [1] | 環境初期化 | 環境変数（`AWS_REGION` 等）・ログ設定・依存関係（`strands_evals`・Strands Agents）の準備 |
| [2] | 評価器初期化 | 評価器インスタンスの生成（LLM-as-Judge 用の判定モデル指定含む） |
| [3] | 実験構成 | `strands_evals.Experiment` を使用して、テストケース群 × 評価器群 × タスク関数を組み立て |
| [4] | 評価実行 | 各テストケースに対してタスク関数実行 → 評価器スコアリング → 結果記録 |
| [5] | レポート保存 | 全ケースの評価結果（スコア・合否・判定理由・全体スコア）を JSON ファイルに永続化 |
| [6] | レポート表示 | 全体スコア・個別スコア・合否判定・判定理由をコンソールに出力 |

### 6.3 非決定性への対処

#### 実行レベルの対処（同一ケースの複数回実行）

| 項目 | 値 |
|------|------|
| 実行回数 | 1回（評価テスト方式設計 U-004 確定済み。各テストケースは1度のみ実行し、各テストケースごとの集計はしない） |
| スコア集約方式 | 集約なし（1回実行のため単一スコアをそのまま使用） |
| 合否判定の適用先 | 各ケースのスコアから `overall_score`（全ケース平均）を算出し、閾値と比較する |

#### ケースレベルの対処（テストケースの多様性確保）

| 項目 | 値 |
|------|------|
| 各評価観点の最低ケース件数 | MET-001: 交通費・経費の各カテゴリで最低3件以上（TC_TOOL_001〜003）。MET-003: 交通費申請・経費申請の各E2Eシナリオで最低3件以上（TC_GOAL_001〜003）（評価テスト方式設計 5.2 準拠） |
| ケースの多様性方針 | 交通手段・経費種別・業務目的・金額・移動区間等、入力表現とシナリオを変えた複数ケースを設計する |

## 9. 回帰評価設計

### 9.1 回帰評価トリガーと実行対象

| トリガー | 実行対象スクリプト | 実施内容 |
|---------|-----------------|---------|
| システムプロンプト変更（`agents/reception_agent.py`・`agents/transport_expense_agent.py`・`agents/expense_agent.py` 変更時） | `eval_tool_selection.py`・`eval_goal_success_rate.py`（全ケース） | 全テストケースを再実行し、変更前スコアと比較する |
| ナレッジ・業務ルール変更（`knowledge/` 配下・`data/rules/` 配下の変更時） | `eval_tool_selection.py`・`eval_goal_success_rate.py`（全ケース） | 全テストケースを再実行する |
| モデルバージョン変更（`config/model_config.py` の `MODEL_ID` 変更時） | `eval_tool_selection.py`・`eval_goal_success_rate.py`（全ケース） | 全テストケースを再実行し、閾値（MET-001 ≥0.95・MET-003 ≥0.90）を確認する |
| 評価フレームワークバージョン変更（Strands Agents・`strands_evals` バージョン変更時） | `eval_tool_selection.py`・`eval_goal_success_rate.py`（全ケース） | 全テストケースを再実行し、評価器の動作（テレメトリ取得・LLM-as-Judge 判定）を確認する |

### 9.2 回帰判定基準

| 項目 | 基準 |
|------|------|
| 合格条件 | MET-001 `overall_score >= 0.95` かつ MET-003 `overall_score >= 0.90` を維持すること |
| 劣化検知基準 | 前回実行との比較でスコア低下幅が 0.05 超の場合に劣化として検知する |
| 不合格時の対応 | 即座に開発チームへ報告し、原因（プロンプト変更・ナレッジ変更・モデル差異等）を分析する |

## 10. テストケースの管理方針

| 項目 | 方針 |
|------|------|
| 配置場所 | `evals/cases/tool_selection_cases.py`・`evals/cases/goal_success_rate_cases.py` に Python モジュール（定数リスト）として定義する |
