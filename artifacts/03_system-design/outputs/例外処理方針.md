# 例外処理方針

> **参照元（システム要件定義資料）:**

## 1. 目的・適用範囲

### 1.1 目的

システム全体として **「例外とは何か」「どう分類し」「どう収束させるか」** の共通方針を定義する

個々のエージェント設計に **一貫した判断軸** を提供する

### 1.2 適用範囲

- マルチエージェントを含む全システム（AG-001/AG-002/AG-003・TOOL-001/TOOL-002・セッションマネージャ）
- 業務処理、外部連携（Amazon Bedrock）、UI/UX（CLIチャット）、運用を含む

### 1.3 本方針で定義しないこと

- エージェント内部の例外検知ロジック
- SDK／ライブラリ依存の例外クラス設計
- 具体的な再試行アルゴリズム

---

## 2. 例外処理の基本原則（システム共通）

- **例外は必ず収束させる（無限リトライ禁止）**
- **失敗は状態として管理する**
- **例外＝即エラー表示ではない**
- **ユーザーに内部失敗を見せない**
- **後続処理に影響を残さない**


---

## 3. 例外の定義と分類（システム観点）

### 3.1 例外の定義

「通常の業務フローから逸脱し、継続判断が必要な状態」

### 3.2 例外分類

| 分類ID | 分類名 | 説明 | 例 |
|-------|-------|------|---|
| EX-01 | 入力例外 | 入力不正・不足 | 申請内容テキスト空入力・500文字超過・対話回数30回超過・CVV検知（GRD-002） |
| EX-02 | 判断不能 | AIが結論を出せない | 申請種別判断不能（JD-02）・社員回答が不明瞭で再質問が必要・領収書画像解析失敗（LLMビジョン） |
| EX-03 | 業務ルール違反 | 定義済ルール不一致 | 申請期限超過（BRL-14）・上長承認要否条件充足（BRL-09/BRL-13）・申請書不備検知（JD-06） |
| EX-04 | 外部I/F例外 | API障害・タイムアウト | Amazon Bedrock 推論エラー・タイムアウト |
| EX-05 | 実行不可 | 権限・状態不整合 | 権限のないツール呼び出し（GRD-009）・申請書自動送信禁止違反（GRD-010） |
| EX-06 | データ不整合 | データ形式不正 | 社内申請ルールファイル（DATA-001）参照失敗・申請書テンプレート（DATA-002）参照失敗・運賃データ（DATA-003/004）参照失敗 |
| EX-07 | システム障害 | 基盤障害 | セッション状態ファイル（DATA-005）読み書き失敗・ログファイル書き込み失敗 |
| EX-08 | 想定外 | 未分類事象 | 上記に分類できない予期しない例外 |

## 4. 例外検知レベル

### 4.1 検知レベルの考え方

| レベル | 説明 | 検知主体 |
|-------|------|---------|
| 業務レベル | 業務定義上の異常（入力不正・申請ルール違反・申請種別判断不能等） | エージェント（AG-001/AG-002/AG-003） |
| 処理レベル | 処理失敗（ツール実行エラー・ファイル参照失敗等） | ツール（TOOL-001/TOOL-002） |
| システムレベル | 基盤・インフラ異常（LLM障害・セッション障害） | ErrorHandler |

### 4.2 検知責務

- 検知は **エージェントまたはツール**
- **判断・収束はErrorHandler** が行う

## 5. 例外発生時の基本対応フロー

### 5.1 共通対応フロー

1. 例外検知（エージェント/ツール）
2. 例外分類（ErrorHandler）
3. 継続可否判定（ErrorHandler）
4. 処理分岐（再試行／代替／停止）
5. 状態更新（セッション管理）
6. ログ・監査記録（ErrorHandler）

## 7. 例外ハンドリング実装

### 7.1 ErrorHandlerの責務方針

ErrorHandlerは以下の責務を担う。

- 例外分類（EX-01〜EX-08）ごとに継続可否を判定し、対応方針（再試行・代替処理・エスカレーション・中断）を決定する
- ユーザー向けの日本語メッセージを生成し、技術的な詳細を隠蔽する
- 例外発生時のログ出力を行い、スタックトレースはログにのみ記録する

**エラー伝播の責務分担**:

| 主体 | 責務 |
|-----|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す |
| 呼び出し元（エージェント・ハンドラー） | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |

具体的なメソッド設計（メソッド名・引数・戻り値）は基本設計（ハンドラー基本設計.md）で定義する。

> ※ 基本設計書への参照: ハンドラー基本設計.md 2.3節


### 7.2 エラーメッセージ設計方針

**ユーザーフレンドリー**:
- 技術的な詳細（スタックトレース・内部エラーコード）を隠蔽する
- 具体的な対処方法（再入力・担当部門への問い合わせ等）を提示する
- 丁寧な言葉遣いで日本語メッセージを生成する

**情報の充実**:
- エラーの原因を業務的な言葉で説明する
- 次のアクション（再入力・続行・問い合わせ先）を必ず提示する
- 必要に応じて担当部門への連絡を促す

**例**:
```
（入力例外：500文字超過時）
「入力内容が長すぎます。500文字以内で申請内容を入力してください。」

（外部I/F例外・システム障害時）
「申請処理中にエラーが発生しました。しばらく経ってから再度お試しいただくか、担当部門にお問い合わせください。」

（データ不整合：電車経路テーブル検索失敗時）
「入力された経路の運賃データが見つかりませんでした。交通費を手動で入力してください。」
```

## 8. 例外とエスカレーションの関係

### 8.1 エスカレーション対象例外

| 例外分類 | エスカレーション要否 | 行き先 |
|---------|------------------|-------|
| EX-04（外部I/F例外） | 要 | 社員への担当部門問い合わせ案内 |
| EX-05（実行不可） | 要 | 社員への担当部門問い合わせ案内 |
| EX-06（データ不整合）のうちテンプレート参照失敗 | 要 | 社員への担当部門問い合わせ案内 |
| EX-07（システム障害） | 要 | 社員への担当部門問い合わせ案内・ERR-003/004/005（エラーログ）記録 |
| EX-08（想定外） | 要 | 社員への担当部門問い合わせ案内 |

### 8.2 エスカレーション判定責務

**判定主体**: ErrorHandler

**判定基準**: 代替処理・再試行で継続不可と判断した場合にエスカレーションを実行する。エスカレーション先は常に「社員への担当部門問い合わせ案内」とする（本システムに自動エスカレーション先の外部システムなし）

## 9. セッション状態への反映方針

### 9.1 例外時の状態更新

| 状態 | 説明 | トリガー |
|-----|------|---------|
| ACTIVE | 処理中（代替処理で継続可能な例外後） | EX-01/EX-02/EX-03の代替処理後に継続 |
| WAITING | 人対応待ち（社員入力待ち） | EX-01（再入力要求）・EX-02（選択肢提示・再質問）・EX-03（通知後の社員確認待ち） |
| TERMINATED | 異常終了 | EX-04/EX-05/EX-07/EX-08・対話回数30回超過（EX-01）・代替処理不可のEX-06 |

### 9.2 再開可否

**再開可能な例外**:
- EX-01（入力例外）：対話回数30回未満の場合、WAITING状態から再開可能
- EX-02（判断不能）：代替処理（選択肢提示・再質問）後にACTIVEから継続可能
- EX-03（業務ルール違反）：通知後に社員が継続意思を示した場合、ACTIVEから継続可能
- EX-06のうち電車経路テーブル検索失敗：手動入力後にACTIVEから継続可能

**再開不可な例外**:
- EX-04（外部I/F例外）：Amazon Bedrock エラー → TERMINATED
- EX-05（実行不可）：権限外操作 → TERMINATED
- EX-06のうちテンプレート参照失敗 → TERMINATED
- EX-07（システム障害） → TERMINATED
- EX-08（想定外） → TERMINATED
- EX-01の対話回数30回超過 → TERMINATED

## 10. UX観点の扱い（概要）

**利用者への通知有無**:
- EX-01〜EX-03：必ず社員に通知する（業務フローの継続・修正に必要な情報を提供する）
- EX-04〜EX-08：社員に抽象化したエラーメッセージと担当部門問い合わせ先を通知する

**エラーメッセージの抽象度**:
- 技術的なエラー詳細（スタックトレース・内部エラーコード・ファイルパス等）は表示しない
- 業務的な言葉で「何が起きたか」「どう対処するか」を提示する

**次の行動を必ず提示**:
- 再入力可能な場合は再入力を促すメッセージを表示する
- 中断が必要な場合は担当部門への問い合わせ先を案内する
- キャンセル可能な場合はキャンセル手順を提示する

## 11. ログ・監査方針

### 11.1 例外ログ

| 項目 | 内容 | 出力先 |
|-----|------|-------|
| error_type | エラー分類（EX-01〜EX-08） | DATA-007（判断ログ）・コンソール |
| error_message | ユーザーへの通知メッセージ（抽象化済み） | DATA-007（判断ログ） |
| context | エラーコンテキスト（request_id・agent_id・step_id） | DATA-007（判断ログ） |
| exc_info | スタックトレース（技術的詳細） | DATA-007（判断ログ）のみ（ユーザーには非表示） |
| timestamp | タイムスタンプ | DATA-007（判断ログ） |


### 11.2 監査観点

**再現可能性**:
- request_id・step_id・agent_id・tool_call_idを全ログに付与し、例外発生時の処理経路を追跡可能にする

**判断経路の説明可能性**:
- 例外分類・継続可否判定の根拠・代替処理の内容をDATA-007（判断ログ）に記録し、事後の監査・検証に対応する

### 11.3 ログ設計方針

ログ出力に関して、以下の設計方針に従う（共通設定方針 SYS-CONFIG-001 7b節・ログ出力方式設計 SYS-LOG-001準拠）。

- ログ設定（レベル・出力先・フォーマット）はエントリーポイント（`main.py`）で一元管理する
- ログ出力は各モジュール（エージェント・ツール・ErrorHandler等）が責任を持って行う
- ログレベルは環境変数（`LOG_LEVEL`）で制御し、未設定時のデフォルトは `WARNING` とする
- ユーザー向けメッセージ（申請者への応答）と技術ログ（デバッグ・監査用）は分離する
- 各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する

## 12. 設計上の留意点・リスク

**例外過多によるUX低下**:
- EX-01（入力例外）が頻発すると社員の対話体験が損なわれる
- 対策: 入力ガイダンスを明確にし、不必要なバリデーションエラーを削減する

**例外の握り潰し**:
- EX-04/EX-07等の重大例外を処理せずに継続すると、不整合な状態で処理が進行するリスクがある
- 対策: 全例外をErrorHandlerで受け取り、握り潰しを禁止する。分類不能な場合はEX-08として処理する

**想定外例外の扱い**:
- EX-08（想定外）に分類される例外が設計時に想定されないパターンで発生するリスクがある
- 対策: EX-08は必ずTERMINATED状態に遷移させ、エスカレーション案内を行う。事後にERR-004/ERR-005ログを分析してEX-01〜EX-07への再分類を検討する
