## 1. 目的・設計方針

### 1.1 目的

- AIエージェントが **セッション単位で保持・参照すべき情報の範囲と方針**を定義する
- 会話継続、再開、引き継ぎ、監査に耐える **一貫した状態管理** を実現する

### 1.2 本方針で定義すること

- セッションの単位・ライフサイクル
- 管理対象情報（会話履歴、判断結果、成果物参照 等）
- メモリの種類と役割分担
- 再開・中断・終了時の扱い

### 1.3 本方針で定義しないこと

- メモリの具体的な保存技術（セッションマネージャ基本設計書・詳細設計書に委ねる）
- エージェント内部の推論用メモリ構造
- 実装クラス／SDK依存設計

## 2. セッションの定義

### 2.1 セッションとは

**本システムにおける「セッション」の定義**:
- 社員が1件の申請フロー（交通費精算申請 または 経費精算申請）を開始してから完了・中断・キャンセルするまでの一連の対話の単位
- 申請フロー全体（CF-001 → CF-002 または CF-003）を1セッションとして管理する

**セッションの範囲**:
- AG-001による申請受付（CF-001開始）からAG-002/AG-003による申請書提示・チェック完了（CF-002/CF-003完了）まで

### 2.2 セッション識別子

| 項目 | 内容 | 形式 | 例 |
|-----|------|------|---|
| session_id | セッション識別子 | UUID形式（ランダム生成） | `sess_xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` |
| request_id | 申請フロー全体の相関ID（ログ追跡用） | UUID形式（セッション開始時に生成） | `req_xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` |

## 3. セッションライフサイクル

### 3.1 状態遷移

| 状態 | 説明 | トリガー |
|-----|------|---------|
| CREATED | セッション生成 | 社員がシステムに接続しAG-001が申請者名を受け付けた時点 |
| ACTIVE | 処理中 | エージェントまたは社員が対話を継続している |
| WAITING | 人間対応待ち | HitL承認確認提示後・社員回答待ち・判断不能時の選択肢提示後 |
| CLOSED | 正常終了 | 申請書ドラフト生成・チェック結果提示完了、または社員がキャンセル選択 |
| TERMINATED | 異常終了 | 対話回数30回上限到達・システムエラー発生・セッション障害 |

### 3.2 状態遷移条件

**状態遷移のトリガー**:
- 申請者名受け付け完了: CREATED → ACTIVE
- エージェントが応答・社員が入力: ACTIVE のまま継続
- HitL承認確認提示 / JD-02（選択肢提示）: ACTIVE → WAITING
- 社員がHitL承認結果を選択 / 申請種別を選択: WAITING → ACTIVE
- 申請書ドラフト生成・チェック結果提示完了 / 社員がキャンセル選択: ACTIVE → CLOSED
- 対話回数30回到達（FR-012）/ システムエラー: ACTIVE or WAITING → TERMINATED

**状態管理責務**:
- セッション状態の読み書きはセッションマネージャ（APP-06）が担当する
- 各エージェント（AG-001/AG-002/AG-003）はセッションマネージャを通じて状態を更新する

## 4. セッションで管理する情報の分類

### 4.1 管理情報カテゴリ

| カテゴリ | 説明 | 保存場所 |
|---------|------|---------|
| 会話履歴 | 各ターンの発話内容（要約）・適用ルール・話者情報 | DATA-006（会話ログ・ファイル）／セッション状態ファイル（DATA-005） |
| 業務コンテキスト | 申請者名・申請種別案内情報・収集済み申請情報・申請書ドラフト参照 | DATA-005（セッション状態ファイル） |
| 実行履歴 | ツール呼び出し結果・委譲記録・HitL承認結果 | DATA-005（セッション状態ファイル）／DATA-007（判断ログ） |
| 成果物参照 | 申請書ドラフトのファイルパス・チェック結果参照ID | DATA-005（セッション状態ファイル） |
| 状態管理 | セッション状態（CREATED/ACTIVE/WAITING/CLOSED/TERMINATED）・対話回数・現在ステップ | DATA-005（セッション状態ファイル） |

## 5. 会話履歴管理方針

### 5.1 会話履歴の範囲

**保存対象**:
- 各ターンの発話要約（話者・発話目的・要約テキスト）
- 適用したルールID（BRL-xx / JD-xx）
- 判断結果（申請種別・チェック結果等）

**保存しないもの**:
- 申請内容テキスト・社員回答の全文（機密情報のためマスキング後に要約のみ保存）
- エージェントの内部推論（Chain-of-Thought）


### 5.2 会話履歴の粒度

| 粒度 | 方針 |
|-----|------|
| フル履歴 | 保存しない（機密情報保護・サイズ削減のため） |
| 要約 | 各ターンの発話を要約して保存する（OPE-001準拠） |
| 重要発言抽出 | 判断結果・申請種別確定・HitL承認結果は個別に記録する（AUD-001〜004準拠） |

---

## 6. 業務コンテキスト管理方針

### 6.1 管理対象

- 申請者名（applicant_name）
- 申請種別案内情報（application_type_info）：申請種別名・申請書名・申請先・判断根拠
- 収集済み申請情報（collected_application_data）：各必須項目の値
- HitL承認結果（hitl_approval_result）：OK / 修正 / キャンセル
- 上長承認要否通知フラグ（approval_required_flag）
- 申請書ドラフトファイルパス（draft_file_path）
- 申請書チェック結果（check_result）


### 6.2 更新ルール

**誰が更新できるか**:
- AG-001：申請者名・申請種別案内情報を書き込む
- AG-002：交通費精算に関する収集済み申請情報・HitL承認結果・申請書ドラフト参照を書き込む
- AG-003：経費精算に関する収集済み申請情報・HitL承認結果・申請書ドラフト参照を書き込む
- セッションマネージャ：セッション状態・対話回数を書き込む

**上書き／追記のルール**:
- 申請者名・申請種別案内情報は上書き禁止（AG-001が確定した値を変更しない）
- 収集済み申請情報は社員の修正回答があった場合のみ上書きを許可する
- HitL承認結果は新しい確認ごとに上書きする

**差分管理の考え方**:
- セッション状態ファイル（DATA-005）は申請フロー全体で1ファイルとして管理する
- 変更履歴の個別管理は行わない（ログファイルで事後追跡可能）

## 7. 実行・判断履歴管理方針

### 7.1 実行履歴

| 項目 | 内容 | 保存場所 |
|-----|------|---------|
| tool_name | ツール名（TOOL-001 / TOOL-002） | DATA-005（セッション状態ファイル）・DATA-007（判断ログ） |
| 実行内容 | ツールパラメータの要約 | DATA-007（判断ログ） |
| 実行結果 | ツール戻り値の要約（交通費金額・申請書ドラフトパス等） | DATA-007（判断ログ） |
| 実行時刻 | タイムスタンプ | DATA-007（判断ログ） |
| 再実行可否 | 冪等性あり（同一入力で同一出力を保証） | - |

### 7.2 判断履歴

**ルーティング結果**:
- AG-001による申請種別判断結果（JD-01/JD-02適用）・適用ルールIDをDATA-007（判断ログ）に記録する（AUD-001準拠）

**エスカレーション判定**:
- 申請ルール準拠チェック結果・期限超過チェック結果をDATA-007（判断ログ）に記録する（AUD-002/AUD-003準拠）

**人間承認の結果**:
- HitL承認結果・上長承認要否判断をDATA-005（セッション状態ファイル）に保存し、DATA-007（判断ログ）にも記録する（AUD-004準拠）

## 8. 成果物・ナレッジ参照管理

### 8.1 管理対象

- 生成ドキュメントID: 申請書ドラフトのファイルパス（Excel生成後に保存）
- 参照ナレッジID: 社内申請ルール（DATA-001）・申請書テンプレート（DATA-002）の参照先パス
- 外部I/F参照ID: 外部I/Fなし（ファイル直接参照のみ）

### 8.2 管理方針

**実体は保持しない**:
- 申請書ドラフト（Excel）・社内申請ルール・テンプレートの実体はセッション状態ファイルには保持しない

**参照IDのみを保持**:
- 申請書ドラフトのファイルパス・参照したデータファイルのパスのみをセッション状態ファイルに記録する

**再取得可能性を前提**:
- ファイルパスが保持されていれば、セッション再開時に申請書ドラフトを再参照可能な設計とする

## 9. メモリの種類と役割分担（概念）

### 9.1 メモリ分類

| 種類 | 用途 | 実装 |
|-----|------|------|
| セッションメモリ | 申請フロー全体の状態・業務コンテキスト・対話回数を保持する | ファイル永続化（DATA-005）；Strands Agents のセッション管理機能を使用 |
| ワーキングメモリ | エージェントが1ターンの推論・ツール呼び出しに使用する一時的な情報 | エージェント内部メモリ（フレームワーク管理）；セッション終了時に破棄 |
| ロングターム参照 | 社内申請ルール・運賃データ等の永続的な参照情報 | ファイルストレージ（DATA-001〜004）；起動時またはアクセス時にロード |

## 10. 再開・引き継ぎ方針

### 10.1 再開条件

- セッション状態ファイル（DATA-005）が存在する: 前回の中断箇所から再開可能
- セッション状態が ACTIVE または WAITING: 対応するエージェントが継続処理を行う
- セッション状態が CLOSED または TERMINATED: 再開不可（新規セッションを開始する）

### 10.2 再開時に復元する情報

| 情報 | 復元要否 | 実装状況 |
|-----|---------|---------|
| 申請者名 | 必須 | DATA-005から復元 |
| 申請種別案内情報（D-004） | 必須 | DATA-005から復元 |
| 収集済み申請情報（D-007） | 必須 | DATA-005から復元 |
| 現在のセッション状態 | 必須 | DATA-005から復元 |
| 現在ステップ（会話フロー位置） | 必須 | DATA-005から復元 |
| 対話回数 | 必須 | DATA-005から復元 |
| HitL承認結果 | 必須（存在する場合） | DATA-005から復元 |
| 申請書ドラフトファイルパス | 必須（存在する場合） | DATA-005から復元 |
| 会話履歴 | 任意（要約形式） | DATA-006から参照（必須ではない） |

## 11. エスカレーション・承認との連携

**エスカレーション発生時に保持すべき情報**:
- セッション状態を WAITING に設定する
- エスカレーション理由・対話回数・現在ステップをDATA-005に記録する

**人間承認時に提示する情報**:
- 収集済み申請情報のサマリー（全必須項目の値）
- 申請種別・申請者名
- 申請書ドラフトの内容確認（HitL承認時）

**承認結果の反映方法**:
- 承認（OK）: HitL承認結果をDATA-005に記録し、申請書生成（TOOL-002）を実行する
- 修正要望: セッション状態を ACTIVE に戻し、情報収集ステップへ戻る
- 拒否（キャンセル）: セッション状態を CLOSED に更新し、申請フローを終了する

## 12. セキュリティ・ガードレール観点

**個人情報の保持方針**:
- 本システムでは個人情報（D-3分類）は取り扱わない（自律度・権限定義.md 3.1節準拠）
- 収集済み申請情報（D-2：機密）はセッション状態ファイル（DATA-005）にマスキングなしで保存する

**セッション内秘匿情報の扱い**:
- 申請内容テキスト・社員回答（D-2：機密）はセッション状態ファイルに保存する
- セッション終了後は関係者以外がアクセスできないようファイル権限を管理する

**マスキング／削除方針**:
- 会話ログ（DATA-006）に書き込む際は申請内容テキスト・社員回答をマスキングした要約のみを記録する（GRD-002/GRD-007準拠）
- CVV（クレジットカードセキュリティコード）は入力・出力両方でブロックし、セッション状態に保存しない（GRD-002/GRD-007準拠）
- セッション状態ファイル（DATA-005）自体の保存期間は要件上未定義

## 13. ログ・監査との関係

**セッション情報とログの関係**:
- セッション内のすべての操作に request_id を付与し、ログ間の相関追跡を可能にする（トレース相関ID設計準拠）
- セッション状態の重要な変化（状態遷移・判断結果・承認結果）はDATA-007（判断ログ）に記録する

**監査時に参照可能な情報**:
- DATA-005（セッション状態ファイル）：申請フローの進行状態・収集情報の全体像
- DATA-006（会話ログ）：各ターンの発話要約・適用ルール
- DATA-007（判断ログ）：申請種別判断・チェック結果・承認要否判断の証跡

**保持期間の考え方**:
- セッション状態ファイル（DATA-005）：要件上未定義
- 会話ログ（DATA-006）：要件上未定義
- 判断ログ（DATA-007）：要件上未定義

## 14. 設計上の留意点・リスク

**セッション肥大化のリスク**:
- 収集済み申請情報が多数の移動区間・経費項目を含む場合にセッションファイルが大きくなる
- 対策: 業務コンテキストの実体は参照IDのみ保持し、フル履歴は保存しない

**メモリ誤参照のリスク**:
- セッション状態ファイルが破損・不整合の場合に誤った状態から再開する可能性がある
- 対策: セッション再開時にセッション状態の整合性チェックを実施し、不整合時は新規セッションを開始する

**再開時の誤動作リスク**:
- 前回のHitL承認結果が残っていると、修正後の情報でも古い承認結果を使用してしまう可能性がある
- 対策: 修正（修正要望）が選択された場合はHitL承認結果を削除してから情報収集ステップへ戻る
