## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通する設定方針を定義し、各エージェントとコンポーネントで一貫した設定を適用する。

### 1.2 適用範囲

- AG-001（申請受付窓口エージェント）
- AG-002（交通費精算申請エージェント）
- AG-003（経費精算申請エージェント）
- TOOL-001（交通費計算ツール）・TOOL-002（申請書生成ツール）

### 1.3 本方針で定義すること

- 会話管理の方針
- リトライ戦略の方針
- ループ制御の方針
- セッション管理の方針
- 承認フックの方針
- ログ設計の方針
- エラー伝播の責務分担

### 1.4 本方針で定義しないこと

- 個々のエージェントの具体的な設定値（各エージェント基本設計書に委ねる）
- 実装レベルのクラス詳細・メソッド定義（基本設計・詳細設計に委ねる）
- SDK固有の設定パラメータの全量

## 2. 会話管理の方針

### 2.1 基本原則

- **会話履歴の保持**: 直近N件の会話のみをLLMに渡すスライディングウィンドウ方式を採用する。全履歴をLLMコンテキストに含めることは行わない
- **メモリ効率**: ウィンドウサイズを役割ごとに最適化し、不要なトークン消費を抑制する
- **コンテキストの継続性**: 申請フロー内で収集済みの業務コンテキスト（申請者名・収集済み項目）はセッション管理方針（SYS-SESSION-001）に従いDATA-005に永続化し、ウィンドウ外に出ても失わない設計とする

### 2.2 ウィンドウサイズの基準

| エージェント種別 | ウィンドウサイズ | 理由 |
|----------------|----------------|------|
| オーケストレーター（AG-001） | 30件 | 申請受付・種別判断・委譲を担当し、会話コンテキストを十分に保持するため |
| 交通費精算申請エージェント（AG-002） | 20件 | 交通費精算申請の必須項目を対話で収集するため |
| 経費精算申請エージェント（AG-003） | 15件 | 経費精算申請の必須項目を対話で収集するため |
| ツール（TOOL-001/TOOL-002） | - | ツール自体は会話管理を持たない |

> ウィンドウサイズは「LLMに渡す会話履歴の保持件数」であり、セッション全体の対話回数上限（30回、FR-012）とは独立した設定値である

### 2.3 会話管理の実装

**使用するクラス**: `SlidingWindowConversationManager`（strands-agents提供）

**設定方法**:
```python
from strands.agent.conversation_manager import SlidingWindowConversationManager

# オーケストレーター（AG-001）
conversation_manager_ag001 = SlidingWindowConversationManager(window_size=30)

# 交通費精算申請エージェント（AG-002）
conversation_manager_ag002 = SlidingWindowConversationManager(window_size=20)

# 経費精算申請エージェント（AG-003）
conversation_manager_ag003 = SlidingWindowConversationManager(window_size=15)
```

**設定パラメータ**:
- `window_size`: LLMコンテキストに含める直近の会話ターン数（メッセージ単位）

## 3. リトライ戦略の方針

### 3.1 基本原則

- **一時的なエラーへの対応**: Amazon Bedrock への API 呼び出しは boto3/botocore の組み込みリトライ機構（指数バックオフ）を使用する
- **Bedrock API 自動リトライ設定**: 最大6回、初期遅延4秒、最大遅延240秒の指数バックオフを適用する
- **最大試行回数の制限**: アプリケーションレベルのリトライは実行制御方針（SYS-EXEC-CTRL-001 7節）で定義された方針に従い、TOOL-001ファイル参照のみ1回のみ許可する

### 3.2 リトライ対象

| 対象 | リトライ可否 | 理由 |
|-----|------------|------|
| Amazon Bedrock 推論呼び出し | △（フレームワーク任せ） | boto3/botocore の組み込みリトライ機構（指数バックオフ）に委ねる。アプリケーションレベルの追加リトライは行わない |
| TOOL-001 ファイル参照失敗 | △（1回のみ） | 一時的なI/Oエラーの可能性があるため1回のみ許可。2回失敗時は手動入力へ代替 |
| TOOL-002 テンプレート参照失敗 | × | 構成上の問題であり再試行より即時エスカレーションが適切 |
| ファイル書き込み（DATA-005/006/007） | × | セッション障害は即時エスカレーション案内・TERMINATED |

### 3.3 リトライ戦略の実装

**Amazon Bedrock リトライ**: boto3/botocore のリトライ設定（`RetryConfig`）で最大6回・初期遅延4秒・最大遅延240秒・指数バックオフを明示的に設定する。

**TOOL-001 ファイル参照リトライ**: TOOL-001の実装内でtry/exceptを使用し、最大1回のファイル参照再試行を実装する。詳細はツール基本設計書に委ねる。

## 4. ループ制御の方針

### 4.1 基本原則

- **無限ループの防止**: 1回のエージェント呼び出し（1ターン）内でLLM推論→ツール実行を繰り返すReActループの上限を設け、エージェントの暴走を防止する
- **処理の可視化**: ループ回数をカスタムフック（LoopControlHook）で追跡し、ログに記録する
- **ユーザーへの通知**: ReActループ上限到達時は社員にエラーを通知し、セッションをTERMINATEDに遷移させる

### 4.2 最大ループ回数の基準

| エージェント種別 | 最大ループ回数 | 理由 |
|----------------|--------------|------|
| 全エージェント（AG-001/AG-002/AG-003） | 10回 | 1ターン内のツール呼び出しは最大でも数回（TOOL-001/TOOL-002）であり、10回は十分な余裕を持ちつつ暴走を防止する値 |

> このReActループ最大回数はセッション全体の対話回数上限（30回、FR-012）とは独立した値である。セッション対話回数はセッション管理方針（SYS-SESSION-001）に従いDATA-005で管理する

### 4.3 ループ制御の実装

**使用するクラス**: `LoopControlHook`（HookProvider として実装）

**登録イベント**:
- `BeforeInvocationEvent`：エージェント呼び出し開始時にループカウンタを0にリセットする
- `AfterModelCallEvent`：LLM呼び出し後にループカウンタをインクリメントし上限到達を監視する
- `AfterInvocationEvent`：エージェント呼び出し完了後にループカウンタを0にリセットする

**設定方法**:
```python
from handlers.loop_control_hook import LoopControlHook

loop_control_hook = LoopControlHook(max_iterations=10)
```

**設定パラメータ**:
- `max_iterations`: 1エージェント呼び出し内のReActループ上限回数（全エージェント共通: 10）

> `LoopControlHook` の詳細な実装方針はハンドラー基本設計書に委ねる

## 5. セッション管理の方針

### 5.1 基本原則

- **会話履歴の永続化**: セッション状態（会話コンテキスト・業務データ・進行ステップ）はDATA-005（ファイル永続化）に保存し、セッション再開（resume）を可能にする
- **セッションIDの一意性**: セッション開始時にUUIDを生成し、申請フロー全体を通じて同一のsession_idを使用する
- **エージェント間の共有**: AG-001が生成したsession_idをAG-002/AG-003が共有する。AG-002/AG-003はAG-001が確立したセッションに追記する形で状態を更新する

### 5.2 セッションIDの形式

**形式**: `sess_xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`（UUID形式）

**例**: `sess_a1b2c3d4-e5f6-7890-abcd-ef1234567890`

### 5.3 セッション管理の実装

**使用するクラス**: `SessionManager`（カスタム実装）

**設定方法**:
```python
from session.session_manager import SessionManager

session_manager = SessionManager(
    session_id=session_id,
    storage_path="data/sessions/"
)
```

**設定パラメータ**:
- `session_id`: UUID形式のセッション識別子
- `storage_path`: セッション状態ファイル（DATA-005）の保存ディレクトリパス

**詳細**: セッション管理方針（`SYS-SESSION-001`）を参照。`SessionManager`の詳細な実装方針はセッションマネージャ基本設計書に委ねる

## 6. 承認フックの方針

### 6.1 基本原則

- **重要な操作の承認**: TOOL-002（申請書生成）の実行前に社員の確認・承認を必須とする。申請書の誤生成を防止する
- **ユーザーの選択肢**: 承認選択肢は「OK」「修正」「キャンセル」の3択とする（実行制御方針 11節準拠）
- **誤操作の防止**: HitL承認結果（OK）を受領するまでTOOL-002の実行を完全に禁止する（GRD-009準拠）

### 6.2 承認対象ツールの基準

| ツール種別 | 承認要否 | 理由 |
|----------|---------|------|
| TOOL-002（申請書生成） | 要 | Excel申請書の生成は不可逆的な副作用を持つ重要操作であるため、社員の明示的な承認が必要 |
| TOOL-001（交通費計算） | 不要 | 参照・計算のみで副作用がなく、結果はエージェントの応答に含めて社員に提示される |
| image_reader（strands_tools） | 不要 | 領収書画像の読み取りのみで副作用がない |

### 6.3 承認フックの実装

**使用するクラス**: `HumanApprovalHook`（カスタム実装）

**設定方法**:
```python
from handlers.human_approval_hook import HumanApprovalHook

human_approval_hook = HumanApprovalHook(
    approval_callback=approval_callback
)
```

**設定パラメータ**:
- `approval_callback`: 社員に承認確認を提示し、OK/修正/キャンセルの選択結果を返すコールバック関数

> `HumanApprovalHook` の詳細な実装方針はハンドラー基本設計書に委ねる

## 7. 設定の適用方法

### 7.1 エージェント作成時の設定

**基本パターン（交通費精算申請エージェント: AG-002）**:
```python
from strands import Agent
from strands.agent.conversation_manager import SlidingWindowConversationManager
from handlers.loop_control_hook import LoopControlHook
from handlers.human_approval_hook import HumanApprovalHook
from session.session_manager import SessionManager

session_manager = SessionManager(
    session_id=session_id,
    storage_path="data/sessions/"
)

agent_002 = Agent(
    model=bedrock_model,  # config/model_config.py で定義
    system_prompt="...",
    tools=[tool_001, tool_002],
    conversation_manager=SlidingWindowConversationManager(window_size=20),
    callback_handler=LoopControlHook(max_iterations=10),
)
```

**基本パターン（経費精算申請エージェント: AG-003）**:
```python
agent_003 = Agent(
    model=bedrock_model,
    system_prompt="...",
    tools=[tool_002],
    conversation_manager=SlidingWindowConversationManager(window_size=15),
    callback_handler=LoopControlHook(max_iterations=10),
)
```

**オーケストレーター（AG-001）のパターン**:
```python
agent_001 = Agent(
    model=bedrock_model,
    system_prompt="...",
    tools=[ag_002_as_tool, ag_003_as_tool],  # Agent As Tools
    conversation_manager=SlidingWindowConversationManager(window_size=30),
    callback_handler=LoopControlHook(max_iterations=10),
)
```

### 7.2 設定のカスタマイズ

**カスタマイズが必要な場合**:
- AG-001は `window_size=30`、AG-002は `window_size=20`、AG-003は `window_size=15` とエージェント種別ごとに設定を変える
- AG-002/AG-003のみ `HumanApprovalHook` を適用する（TOOL-002実行前の承認が必要なため）

**カスタマイズ方法**: 各エージェントの基本設計書を参照

## 8. 設計上の留意点・リスク

### 8.1 会話管理

**リスク**: ウィンドウサイズを超えた会話履歴が切り落とされると、エージェントが収集済み申請情報を「忘れる」リスクがある

**対策**: 収集済み申請情報はDATA-005（セッション状態ファイル）に永続化し、毎ターン開始時にセッション管理方針に従って業務コンテキストを復元する。ウィンドウ外に出た情報を補完する設計を各エージェント基本設計書で定義する

### 8.2 ループ制御

**リスク**: `max_iterations=10` の上限に達した場合、申請フローが中途で強制終了される可能性がある

**対策**: TOOL-001/TOOL-002の呼び出し回数は1ターン内で1〜2回程度であるため、10回は十分な余裕がある。上限到達時は社員にエラーメッセージを提示してセッションをTERMINATEDに遷移させ、新規セッション開始を案内する

### 8.3 セッション管理

**リスク**: DATA-005（セッション状態ファイル）のファイルI/OエラーによりSessionManagerが正常に機能しないリスクがある

**対策**: SessionManagerのファイルI/O失敗はEX-07（システム障害）として扱い、即時エスカレーション案内・TERMINATED遷移とする（例外処理方針 SYS-EXCEPTION-001準拠）

### 8.4 承認フック

**リスク**: HumanApprovalHookがTOOL-002実行前に正常に発火しない場合、承認なしで申請書が生成されるリスクがある

**対策**: AG-002/AG-003の実装でTOOL-002実行前のHumanApprovalHook発火を必須とし、GRD-009（ツール呼び出し権限制御）とあわせて二重に誤実行を防止する。詳細はハンドラー基本設計書に委ねる

## 7b. ログ設計の方針

### 7b.1 基本原則

- **一元管理**: ログ設定（レベル・出力先・フォーマット）はエントリーポイント（`main.py`）で一元管理する
- **モジュール責務**: ログ出力は各モジュール（エージェント・ツール・ハンドラー等）が責任を持って行う
- **環境変数制御**: ログレベルは環境変数（`LOG_LEVEL`）で制御し、未設定時のデフォルトは `WARNING` とする
- **メッセージ分離**: ユーザー向けメッセージ（申請者への応答）と技術ログ（デバッグ・監査用）は分離する
- **コンテキスト情報**: 各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する

## 7c. エラー伝播の責務分担

### 7c.1 基本原則

| 主体 | 責務 |
|-----|------|
| 各モジュール（エージェント・ツール） | 例外発生箇所でログを出力し、ErrorHandler へ処理を委譲する |
| ErrorHandler | ユーザー向けメッセージを生成して呼び出し元に返す |
| 呼び出し元（エージェント・ハンドラー） | 受け取ったメッセージを戻り値または表示として上位レイヤーへ伝播させる |

詳細はログ出力方式設計（`SYS-LOG-001`）および例外処理方針（`SYS-EXCEPTION-001`）を参照。
