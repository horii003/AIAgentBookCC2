## 1. 目的・適用範囲

### 1.1 目的

ガードレール要件定義（SR-13）で定義された全ガードレール要件（GRD-001〜GRD-010）を実装方式として具体化し、各エージェント・コンポーネントで一貫したガードレール処理を実現する。

### 1.2 適用範囲

- AG-001（申請受付窓口エージェント）
- AG-002（交通費精算申請エージェント）
- AG-003（経費精算申請エージェント）
- TOOL-001（交通費計算ツール）・TOOL-002（申請書生成ツール）
- SessionManager・ErrorHandler・各カスタムフック

### 1.3 本方針で定義しないこと

- 具体的なクラス設計・メソッド定義（ハンドラー基本設計書に委ねる）
- ガードレール詳細実装（ガードレール詳細設計書に委ねる）
- Amazon Bedrock Guardrails の具体的なパラメータ値（詳細設計フェーズで定義）

## 2. ガードレール全体構成

### 2.1 ガードレール層の構成

```text
[社員入力]
   ↓
[入力ガード層：Applicationレベル]
   GRD-001: プロンプトインジェクション検知
   GRD-002: PII検知・CVVブロック（入力）
   GRD-003: 業務外トピック拒否
   ↓
[会話・判断処理]
   ↓
[ルール検証層：Applicationレベル]
   GRD-004: 申請書推測補完禁止（申請書生成前）
   GRD-005: 必須項目充足確認（TOOL-002呼び出し前）
   GRD-006: グラウンディング確認（LLM応答後）
   ↓
[生成・実行処理]
   ↓
[出力ガード層：Applicationレベル]
   GRD-007: 機密情報漏洩防止・CVVブロック（出力）
   GRD-008: 内部推論非表示
   ↓
[実行ガード層：Applicationレベル]
   GRD-009: ツール呼び出し権限制御
   GRD-010: 申請書自動送信禁止
   ↓
[社員への応答 / 成果物出力]
```

### 2.2 実現方式の分担

| ガードレール層 | 実現方式 | 担当コンポーネント |
|-------------|---------|----------------|
| 入力ガード（GRD-001〜003） | Applicationレベル（カスタムハンドラー） | InputGuardHandler（カスタム実装） |
| ルール検証（GRD-004〜006） | Applicationレベル（カスタムハンドラー） | RuleValidationHandler（カスタム実装） |
| 出力ガード（GRD-007〜008） | Applicationレベル（カスタムハンドラー） | OutputGuardHandler（カスタム実装） |
| 実行ガード（GRD-009〜010） | Applicationレベル（LoopControlHook / HumanApprovalHook） | LoopControlHook・HumanApprovalHook（カスタム実装） |

> **本システムの方針**: Amazon Bedrock Guardrailsの利用は詳細設計フェーズで判断する。現時点ではすべてApplicationレベルのカスタム実装で実現する方針とする。

### 2.3 エージェントごとのガードレール適用方式

| エージェント | 適用ガードレール | 適用方式 |
|-----------|--------------|---------|
| AG-001 | GRD-001/002/003（入力）、GRD-006（グラウンディング）、GRD-007/008（出力）、GRD-009（ツール権限） | Strands Agentsの `callback_handler` フックを通じて各ターンの入出力時に適用 |
| AG-002 | GRD-001/002/003（入力）、GRD-004（推測補完禁止）、GRD-005（必須項目充足）、GRD-006（グラウンディング）、GRD-007/008（出力）、GRD-009（ツール権限）、GRD-010（自動送信禁止） | AG-001と同じフック方式 + TOOL-002呼び出し前のHitL承認フック |
| AG-003 | AG-002と同じ | AG-002と同じ |

## 3. ガードレールツール設定方式

### 3.1 設定パラメータ管理方式

| 設定項目 | 管理方式 | 備考 |
|--------|---------|------|
| ガードレール識別子（Amazon Bedrock Guardrails利用時） | 環境変数（`.env`）で管理 | 詳細設計フェーズで判断 |
| コンテンツフィルタリング設定 | `config/guardrail_config.py` に集約 | ガードレール設定は `config/` 配下で一元管理する |
| CVVブロックパターン（正規表現） | `config/guardrail_config.py` に集約 | |
| 業務外トピック判定プロンプト | `config/guardrail_config.py` に集約 | 社内申請業務（交通費精算・経費精算）以外を拒否する定義 |
| グラウンディング閾値 | 要件上未定義 | 詳細設計フェーズで定義する |

### 3.2 エージェントへの適用方式

各エージェント（AG-001/AG-002/AG-003）のコンストラクタ引数として `callback_handler` にガードレールハンドラーを設定する。入力ガード・出力ガードは `callback_handler` の入力受信・出力送信イベントフックで実行する。

```python
from handlers.input_guard_handler import InputGuardHandler
from handlers.output_guard_handler import OutputGuardHandler
from handlers.loop_control_hook import LoopControlHook
from handlers.human_approval_hook import HumanApprovalHook

agent = Agent(
    model=bedrock_model,
    system_prompt="...",
    callback_handler=CompositeHandler([
        InputGuardHandler(),   # GRD-001/002/003
        OutputGuardHandler(),  # GRD-007/008
        LoopControlHook(max_iterations=10),  # GRD-009
    ])
)
```

### 3.3 設定値の管理方式

- `config/guardrail_config.py` に全ガードレール設定を集約する
- 機密情報（正規表現パターン等）は環境変数（`.env` / `python-dotenv`）で管理する
- `config/model_config.py` とは分離して管理する

## 4. ガードレール要件ごとの実装方式

| ガードレールID | 機能カテゴリ | 実現方式 | 実装箇所 | 検知時の処理 |
|-------------|-----------|---------|---------|-----------|
| GRD-001 | 入力ガード（プロンプトインジェクション） | Applicationレベル：入力テキストにシステムプロンプト書き換えパターン（正規表現）が含まれるかを検査する | InputGuardHandler（`callback_handler` の入力受信フック） | 入力をブロックし、社員に通常の申請内容入力を促す。EX-01として処理 |
| GRD-002 | 入力ガード（PII検知・CVVブロック） | Applicationレベル：CVVは正規表現でブロック。PII（氏名・メールアドレス・電話番号等）はマスキング処理（`_mask_sensitive()`）を適用 | InputGuardHandler（`callback_handler` の入力受信フック） | CVV検知時は入力をブロックし社員に再入力を促す。PII検知時はマスキングして処理継続。GRD-002ログを記録 |
| GRD-003 | 入力ガード（業務外トピック拒否） | Applicationレベル：LLMによる意図分類（業務内/業務外の2分類プロンプト）で業務外と判定された入力を拒否する | InputGuardHandler（`callback_handler` の入力受信フック） | 「申請業務の観点からお答えできません」と応答し申請内容の入力を促す。EX-01として処理 |
| GRD-004 | ルール検証（申請書推測補完禁止） | Applicationレベル：TOOL-002実行前に収集済み申請情報（D-007）と生成対象フィールドの一致検証を行う | RuleValidationHandler（TOOL-002呼び出し前） | 申請書生成を停止し、不足項目の再収集を継続する。EX-03として処理 |
| GRD-005 | ルール検証（必須項目充足確認） | Applicationレベル：TOOL-002呼び出し前に申請種別ごとの必須項目リストと収集済み申請情報を照合する | RuleValidationHandler（TOOL-002呼び出し前） | TOOL-002の呼び出しをブロックし、不足項目の対話収集を継続する。EX-01として処理 |
| GRD-006 | ルール検証（グラウンディング確認） | Applicationレベル：LLM応答後、申請ルールファイル（DATA-001）に基づかない申請種別判断・チェック結果を検出する（閾値は要件上未定義） | RuleValidationHandler（LLM応答後フック） | 幻覚として検出し、DATA-001を参照して判断を再実行する。閾値以下の場合は再実行をトリガーする |
| GRD-007 | 出力ガード（機密情報漏洩防止・CVVブロック） | Applicationレベル：LLM出力テキストに対して `_mask_sensitive()` を適用し、CVVを出力ブロック・PIIをマスキングする | OutputGuardHandler（`callback_handler` の出力送信フック） | PII検知時はマスキングして出力する。CVV検知時は出力をブロックしログを記録する |
| GRD-008 | 出力ガード（内部推論非表示） | Applicationレベル：LLM応答テキストからChain-of-Thought部分（`<thinking>...</thinking>` タグ等）を除去して出力する | OutputGuardHandler（`callback_handler` の出力送信フック） | 内部推論部分を除去して社員に提示する |
| GRD-009 | 実行ガード（ツール呼び出し権限制御） | Applicationレベル：Strands Agentsのツール実行フック（`callback_handler.on_tool_start`）でエージェント×ツールの許可テーブルを参照して権限チェックを行う | LoopControlHook（`callback_handler` のツール実行前フック） | 権限外ツール呼び出しをブロックし、ERR-001ログを記録する。EX-05として処理 |
| GRD-010 | 実行ガード（申請書自動送信禁止） | Applicationレベル：TOOL-002の出力（申請書ドラフト）生成後に自動送信操作が発生しないようツール定義を申請書ドラフト生成のみに限定する | TOOL-002の実装（申請書生成ツール） | 申請書は生成・提示のみを行う。自動送信APIは実装しない（設計上の禁止） |

## 5. アプリケーションレベルのガードレール

### 5.1 承認ゲートの実装方式

TOOL-002（申請書生成）の実行前に `HumanApprovalHook` による承認ゲートを設ける（実行制御方針 SYS-EXEC-CTRL-001 11節準拠）。

**実装フロー**:
1. AG-002/AG-003が全必須項目収集完了と判断する
2. GRD-005（必須項目充足確認）チェックを通過する
3. `HumanApprovalHook` が発火して申請情報サマリーを社員に提示し、OK/修正/キャンセルの確認を行う
4. OK選択時のみ TOOL-002 の呼び出しが許可される
5. 修正選択時は情報収集ステップへ戻る（HitL承認結果をリセット）
6. キャンセル選択時はセッションをCLOSEDに遷移する

**GRD-009との関係**: HitL承認（OK）前のTOOL-002呼び出し試行は GRD-009 でブロックされる（P-EXEC-001準拠）

### 5.2 ループ制御との関係

- GRD-009（ツール呼び出し権限制御）は `LoopControlHook` の `on_tool_start` フックで実装する
- ガードレールチェックはReActループの各ターンで実行されるが、ループ回数カウントには含めない
- ガードレールチェック失敗によるターン中断はReActループ回数に加算しない

## 7. ガードレール検知時の処理フロー

### 7.1 入力ガード検知時（GRD-001/002/003）

1. `InputGuardHandler` がガードレール違反を検知する
2. EX-01（入力例外）として ErrorHandler に渡す
3. ErrorHandler がユーザー向けメッセージを生成する（技術的詳細を隠蔽）
4. セッション状態はWAITINGのまま継続し、社員に再入力を促す
5. CVVブロック（GRD-002）時は GRD-002ブロックログを記録する

### 7.2 ルール検証・実行ガード検知時（GRD-004/005/006/009）

1. RuleValidationHandler または LoopControlHook がガードレール違反を検知する
2. GRD-004/005: EX-01（入力例外）または EX-03（業務ルール違反）として ErrorHandler に渡す
3. GRD-006（グラウンディング）: 幻覚として検出し、DATA-001参照を再実行する
4. GRD-009（ツール権限）: EX-05（実行不可）として ErrorHandler に渡す。ERR-001ログを記録してTERMINATEDへ遷移する

### 7.3 出力ガード検知時（GRD-007/008）

1. `OutputGuardHandler` がガードレール違反を検知する
2. PII検知（GRD-007）: マスキング後の出力を社員に提示する。セッション継続
3. CVV検知（GRD-007）: 出力をブロックしCVV検知ログを記録する
4. 内部推論検知（GRD-008）: 内部推論部分を除去した出力を社員に提示する

### 7.4 例外処理方針との連携

| ガードレールID | 例外分類（例外処理方針準拠） | セッション状態 |
|-------------|----------------------|-----------|
| GRD-001（プロンプトインジェクション） | EX-01（入力例外） | WAITING（再入力を促す） |
| GRD-002（CVVブロック） | EX-01（入力例外） | WAITING（再入力を促す） |
| GRD-003（業務外トピック） | EX-01（入力例外） | WAITING（再入力を促す） |
| GRD-004（推測補完禁止） | EX-03（業務ルール違反） | ACTIVE（不足項目収集へ戻る） |
| GRD-005（必須項目充足） | EX-01（入力例外） | WAITING（不足項目収集へ戻る） |
| GRD-006（グラウンディング） | EX-02（判断不能）として再実行 | ACTIVE（再実行後に継続） |
| GRD-007（機密情報漏洩・CVV） | CVV：EX-01 / PII：継続（マスキング） | CVV：WAITING / PII：ACTIVE |
| GRD-008（内部推論非表示） | 例外扱いしない（フィルタリングのみ） | ACTIVE（継続） |
| GRD-009（ツール権限） | EX-05（実行不可） | TERMINATED |
| GRD-010（申請書自動送信禁止） | EX-05（実行不可）として設計的に禁止 | TERMINATED（試みた場合） |

## 8. 設計上の留意点・リスク

**GRD-003の精度リスク**:
- LLMによる業務外トピック判定は誤検知（申請に関連する質問が業務外と判定される）リスクがある
- 対策: システムプロンプトに業務外判定基準を明示し、「申請業務に直接関係しない」という判定基準を厳密に定義する。詳細はシステムプロンプト詳細設計書で定義する

**GRD-006グラウンディング閾値未定義リスク**:
- グラウンディング確認の閾値が要件上未定義であり、実装時に適切な値が設定できない可能性がある
- 対策: 詳細設計フェーズでAmazon Bedrock Guardrailsのグラウンディング機能またはアプリケーションレベルの類似度スコアを使用して閾値を定義する

**GRD-009の回避リスク**:
- LLMがツール呼び出し権限テーブルを「理解」して回避しようとするリスクがある（主にAG-001がTOOL-002を直接呼び出そうとするケース）
- 対策: 権限チェックはLLMの判断に依存せず、`LoopControlHook.on_tool_start()` でエージェントIDとツールIDの組み合わせを強制的にチェックする
