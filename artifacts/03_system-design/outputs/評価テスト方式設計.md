## 1. 目的・適用範囲

### 1.1 目的

AIエージェントシステムの品質を保証するための評価テスト方式を定義する。本書は詳細設計フェーズの評価テスト詳細設計および実装フェーズの評価テストコード生成へのインプットとなる。

> **評価指標・合否基準・実施方法・回帰評価方針は `評価指標一覧.md` を正とする。本書はそれらを参照し、評価の「仕組み・アーキテクチャ・環境」を定義する。**

### 1.2 適用範囲

- AIエージェント評価テスト：LLM-as-Judge方式によるエージェント品質の自動評価
  - MET-001（申請種別判断正確率）：AG-001の申請種別判断精度の評価
  - MET-003（業務ゴール達成率）：申請フロー全体（CF-001〜CF-005）のE2E評価

### 1.3 本方針で定義しないこと

- 個別テストケースの入力データと期待結果（詳細設計で定義）
- 評価テストコードの実装詳細（実装フェーズで定義）
- 評価フレームワーク固有のAPI仕様（詳細設計で定義）
- 評価指標の定義・閾値・実施方法・回帰評価方針（評価指標一覧.mdで定義）

## 2. 評価戦略・基本方針

### 2.1 AIエージェント固有のリスクと評価戦略

| リスク | 対応戦略 |
|--------|---------|
| LLM応答の非決定性 | 同一テストケースでの試行回数は1回とする。各テストケースは1度のみ実行し、各テストケースごとの集計はしない |
| プロンプトの意図逸脱 | LLM-as-Judge（二値）による判定と、業務ルール（KN-001）に基づくグラウンディング確認を組み合わせて検証する |
| エージェント間の情報伝達誤り | マルチターンのE2Eシナリオ（CF-001〜CF-005全体）で連携評価を実施し、AG-001→AG-002/AG-003への委譲精度を検証する |
| ガードレール漏れ | ガードレール要件（GRD-001〜010）に対応した異常系テストケースを含め、各ガードレールの動作を個別に検証する |
| ユーザーシミュレーションの限界 | ユーザーシミュレーターの発話は会話フロー一覧（CF-001〜CF-005）に基づくシナリオを起点とし、実際の利用データで継続的に拡充する |
| LLM-as-Judgeの信頼性 | 判定モデルは評価対象エージェントとは別モデルを使用する（詳細設計で定義）。判定理由を必ずレポートに含め、人手でのサンプリング確認を実施する |
| フレームワーク間の差異 | 評価テストコードはフレームワーク依存の抽象概念（Section 3参照）を介して記述し、フレームワーク変更時の影響を局所化する |

## 3. 評価テストの共通抽象概念

### 3.1 コア抽象概念の定義

| 抽象概念 | 定義 |
|---------|------|
| テストケース | 評価の単位。入力・期待出力・期待軌跡・メタデータを持つ |
| 評価方式 | テストケースの実行結果に対してスコアリング（LLM-as-Judge・軌跡マッチ等）を適用し、スコア・合否・判定理由を返す評価の方法 |
| 評価結果 | スコア (float 0-1) + 合否 (bool) + 判定理由 (str) |
| 実験 | テストケース群 × 評価方式群のバッチ実行単位 |
| レポート | 全体スコア・個別スコア・判定理由の集約 |

### 3.2 テストケースの構成要素

| 要素 | 型 | 必須/任意 | 説明 |
|------|---|---------|------|
| 識別名 | string | 必須 | テストケースの一意識別子（例: `MET001_transport_001`） |
| 入力 | string / messages | 必須 | ユーザー発話またはメッセージ配列 |
| 期待出力 | string | 任意 | 期待する最終応答（MET-001: 判断結果文字列） |
| 期待軌跡 | list[string] | 任意 | 期待するツール呼び出し順序（MET-003: TOOL-001, TOOL-002等） |
| メタデータ | dict | 任意 | 参照する追加情報（正解ラベル・シナリオ種別・評価観点ID） |
| セッションID | string | 自動生成 | テレメトリ分離用（`sess_` + UUID形式） |

### 3.3 評価方式の種類

| 評価方式種別 | スコアリング方式 | 適用場面 |
|-----------|---------------|---------|
| **LLM-as-Judge（二値）** | 達成/未達成 → 1.0/0.0 | MET-001（申請種別判断正確率）・MET-003（業務ゴール達成率） |
| **LLM-as-Judge（段階）** | N段階評価 → 0.0〜1.0 | 応答品質・一貫性の詳細評価（将来拡張用） |
| **軌跡マッチ（決定的）** | ツール呼び出し列の一致判定 | AG-001→AG-002/AG-003委譲の正確性（LNK-001/LNK-002） |
| **完全一致/文字列マッチ** | 出力文字列の一致判定 | 申請種別判断結果（「交通費精算申請」「経費精算申請」の文字列一致） |
| **類似度評価** | コサイン類似度・ROUGE-L | 将来拡張用 |
| **カスタムコード** | Python関数によるスコアリング | 申請書Excelファイルの生成有無確認・必須項目充足確認 |

### 3.4 軌跡マッチモード

| マッチモード | 判定基準 | 適用場面 |
|-----------|---------|---------|
| strict | 完全一致（順序含む） | AG-001→AG-002委譲（LNK-001）の順序検証 |
| unordered | 順不同で全要素一致 | MET-003のフロー全体でのツール呼び出し網羅性確認 |
| subset | 期待が実際の部分集合 | 最低限必要なツール呼び出し（TOOL-001→TOOL-002）の検証 |
| superset | 実際が期待の部分集合 | 不要なツール呼び出し（GRD-009検証）の検出 |

## 4. 評価レベルの定義

### 4.1 評価レベル体系

| 評価レベル | 評価対象 | 評価方式 | 判定粒度 | 対応指標 |
|-----------|---------|---------|---------|---------|
| セッションレベル | 会話全体のゴール達成（CF-001〜CF-005） | LLM-as-Judgeがセッション全体の軌跡を評価 | セッション単位 | MET-003 |
| ツールレベル | 個別のツール選択判断（TOOL-001/TOOL-002） | LLM-as-Judgeが各ツール呼び出しの妥当性を評価 | ツール呼び出し単位 | MET-003（補助） |
| 出力レベル | 最終応答の品質（申請種別判断結果・申請書生成結果） | LLM-as-Judge（二値）/ 完全一致 | 応答単位 | MET-001 |
| 軌跡レベル | ツール呼び出し列全体（AG-001→AG-002/AG-003連携） | 決定的マッチ / LLM-as-Judge | 軌跡単位 | MET-001・MET-003（補助） |

### 4.2 評価レベルの選択基準

| 確認したいこと | 選択すべきレベル | 実行方式 |
|-------------|---------------|---------|
| 社員が申請書作成を完了できるか（MET-003） | セッションレベル | マルチターン |
| 申請種別判断が正確か（MET-001） | 出力レベル | シングルターン |
| AG-001が正しくAG-002/AG-003を選択するか | ツールレベル | シングルターン |
| ツール呼び出しの順序・網羅性（TOOL-001→TOOL-002） | 軌跡レベル | マルチターン |

## 5. 評価方式の共通アーキテクチャ

### 5.1 実行フロー

```
┌─────────────────────────────────────────────────────────┐
│                    評価実行基盤                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [1. テストケース定義]                                    │
│    - 入力（ユーザー発話）                                  │
│    - メタデータ（ゴール / 期待ツール / ルーブリック）         │
│                                                         │
│  [2. タスク実行]                                         │
│    - エージェントにテストケースの入力を送信                  │
│    - テレメトリによる実行軌跡（スパン）の自動記録            │
│    - シングルターン or マルチターン（ユーザーシミュレーター）  │
│                                                         │
│  [3. 評価判定]                                           │
│    - 評価方式が実行結果（出力 + 軌跡）をスコアリング          │
│    - LLM-as-Judge / 決定的マッチ / カスタムコード           │
│                                                         │
│  [4. 結果レポート]                                        │
│    - 全体スコア（overall_score）                          │
│    - 個別スコア・合否・判定理由                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 テストケース設計方針

| 項目 | 方針 |
|------|------|
| テストケースの構成要素 | 識別名・入力テキスト・タスク説明（評価観点ID・シナリオ種別）・判定基準（ルーブリック） |
| テストデータの独立性 | 各テストケースは他のケースに依存しない。セッション状態はケースごとに初期化する |
| テレメトリの分離 | ケース実行前にスパンをクリアし、ケース間のデータ混入を防止する |
| 評価データセットの最低件数 | MET-001: 交通費・経費・判断不能の各ケースを最低5件以上。MET-003: 交通費申請・経費申請の各E2Eシナリオを最低5件以上（詳細設計で定義） |
| 評価データセットの拡充 | 運用開始後も実際の利用データをもとに継続的にケースを追加する |

### 5.3 ユーザーシミュレーター方式

| 項目 | 方針 |
|------|------|
| シミュレーターの役割 | テストケースに定義されたゴールに基づき、申請フローの各ステップに対応した動的なユーザー発話を生成する |
| 最大ターン数 | 30ターン（FR-012のセッション対話回数上限に合わせる） |
| 会話終了条件 | ゴール達成判定（申請書ドラフト生成かつチェック結果受領）または最大ターン到達 |
| 入力の起点 | テストケースの入力フィールド（最初のユーザー発話） |

### 5.4 LLM-as-Judge設計方針

| 項目 | 方針 |
|------|------|
| 判定モデル | 評価対象エージェント（Claude Sonnet 4.5）とは別モデルを推奨（詳細設計フェーズで定義） |
| 判定根拠の出力 | 判定理由を必ず出力しレポートに含める。スコアのみの判定は禁止 |
| スコアリング尺度 | 二値を基本（MET-001・MET-003）。段階評価が必要な場合はルーブリックを明示する |
| 試行回数 | 同一テストケースでの試行回数は1回。各テストケースは1度のみ実行し、テストケースごとの集計はしない |
| Few-shot例 | MET-001の判断不能ケース（JD-02）のような複雑な判定にはfew-shot exampleを活用する |

## 6. AIエージェント評価方針

### 6.1 評価観点マッピング

> 各評価観点の確認内容・対応指標IDは `評価指標一覧.md` の「1. 評価観点一覧（マスター）」を参照。

| 評価観点 | 評価方式 | 対応指標 |
|---------|---------|---------|
| 妥当性 | LLM-as-Judge（二値）+ 完全一致（申請種別ラベルとの比較） | MET-001 |
| 有用性 | LLM-as-Judge（二値）: 申請書ドラフト生成かつチェック結果提示の達成判定 | MET-003 |
| 安全性 | カスタムコード: ガードレール（GRD-001〜010）の動作確認（異常系テストケース） | MET-001（補助）・MET-003（補助） |
| 一貫性 | 軌跡マッチ（subset）: セッション内での申請フローの一貫した進行確認 | MET-003（補助） |
| ツール利用の適切性 | 軌跡マッチ（strict）: TOOL-001→TOOL-002の呼び出し順序・AG-001からの正しい委譲先選択 | MET-003（補助） |
| 制御機構の有効性 | カスタムコード: HitL承認フロー（HumanApprovalHook）の動作確認・ループ制御（ReAct上限10回）の動作確認 | MET-003（補助） |
| 運用品質 | カスタムコード: LLMトークン数・応答時間の記録（評価指標未定義のため参考値として記録） | 要件上未定義 |

## 7. 評価環境設計

### 7.1 実施環境

| 項目 | 値 |
|------|-----|
| Pythonバージョン | Python 3.x（本番環境と同一バージョン） |
| モデルID | Claude Sonnet 4.5（`config/model_config.py` の設定に従う） |
| 評価用LLM-as-JudgeモデルID | 要件上未定義（詳細設計フェーズで定義） |
| AWS リージョン | 要件上未定義 |
| 評価ログ出力先 | `evals/logs/` |

### 7.2 評価実行基盤の要件

| 項目 | 要件 |
|------|------|
| テレメトリ | インメモリスパンエクスポーターで実行軌跡（ツール呼び出し・エージェント委譲）を記録する |
| 承認フック無効化 | `HumanApprovalHook` のブロッキング動作を自動承認（OK）に差し替える評価用パッチを適用する |
| エージェント生成 | 本番と同一の構成（モデルID・システムプロンプト・ツール定義・ループ制御設定）で評価用エージェントを生成する |
| ログ設定 | UTF-8対応・ファイル出力（`evals/logs/`）。Strands Agents / botocore等のSDKログは WARNINGレベル以上のみ出力する |
| LLM呼び出し | モックなし（Amazon Bedrock への実呼び出し）。本番同等環境で評価する |
| ブロッキングフック | `HumanApprovalHook` を自動承認パッチに差し替える（TOOL-002呼び出し前の承認確認をスキップ） |
| セッション管理 | テストケースごとに新規セッションを生成し、テスト間のセッション状態の混入を防止する |

### 7.3 テストデータ管理方針

| 項目 | 方針 |
|------|------|
| テストケースの配置 | `evals/` 配下にPythonモジュールとして定義する（`evals/cases/met001_cases.py`、`evals/cases/met003_cases.py` 等） |
| 本番データの利用 | 本番データ（実際の申請データ）は個人情報を含むため、匿名化またはダミーデータに置き換えてから評価データセットに追加する |
| バージョン管理 | テストケース定義ファイルをGitで管理する。変更時はコミット履歴でバージョンを追跡する |
| 評価データセットの更新 | 運用開始後のシステムプロンプト変更・業務ルール変更時は、対応するテストケースを同時に更新する |

## 8. 評価実施フロー

### 8.1 実施工程

```
詳細設計完了
  ↓
テストケース実装・評価テストコード作成
  ↓
評価テスト実施（python evals/eval_met001.py, python evals/eval_met003.py）
  ↓
評価指標測定（MET-001・MET-003スコア算出）
  ↓
合否判定（評価指標一覧.mdのゲート条件確認）
  ↓
品質チェック通過
```

### 8.2 ゲート条件（合格基準）

> 各MET-xxxの閾値は `評価指標一覧.md` の各指標定義書「5. 合格基準・閾値」を参照。

| ゲート | 移行基準 |
|--------|---------|
| 評価テスト完了 | MET-001（申請種別判断正確率）≥ 0.95 かつ MET-003（業務ゴール達成率）≥ 0.90 を達成すること |
| 品質チェック完了 | 評価指標全項目（MET-001・MET-003）が合否基準を満たすこと |

### 8.3 回帰評価方針

| トリガー | 実施内容 | 合否基準 |
|---------|---------|---------|
| プロンプト変更時 | 全テストケース（MET-001・MET-003）を再実行する | MET-001・MET-003の閾値を満たすこと（評価指標一覧.md準拠） |
| ナレッジ変更時（DATA-001更新時） | 全テストケース（MET-001・MET-003）を再実行する | MET-001・MET-003の閾値を満たすこと（評価指標一覧.md準拠） |
| モデルバージョン変更時 | 全テストケース（MET-001・MET-003）を再実行する | MET-001・MET-003の閾値を満たすこと（評価指標一覧.md準拠） |

## 9. 設計上の留意点・リスク

**LLM応答の非決定性による不安定な評価**:
- リスク: 同一テストケースでもスコアが実行のたびに変動し、合否判定が安定しない
- 対策: 同一テストケースでの試行回数は1回とする。各テストケースは1度のみ実行するため、各テストケースごとの集計はしない。LLMの `temperature` はゼロに設定して決定性を高める

**LLM-as-Judge の信頼性**:
- リスク: LLM-as-Judgeが申請種別判断の正誤を誤判定し、実際のエージェント品質と評価スコアが乖離する
- 対策: 判定モデルを評価対象（Claude Sonnet 4.5）とは別にする。判定理由を全件レポートに出力し、初期段階は人手でサンプリング確認を実施する

**フレームワーク間の差異**:
- リスク: 評価フレームワーク（詳細設計で選定）の仕様変更により、評価テストコードが動作しなくなる
- 対策: 評価テストコードはSection 3で定義した抽象概念（テストケース・評価方式・評価結果）を介して記述し、フレームワーク固有APIへの直接依存を最小化する
