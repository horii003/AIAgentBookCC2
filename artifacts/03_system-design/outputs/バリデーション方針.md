## 1. 目的・適用範囲

### 1.1 目的

システム全体で共通するバリデーション方針を定義し、業務ルール・ガードレール要件に基づいた一貫したデータ検証を実現する。

### 1.2 適用範囲

- 全エージェント（AG-001/AG-002/AG-003）の入力データ
- ツールパラメータ（TOOL-001/TOOL-002の入力値）
- セッション状態データ（DATA-005）
- エージェント間メッセージ（委譲リクエスト）

### 1.3 本方針で定義すること

- バリデーションの基本原則とタイミング
- Pydanticモデルの使用方針
- 本システムで使用するデータ型別バリデーション（日付・金額・文字列・列挙型）
- 業務ルールに基づくカスタムバリデーションルール
- バリデーションエラーメッセージの方針

### 1.4 本方針で定義しないこと

- 具体的なPydanticモデルクラスの全定義（データモデル基本設計書に委ねる）
- ツールパラメータの詳細な型定義（ツール基本設計書に委ねる）
- テストコードの実装（評価テスト方式設計書に委ねる）

## 2. バリデーションの基本原則

### 2.1 基本方針

- **早期検証**: 入力がシステムに入った最初の時点でバリデーションを実施し、不正なデータが後続処理に伝播することを防止する
- **型安全性**: Pydantic v2+を使用してすべての入力と状態モデルに型アノテーションを付与し、型不一致を実行前に検出する
- **ユーザーフレンドリー**: バリデーションエラー発生時は技術的な詳細を隠蔽し、日本語で対処方法を含むメッセージを社員に提示する
- **自動正規化**: 表記揺れ（駅名の「〇〇駅」→「〇〇」等）を自動正規化し、後続処理の精度を担保する（BRL-10準拠）
- **一貫性**: すべてのバリデーションロジックをPydanticモデル内に集約し、エージェント・ツール間でルールの重複定義を防止する

### 2.2 バリデーションのタイミング

| タイミング | 対象 | 実装場所 | 根拠 |
|----------|------|---------|------|
| 社員入力受領後（各ターン） | 申請内容テキスト（文字数制限・空入力） | エージェント入力前処理 | FR-011・GRD-001〜003 |
| ツール呼び出し前 | ツールパラメータ（TOOL-001/TOOL-002入力値） | Pydanticモデル | GRD-005・GRD-009 |
| 収集済み申請情報更新時 | 収集済みデータ（日付・金額・交通手段・経費区分） | Pydanticモデル | BRL-03・BRL-07 |
| 申請書生成前（TOOL-002呼び出し前） | 全必須項目充足確認 | GRD-005ハンドラー | GRD-005・BRL-04 |
| セッション状態読み込み時（resume時） | セッション状態ファイル（DATA-005）の整合性 | SessionManager | セッション管理方針 SYS-SESSION-001 |

### 2.3 バリデーション失敗時の扱い

**ValidationError発生時**:
- EX-01（入力例外）に分類してErrorHandlerに渡す（例外処理方針 SYS-EXCEPTION-001準拠）
- エラーメッセージを日本語に変換して社員に提示する
- セッション状態はACTIVEまたはWAITINGのまま継続し、社員に再入力を促す
- 技術的なスタックトレース・フィールドIDはDATA-007（判断ログ）にのみ記録し、社員には提示しない

## 3. Pydanticモデルの使用方針

### 3.1 基本原則

- Pydantic v2モデルによる全入出力データの集中バリデーションを実施する
- Pydantic v2+の `BaseModel` を継承したモデルクラスを使用する
- `field_validator` デコレータでフィールドレベルのカスタムバリデーションを実装する
- `model_validator` デコレータでモデルレベルのクロスフィールドバリデーションを実装する

### 3.2 Pydanticモデルの定義場所

**定義場所**: `models/data_models.py`

**命名規則**:
- 入力モデル: `{用途}Input`（例: `TransportFareInput`, `ExpenseFormInput`）
- 出力モデル: `{用途}Output`（例: `TransportFareOutput`, `ExpenseFormOutput`）
- 状態モデル: `{用途}State`（例: `SessionState`, `ApplicationContext`）

### 3.3 Pydanticモデルの基本構造

```python
from pydantic import BaseModel, field_validator, Field
from typing import Optional, Literal
from datetime import date

class TransportSectionInput(BaseModel):
    travel_date: date
    departure: str
    destination: str
    transport_type: Literal["電車", "バス", "タクシー", "飛行機"]
    purpose: str

    @field_validator("departure", "destination")
    @classmethod
    def normalize_station_name(cls, v: str) -> str:
        return v.removesuffix("駅").strip()

    @field_validator("purpose")
    @classmethod
    def validate_purpose_required(cls, v: str) -> str:
        if not v or not v.strip():
            raise ValueError("業務目的は必須です")
        return v.strip()
```

## 4. 日付のバリデーション

### 4.1 基本原則

- 日付は `YYYY-MM-DD` 形式を標準とし、Pythonの `datetime.date` 型として扱う
- `python-dateutil` を用いて日本語形式の日付文字列（「2026年7月3日」等）を自動変換する
- 未来日付の入力は申請日（移動日・購入日）として許容しない

### 4.2 日付の定義

**標準形式**: `YYYY-MM-DD`（例: `2026-07-03`）

**許容入力形式**:
- `YYYY-MM-DD`（標準）
- `YYYY/MM/DD`（自動変換）
- `YYYY年MM月DD日`（自動変換）

### 4.3 日付バリデーションの実装

```python
from pydantic import field_validator
from datetime import date
from dateutil import parser as dateutil_parser

@field_validator("travel_date", "purchase_date", mode="before")
@classmethod
def parse_date(cls, v):
    if isinstance(v, date):
        return v
    try:
        return dateutil_parser.parse(str(v)).date()
    except (ValueError, TypeError):
        raise ValueError(f"日付の形式が不正です。YYYY-MM-DD形式で入力してください（例: 2026-07-03）")
```

### 4.4 日付の特殊ルール

**申請期限チェック（BRL-14）**:
- 申請日（申請書生成時点の日付） - 経費発生日（移動日または購入日） ≤ 3ヶ月（JD-07）
- 期限超過の場合は申請書生成を停止せず、社員に通知してから申請書生成を継続する（EX-03の代替処理）
- 詳細な期限計算ロジックはハンドラー基本設計書で定義する

## 5. 金額のバリデーション

### 5.1 基本原則

- 金額は0以上の整数（円単位）として扱う
- 小数・負数・非数値は許容しない
- 上長承認要否判定（BRL-09/BRL-13）のための閾値チェックはバリデーションではなくビジネスロジックとして別途ハンドラーで実装する

### 5.2 金額の定義

**標準形式**: 非負整数（例: `10000`）

**許容入力**:
- 半角数字のみ（カンマ区切り `10,000` は自動除去）
- 「円」を含む文字列 `10000円` は自動除去

**上限値**: 要件上未定義

### 5.3 金額バリデーションの実装

```python
from pydantic import field_validator

@field_validator("amount", mode="before")
@classmethod
def parse_amount(cls, v):
    if isinstance(v, int) and v >= 0:
        return v
    try:
        cleaned = str(v).replace(",", "").replace("円", "").strip()
        amount = int(cleaned)
        if amount < 0:
            raise ValueError("金額は0以上の整数で入力してください")
        return amount
    except (ValueError, TypeError):
        raise ValueError("金額の形式が不正です。半角数字で入力してください（例: 10000）")
```

### 5.4 金額の特殊ルール

**交通費上長承認判定（BRL-09）**: 交通費合計 > 10,000円の場合に通知（ハンドラーで実装）

**経費上長承認判定（BRL-13）**: 経費合計 > 5,000円の場合に通知（ハンドラーで実装）

## 6. 文字列のバリデーション

### 6.1 基本原則

- 空文字・空白のみの文字列は必須フィールドとして許容しない
- 入力文字数上限は500文字（FR-011・GRD-001）
- 前後の空白は自動トリム（`strip()`）する

### 6.2 文字列の定義

**文字数上限**: 500文字（FR-011）

**必須チェック対象フィールド**:
- 申請内容テキスト（D-001）
- 業務目的（BRL-15）
- 申請者名

### 6.3 文字列バリデーションの実装

```python
from pydantic import field_validator

@field_validator("application_text", "purpose", "applicant_name")
@classmethod
def validate_non_empty_string(cls, v: str) -> str:
    v = v.strip()
    if not v:
        raise ValueError("この項目は必須です。入力内容を確認してください")
    if len(v) > 500:
        raise ValueError(f"入力内容が長すぎます。500文字以内で入力してください（現在: {len(v)}文字）")
    return v
```

### 6.4 文字列の特殊ルール

- 申請内容テキスト（D-001）の500文字超過は EX-01（入力例外）として処理し、社員に短縮を促す（FR-011準拠）
- 業務目的（D-006の一部）は全申請種別で必須（BRL-15）

## 7. カスタムバリデーションルール

### 7.1 列挙型フィールドのバリデーションと正規化

**交通手段フィールド**:

```python
from typing import Literal
from pydantic import field_validator

transport_type: Literal["電車", "バス", "タクシー", "飛行機"]

@field_validator("transport_type", mode="before")
@classmethod
def normalize_transport_type(cls, v: str) -> str:
    mapping = {
        "train": "電車",
        "鉄道": "電車",
        "bus": "バス",
        "taxi": "タクシー",
        "タクシー車": "タクシー",
        "plane": "飛行機",
        "airplane": "飛行機",
        "飛行": "飛行機",
    }
    normalized = mapping.get(v, v)
    if normalized not in ["電車", "バス", "タクシー", "飛行機"]:
        raise ValueError(
            f"交通手段「{v}」は対象外です。電車・バス・タクシー・飛行機のいずれかを入力してください"
        )
    return normalized
```

**経費区分フィールド**:

```python
expense_category: Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]

@field_validator("expense_category", mode="before")
@classmethod
def normalize_expense_category(cls, v: str) -> str:
    mapping = {
        "事務用品": "事務用品費",
        "文房具": "事務用品費",
        "宿泊": "宿泊費",
        "ホテル": "宿泊費",
        "資格": "資格精算費",
        "研修": "資格精算費",
        "その他": "その他経費",
    }
    normalized = mapping.get(v, v)
    if normalized not in ["事務用品費", "宿泊費", "資格精算費", "その他経費"]:
        return "その他経費"  # BRL-12: 判断不能の場合は「その他経費」を適用
    return normalized
```

### 7.2 駅名正規化バリデーション（BRL-10）

```python
@field_validator("departure", "destination", mode="before")
@classmethod
def normalize_station_name(cls, v: str) -> str:
    v = v.strip()
    if v.endswith("駅"):
        v = v[:-1]
    if not v:
        raise ValueError("駅名を入力してください")
    return v
```

### 7.3 HitL承認結果バリデーション

```python
from typing import Literal

hitl_result: Literal["OK", "修正", "キャンセル"]

@field_validator("hitl_result", mode="before")
@classmethod
def normalize_hitl_result(cls, v: str) -> str:
    mapping = {
        "ok": "OK",
        "yes": "OK",
        "はい": "OK",
        "修正する": "修正",
        "cancel": "キャンセル",
        "いいえ": "キャンセル",
    }
    normalized = mapping.get(str(v).lower(), v)
    if normalized not in ["OK", "修正", "キャンセル"]:
        raise ValueError(
            f"「{v}」は無効な選択です。OK・修正・キャンセルのいずれかを選択してください"
        )
    return normalized
```

## 8. バリデーションのテスト

### 8.1 テスト観点

| 観点 | 内容 |
|-----|------|
| 正常系 | 有効な入力値がバリデーションを通過すること |
| 異常系 | 無効な入力値（空文字・型不一致・範囲外）でエラーが発生すること |
| 境界値 | 文字数上限（500文字）・金額0円・金額閾値（10,000円/5,000円）が正しく処理されること |
| 正規化 | 表記揺れ（「渋谷駅」→「渋谷」等）が正規化されること |
| エラーメッセージ | エラーメッセージが日本語で対処方法を含むこと |

### 8.2 テストケースの例

| テストケース | 入力 | 期待結果 |
|------------|------|---------|
| 申請内容テキスト正常 | `"交通費を精算したい"` | バリデーション通過 |
| 申請内容テキスト空 | `""` | エラー：必須 |
| 申請内容テキスト501文字 | 501文字の文字列 | エラー：500文字超過 |
| 駅名正規化 | `"渋谷駅"` | `"渋谷"` |
| 交通手段正規化 | `"train"` | `"電車"` |
| 交通手段不正 | `"自転車"` | エラー：対象外の交通手段 |
| 日付正常（YYYY/MM/DD） | `"2026/07/03"` | `date(2026, 7, 3)` |
| 日付不正 | `"20260703"` | エラー：形式不正（実装依存）または変換 |
| 金額正常（カンマあり） | `"10,000"` | `10000` |
| 金額負数 | `"-100"` | エラー：0以上の整数 |
| HitL承認正規化 | `"ok"` | `"OK"` |
| HitL承認不正 | `"yes please"` | エラー：無効な選択 |
| 経費区分判断不能 | `"雑費"` | `"その他経費"` |

> バリデーションテストの詳細はテスト設計書（評価テスト方式設計書・評価テスト詳細設計書）に委ねる

## 9. 設計上の留意点・リスク

### 9.1 LLMの出力とバリデーション

**リスク**: LLMが生成した値（経費区分・交通手段等）がバリデーションで失敗した場合、エージェントのターンが中断されるリスクがある

**対策**: 経費区分の判断不能は「その他経費」にフォールバックする設計とし（BRL-12準拠）、交通手段は許容する別表記をmappingに網羅することでLLM出力の揺れを吸収する

### 9.2 申請期限チェックと日付処理

**リスク**: タイムゾーン設定によって期限計算結果が変わる可能性がある

**対策**: 日付比較は `datetime.date` 型のみで行い、タイムゾーンを含む `datetime.datetime` 型を申請期限計算に使用しない。詳細はハンドラー基本設計書で定義する

### 9.3 Pydanticモデルの二重定義

**リスク**: エージェント・ツールそれぞれにバリデーションロジックを実装すると、ルールの不整合が発生するリスクがある

**対策**: `models/data_models.py` を唯一のバリデーション定義場所とし、エージェント・ツールはこのモデルを参照してバリデーションを実施する

## 9b. 運賃データのバリデーション方針

**運賃データの読み込み方式**:
- 電車経路・運賃データ（DATA-003：`data/fares/train_fares.json`）および固定運賃データ（DATA-004：`data/fares/fixed_fares.json`）はJSONファイルからの静的読み込みとする
- アプリケーション起動時またはTOOL-001の初回呼び出し時にロードし、メモリ上に保持する（都度ファイルI/Oを行わない）
- ロード時にPydanticモデルを使用してデータ形式の整合性を検証する
