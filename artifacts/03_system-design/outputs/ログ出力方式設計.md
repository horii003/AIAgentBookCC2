## 1. 目的・適用範囲

### 1.1 目的

ログ出力要件定義（SR-14）で定義された要件を実装方式として具体化し、全エージェント・全コンポーネントで一貫したログ出力を実現する。

### 1.2 適用範囲

- AG-001（申請受付窓口エージェント）
- AG-002（交通費精算申請エージェント）
- AG-003（経費精算申請エージェント）
- TOOL-001（交通費計算ツール）・TOOL-002（申請書生成ツール）
- SessionManager・ErrorHandler・各カスタムフック

### 1.3 本方針で定義しないこと

- 個々のハンドラーのメソッド詳細設計（ハンドラー基本設計書に委ねる）
- ログファイルのクラウドストレージ転送設計（本システムはファイルストレージのみ）
- ログ管理ツール・監視ダッシュボードの構築（本システムスコープ外）

## 2. ログ出力基盤の構成

### 2.1 使用ライブラリ・モジュール

| 区分 | 名称 | 用途 |
|-----|------|------|
| 標準ライブラリ | `logging`（Python標準） | ログレベル管理・フォーマット設定・ファイルハンドラー |
| 標準ライブラリ | `logging.handlers.RotatingFileHandler` | ログファイルローテーション |
| 標準ライブラリ | `json` | JSON構造化ログの生成 |
| フレームワーク | Strands Agents `callback_handler` | トレースログ・ツール実行ログのフック取得 |

### 2.2 ログレベル設定方針

| 対象 | ログレベル | 理由 |
|-----|----------|------|
| 本システムのLogger（ルート） | 環境変数（LOG_LEVEL）で制御。未設定時のデフォルトは WARNING | ログレベルは環境変数で制御し、開発・本番環境での使い分けを可能にする |
| Strands Agentsフレームワーク（`strands`） | WARNING | フレームワーク内部のデバッグ出力を抑制し、アプリケーションログの視認性を確保する |
| botocore・boto3 | WARNING | AWS SDK内部ログを抑制する |
| デバッグ時 | DEBUG | TRC-004（セッション状態更新ログ）は障害時のみDEBUGレベルで記録する |

**ログ設計の基本原則**:
- ログ設定（レベル・出力先・フォーマット）はエントリーポイント（`main.py`）で一元管理する
- ログ出力は各モジュール（エージェント・ツール・ハンドラー等）が責任を持って行う
- ログレベルは環境変数（`LOG_LEVEL`）で制御し、未設定時のデフォルトは `WARNING` とする
- ユーザー向けメッセージ（申請者への応答）と技術ログ（デバッグ・監査用）は分離する
- 各モジュールが付加するコンテキスト情報の詳細は基本設計・詳細設計で定義する

### 2.3 出力先・ローテーション設定

| 出力先 | 対象ログレベル | ローテーション方式 | 備考 |
|-------|-------------|----------------|------|
| `logs/app.log` | INFO以上 | RotatingFileHandler: 最大10MB / 最大5世代 | 監査・操作・トレース・エラーログを統合出力 |
| コンソール（stderr） | WARNING以上 | なし | 開発環境のリアルタイム確認用。本番環境では抑制可 |
| `logs/audit.log` | INFO以上（監査ログのみ） | RotatingFileHandler: 最大50MB / 最大10世代 | DATA-007（判断ログ）に対応するファイル。監査ログ（AUD-001〜004）専用 |
| `logs/conversation.log` | INFO以上（会話ログのみ） | RotatingFileHandler: 最大50MB / 最大10世代 | DATA-006（会話ログ）に対応するファイル。OPE-001専用 |

## 3. 相関ID設計

### 3.1 相関IDの種類と役割

| ID名 | 役割 | 生成タイミング | 生成箇所 |
|-----|------|-------------|---------|
| session_id | 1申請フロー全体を通じたセッション識別子 | AG-001が申請者名を受け付けた時点 | SessionManager |
| request_id | 申請フロー全体の相関ID（ログ追跡用） | セッション開始時（session_idと同タイミング） | SessionManager |
| step_id | 会話フローの各ステップ識別子（CF-001 Step1等） | 各ステップ開始時 | SessionManager |
| tool_call_id | ツール実行単位の識別子（TOOL-001/TOOL-002の各呼び出し） | ツール呼び出し時 | ToolHandler |

### 3.2 相関IDの形式

| ID名 | 形式 | 例 |
|-----|------|---|
| session_id | `sess_` + UUID形式 | `sess_a1b2c3d4-e5f6-7890-abcd-ef1234567890` |
| request_id | `req_` + UUID形式 | `req_b2c3d4e5-f607-8901-bcde-f01234567891` |
| step_id | `{CF-ID}_step{番号}` | `CF-001_step1`、`CF-002_step3` |
| tool_call_id | `{tool_id}_{timestamp}` | `TOOL-001_20260703120000` |

### 3.3 マルチエージェント間での伝播方式

AG-001が生成した `request_id` と `session_id` はセッション状態ファイル（DATA-005）に書き込まれ、AG-002/AG-003がDATA-005を参照することで同一の相関IDを使用する。

**伝播フロー**:
1. AG-001: `request_id` と `session_id` を生成 → DATA-005に書き込む
2. AG-001からAG-002/AG-003への委譲リクエスト（LNK-001/LNK-002）にも `request_id` を含める
3. AG-002/AG-003: DATA-005から `request_id` を読み込み、以降の全ログに付与する

### 3.4 全ログへの付与方式

すべてのログエントリは `request_id` を必須フィールドとして含む。Pythonのloggingでは `extra` パラメータを使用して付与する。

```python
logger.info(
    "申請種別判断完了",
    extra={
        "request_id": request_id,
        "session_id": session_id,
        "step_id": step_id,
        "agent_id": "AG-001",
        "event_type": "JUDGMENT",
    }
)
```

## 4. ログ種別ごとの実装方式

### 4.1 監査ログ（Audit Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| AUD-001 | AG-001がJD-01またはJD-02を適用した直後 | AG-001の判断ロジック内 | `audit_logger.info(...)` でDATA-007（`logs/audit.log`）に構造化JSON出力 |
| AUD-002 | AG-002/AG-003がJD-05またはJD-06を適用した直後 | AG-002/AG-003のチェックロジック内 | `audit_logger.info(...)` でDATA-007（`logs/audit.log`）に構造化JSON出力 |
| AUD-003 | AG-002/AG-003がJD-07を適用した直後 | AG-002/AG-003の期限チェックロジック内 | `audit_logger.info(...)` でDATA-007（`logs/audit.log`）に構造化JSON出力 |
| AUD-004 | AG-002がJD-08、AG-003がJD-09を適用した直後 | AG-002/AG-003の承認判定ロジック内 | `audit_logger.info(...)` でDATA-007（`logs/audit.log`）に構造化JSON出力 |

### 4.2 操作ログ（Operation Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| OPE-001 | 各会話ターン終了後（エージェント応答後） | LoopControlHookまたはSessionManager | `conversation_logger.info(...)` でDATA-006（`logs/conversation.log`）に構造化JSON出力。申請内容テキスト・社員回答はマスキング後に要約のみ記録 |
| OPE-002 | エージェント間委譲時（LNK-001/LNK-002発火時） | AG-001の委譲ロジック内 | `app_logger.info(...)` でDATA-007（`logs/audit.log`）に構造化JSON出力 |

### 4.3 トレースログ（Trace Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| TRC-001 | ツール呼び出し開始・終了時（TOOL-001/TOOL-002） | ToolHandler（カスタムフック） | Strands Agentsの `callback_handler` フックを使用してツール実行イベントを捕捉し、`app_logger.info(...)` で記録 |
| TRC-003 | エージェント間委譲時（LNK-001/LNK-002） | AG-001の委譲ロジック内 | `app_logger.info(...)` で委譲開始・終了を記録 |
| TRC-004 | セッション状態更新時 | SessionManager | `app_logger.debug(...)` でDATA-005更新内容の要約を記録（障害調査時のみ参照） |

#### エージェントSDK/フレームワークフックとの連携方式

Strands Agentsは `callback_handler` インタフェースを提供する。`LoopControlHook` クラスに `callback_handler` を実装し、各エージェントに設定することで、ツール実行・LLM推論サイクルのイベントを捕捉してログを記録する。

```python
from strands import Agent

class LoggingCallbackHandler:
    def on_tool_start(self, tool_name: str, tool_input: dict) -> None:
        app_logger.info("ツール実行開始", extra={
            "event_type": "TOOL_START",
            "tool_name": tool_name,
            "input_summary": _mask_sensitive(tool_input),
            "request_id": current_request_id,
        })

    def on_tool_end(self, tool_name: str, tool_output: dict, duration_ms: int) -> None:
        app_logger.info("ツール実行完了", extra={
            "event_type": "TOOL_END",
            "tool_name": tool_name,
            "output_summary": _mask_sensitive(tool_output),
            "duration_ms": duration_ms,
            "request_id": current_request_id,
        })
```

#### 推論トレース（内部推論）の扱い

| 観点 | 方針 |
|-----|------|
| ユーザーへの表示 | **禁止**。エージェントの内部推論（Chain-of-Thought）は社員には表示しない（GRD-008準拠・根拠提示方針準拠）。`callback_handler` でエージェント応答からChain-of-Thought部分を除去してから社員に提示する |
| ログファイルへの記録 | **記録しない**。ログ出力要件定義（SR-14）の原則「内部推論（Chain-of-Thought）は保存しない」に従い、`logs/app.log` および `logs/audit.log` にも記録しない |
| 記録する場合の形式 | 記録しない（上記参照） |

### 4.4 エラーログ（Error Log）

| ログID | 記録タイミング | 実装箇所 | 実装方式 |
|-------|-------------|---------|---------|
| ERR-001 | AG-001がJD-02（申請種別判断不能）を適用した時 | AG-001の判断ロジック内 | `app_logger.info(...)` で記録（正常フローのため WARNING以下）。AUD-001と統合してDATA-007に出力 |
| ERR-002 | AG-002/AG-003がJD-06（チェック不備）を適用した時 | AG-002/AG-003のチェックロジック内 | `app_logger.info(...)` で記録（正常フローのため WARNING以下）。AUD-002と統合してDATA-007に出力 |
| ERR-003 | TOOL-001/TOOL-002の実行失敗時 | ToolHandler | `app_logger.error(...)` でスタックトレース含めてDATA-007に記録 |
| ERR-004 | Amazon Bedrock推論エラー発生時 | ErrorHandler | `app_logger.error(...)` でエラー概要（スタックトレース含む）をDATA-007に記録 |
| ERR-005 | セッション障害発生時 | SessionManager | `app_logger.error(...)` でエラー概要をDATA-007に記録 |

#### ErrorHandlerとの連携方式

ErrorHandlerがEX-01〜EX-08の例外を受け取ったタイミングでログ出力を実施する。エラーログの内容は以下の4要素を必須とする。

| フィールド | 内容 |
|----------|------|
| `error_type` | 例外分類（EX-01〜EX-08） |
| `error_message_summary` | 技術的詳細を除いたエラー概要（ユーザーには非表示） |
| `exc_info` | スタックトレース（ログファイルのみ。ユーザーには非表示） |
| `request_id` | 相関ID |

### 4.5 デバッグログ（Debug Log）

- `app_logger.debug(...)` を使用し、本番環境ではINFOレベル設定により自動抑制される
- TRC-004（セッション状態更新）のみDEBUGログに記録する
- セッション状態ファイルの更新内容（収集済み申請情報等）は機密情報のため、DEBUGログには更新フィールド名のみを記録し、値は含めない

---

### 4.6 メトリクスログ（Metrics Log）

LLMOpsの継続的改善サイクルで活用するためのコスト・パフォーマンスメトリクスを記録する。

| 計測対象 | 記録タイミング | 実装箇所 | 実装方式 |
|---------|-------------|---------|---------|
| LLM推論コスト（入出力トークン数） | Amazon Bedrock APIレスポンス受信後 | `callback_handler` のLLM応答イベントフック | 入力トークン数・出力トークン数をINFOレベルで `logs/app.log` に記録。Bedrockのレスポンスメタデータから取得 |
| 会話ターン応答時間 | 各ターン開始〜エージェント応答完了 | LoopControlHook | 開始時刻・終了時刻・経過時間（ms）をINFOレベルで記録 |
| ツール実行処理時間 | ツール実行開始〜終了 | LoggingCallbackHandler（on_tool_start/on_tool_end） | `duration_ms` をTRC-001ログに含めて記録（再掲） |

## 5. ログフォーマット定義

### 5.1 共通フォーマット

| 項目 | 内容 |
|-----|------|
| フォーマット文字列 | `%(asctime)s %(levelname)s %(name)s %(message)s` + extraフィールド（JSON埋め込み） |
| エンコーディング | UTF-8 |
| タイムスタンプ形式 | ISO 8601（`%Y-%m-%dT%H:%M:%S.%f`）例: `2026-07-03T12:00:00.123456` |

### 5.2 構造化ログの形式

監査ログ・操作ログ・トレースログ・エラーログはJSON形式の構造化ログとして出力する。

```json
{
  "timestamp": "2026-07-03T12:00:00.123456",
  "level": "INFO",
  "log_id": "AUD-001",
  "request_id": "req_b2c3d4e5-f607-8901-bcde-f01234567891",
  "session_id": "sess_a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "step_id": "CF-001_step2",
  "agent_id": "AG-001",
  "event_type": "JUDGMENT",
  "applied_rule": "JD-01",
  "result": "交通費精算申請",
  "judgment_status": "特定"
}
```

**Pythonの実装方針**:
- `logging.Formatter` をカスタマイズしてJSONフォーマッターを作成する
- `extra` パラメータで渡したフィールドをJSON出力に含める
- 詳細な実装はハンドラー基本設計書に委ねる

## 6. マスキング実装方式

| 対象データ | マスキング方式 | 適用ログ種別 | 実装箇所 |
|----------|-------------|-----------|---------|
| 申請内容テキスト（D-001）全文 | ログに含めない（要約のみ記録） | OPE-001（会話ログ） | LoopControlHookのターン終了処理 |
| 社員回答（D-006）全文 | ログに含めない（要約のみ記録） | OPE-001（会話ログ） | LoopControlHookのターン終了処理 |
| CVV（3〜4桁セキュリティコード） | 記録禁止（ブロック後のため入力されない前提） | 全ログ | GRD-002ハンドラー（入力段階でブロック済み） |
| PII（氏名・メール・電話番号等） | `[MASKED]` に置換 | AUD-002（申請書内容を含む場合） | マスキングユーティリティ関数（`_mask_sensitive()`） |
| 申請書ドラフト内容（D-009） | 申請書パスのみ記録（内容は含めない） | AUD-002 | AG-002/AG-003のチェックロジック |

**マスキング実装方針**:
- `_mask_sensitive(data: dict) -> dict` ユーティリティ関数を `handlers/masking.py` に実装する
- ツール入出力の要約記録（TRC-001）時にこの関数を通じてマスキングを適用する
- 詳細な実装はハンドラー基本設計書に委ねる

## 7. 例外処理方針・実行制御方針との整合

### 7.1 例外ログの出力フロー

1. エージェント/ツール: 例外を捕捉 → ErrorHandlerに渡す
2. ErrorHandler: EX-01〜EX-08に分類 → ログ出力ロジックを呼び出す
3. ログ出力: `error_type`・`error_message_summary`・`exc_info`・`request_id` を含む構造化JSONを `logs/app.log` に出力
4. 社員向けメッセージ: 技術的詳細を除いた日本語メッセージのみを返却する（ログ内容は非公開）

例外処理方針（SYS-EXCEPTION-001）で定義された EX-01〜EX-08 の各ログIDは以下に対応する：
- EX-04/EX-05/EX-07/EX-08: ERR-003〜ERR-005（技術エラーログ）
- EX-01/EX-02/EX-03: ERR-001〜ERR-002（業務エラーログ）または通常の操作ログとして処理

### 7.2 実行制御ログの出力フロー

実行制御方針（SYS-EXEC-CTRL-001）15節で定義された実行制御ログ項目を以下のように出力する：
- `event_type`（START/COMPLETE/PAUSE/TERMINATE/TOOL_EXEC/APPROVAL等）: LoopControlHookまたはHumanApprovalHookが `app_logger.info(...)` で記録
- HitL承認結果（OK/修正/キャンセル）: HumanApprovalHookが AUD-004と統合してDATA-007に記録
- 対話回数（iteration）: SessionManagerがDATA-005更新時に OPE-001ログにも含めて記録

## 8. 設計上の留意点・リスク

**ログ肥大化リスク**:
- 対話30回 × セッション多数の場合、`logs/conversation.log` が急速に肥大化する可能性がある
- 対策: RotatingFileHandler（最大50MB / 10世代）を設定する。セッション終了後に不要なセッションのログを圧縮するバッチ処理は要件上未定義

**マスキング漏れリスク**:
- `_mask_sensitive()` 関数の未適用によって機密情報がログに記録されるリスクがある
- 対策: ツール呼び出しのログ記録は必ず `_mask_sensitive()` を通じて出力する。開発時のコードレビューでチェックする

**相関ID未設定リスク**:
- セッション開始前（SessionManager初期化前）のエラーには `request_id` が存在しない
- 対策: セッション開始前のエラーには `request_id: "N/A"` を設定して記録する

## 9. ログの活用方針（LLMOps連携）

### 9.1 評価・改善サイクルへの接続

収集したログを以下の用途でLLMOpsの評価・改善サイクルに活用する。

| ログ種別 | 活用目的 | 活用方法 |
|---------|---------|---------|
| 操作ログ（OPE-001） | 会話品質評価・ユーザー体験改善 | 会話ターン数・申請種別判断精度・HitL承認結果を集計し、エージェントのシステムプロンプト改善に活用する |
| トレースログ（TRC-001〜003） | ツール実行性能分析・障害原因調査 | ツール実行時間・失敗率を分析し、TOOL-001の経路テーブル検索精度改善や TOOL-002の生成精度改善に活用する |
| メトリクスログ | コスト管理・応答時間最適化 | 1申請フローあたりのトークン消費量・応答時間を分析し、システムプロンプトの圧縮・モデル設定の最適化に活用する |
| エラーログ（ERR-001〜005） | 品質劣化検知・障害対応 | エラー率・エラー種別を定期的に確認し、EX-02（判断不能）の頻発時はシステムプロンプトやナレッジの改善を実施する |
| 監査ログ（AUD-001〜004） | 申請精度分析・業務ルール検証 | 申請種別判断精度・申請書チェック不備率・期限超過率を集計し、ガードレール設定や業務ルール改善に活用する |

### 9.2 品質劣化の検知方針

- **エラー率の閾値**: EX-02（判断不能）の発生率が全セッションの20%を超えた場合に、システムプロンプトの見直しを検討する（閾値は要件上未定義）
- **応答時間の監視**: 会話ターン応答時間の平均が要件上未定義の閾値を超えた場合に性能劣化として検知する
- **定期レビュー**: OPE-001（会話ログ）の要約を定期的にレビューし、社員の不満点・申請フロー中断率を確認する
