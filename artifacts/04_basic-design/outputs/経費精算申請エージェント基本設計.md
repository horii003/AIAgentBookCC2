## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001 から委譲された経費精算申請フロー全体（情報収集・領収書LLMビジョン解析・経費区分判断・HitL承認・申請書生成・申請書チェック・申請期限チェック）を自己完結で実行する
- 社員との対話を通じて経費情報（購入日・店舗名・品目・経費区分・金額・業務目的）を一括確認で収集する。領収書画像がある場合はLLMのビジョン機能で情報を抽出する
- 収集した情報を申請書テンプレート（DATA-002）にマッピングして Excel 申請書ドラフトを生成し、社員が確認できる状態にする

### 2.2 主な責務
- **申請情報収集**: CF-003 の会話ステップに従って経費情報を一括確認・追加収集する（BRL-03: 不足情報の追加収集）
- **領収書情報抽出**: 社員が領収書画像を提供した場合に、LLM のネイティブビジョン機能で購入日・店舗名・品目・金額を抽出する（BRL-11）
- **経費区分自動判断**: 品目から経費区分（事務用品費・宿泊費・資格精算費・その他経費）を判断する（BRL-12: 判断不能時は「その他経費」を適用）
- **上長承認要否判断**: 経費合計が5,000円を超える場合に上長承認が必要な旨を通知する（BRL-13・JD-09）
- **申請期限チェック**: 経費発生日から3ヶ月以内かどうかをチェックし、超過時は社員に通知する（BRL-14・JD-07）
- **HitL承認**: Excel生成前に申請情報サマリーを提示し社員の承認（OK/修正/キャンセル）を確認する（GRD-009）
- **申請書ドラフト生成**: HitL承認（OK）後に TOOL-002（申請書生成ツール：経費用）で Excel 申請書ドラフトを生成する（BRL-04: 収集情報のみ使用）
- **申請書チェック**: 申請ルール（DATA-001）に基づいて申請書の不備を確認し結果を社員に提示する（BRL-05・JD-06）

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001が担当
- **交通費計算（TOOL-001）**: AG-002が担当（AG-003は交通費計算ツールを使用しない）
- **申請書の実際の提出**: 社員がシステム外で実施（自律度Lv0：実行不可）
- **HitL承認制御（TOOL-002呼び出し制御）**: HumanApprovalHookがハンドラー設計で担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（DATA-001）・申請書テンプレート（DATA-002）のファイル参照；領収書画像の読み取り（LLMビジョン） | ACT-READ-01/02/03 |
| 提案 | 不足情報の特定・経費区分の自動判断・上長承認要否の判断・申請書チェック結果の提示 | ACT-DEC-02；ACT-DEC-03（COND-002）；ACT-DEC-05（COND-004）；ACT-DEC-04（COND-006） |
| 生成 | 申請書ドラフト（Excel）生成・確認質問生成 | ACT-GEN-01（HitL承認後）；ACT-GEN-02 |
| 実行（承認必須） | TOOL-002（申請書生成）の実行はHitL承認（OK）後のみ可能 | P-EXEC-001；APR-002（金額閾値超過時） |

### 3.2 判断権限の範囲
- **自律判断可能**: 経費情報の充足判断（JD-03/JD-04）；領収書情報抽出（BRL-11/ACT-READ-03）；経費区分自動判断（BRL-12）；申請期限チェック（BRL-14/JD-07）；申請ルール準拠チェック（BRL-05/JD-06）
- **人間確認必要**: HitL承認（OK/修正/キャンセル）の選択；経費区分自動判断結果の確認（COND-002）；上長承認要否通知の受領（COND-004：金額5,000円超）；申請書チェック不備時の修正
- **実行不可**: 申請書の自動送信・実際の提出操作（GRD-010）；HitL承認なしの TOOL-002 呼び出し（GRD-009）；交通費計算ツール（TOOL-001）の呼び出し

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
委譲受付（invocation_state受取）
→ 必須項目一括確認（Step1）
→ 申請情報回答収集・領収書画像提供（Step2）
→ 領収書情報抽出（LLMビジョン）（Step3：画像あり時のみ）
→ 抽出結果・収集状況確認（Step4）
→ 社員確認・修正（Step5）
→ 上長承認要否通知（金額閾値超過時）（Step6）
→ 申請期限チェック（Step7）
→ HitL承認確認（Step8-9）
→ 申請書ドラフト生成（TOOL-002：経費用）（Step10）
→ 申請書チェック結果提示（Step11）
→ 終了
```

**フローの詳細**: 詳細設計書を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理。情報収集完了 → 上長承認通知 → 期限チェック → 申請書ドラフト提示（テキスト整理） → HitL承認（BeforeToolCallEvent） → 申請書生成ツール実行 → チェック の順序を厳守する
- **申請書ドラフト提示とツール実行の分離**: 収集済み申請情報のテキスト整理・提示（ドラフト提示ステップ）とTOOL-002呼び出し（ツール実行ステップ）を明確に分離する。HitL承認（HumanApprovalHook）は BeforeToolCallEvent でトリガーし、社員の承認（OK）後にはじめて TOOL-002 を実行する。承認前に TOOL-002 を実行してはならない
- **処理をスキップする考え方**: 領収書画像がない場合は Step3（LLMビジョン抽出）をスキップして手動入力（Step4）へ進む；上長承認要否通知は金額が5,000円以下の場合スキップ（JD-09）；HitL承認で「キャンセル」を選択した場合はフロー終了
- **同一処理を複数回実行**: 経費情報が複数ある場合に Step1〜5 を経費件数分繰り返す；HitL承認で「修正」を選択した場合に情報収集ループへ戻る（Step1へ）；対話回数30回上限あり（FR-012）

### 4.3 状態管理
- **収集済み申請情報（D-007）**: `CollectedExpenseInfo` モデル（経費リスト・合計金額）としてセッション状態ファイル（DATA-005）に保存する
- **HitL承認結果**: DATA-005 の `hitl_result` フィールドに保存し、TOOL-002 呼び出し前の前提条件として参照する
- **対話回数**: DATA-005 の `iteration` フィールドで管理する（上限30回：FR-012）

## 6. ツール設計

各エージェントが使用するツールは業務機能（計算・生成など）に特化したツールのみとする。

### 6.1 利用ツール

- **TOOL-002（申請書生成ツール：経費用）**: CF-003 Step10（HitL承認OK後）で経費精算申請書ドラフト（Excel）を生成するために呼び出す（再試行なし・HitL承認後のみ呼び出し可能）

領収書画像からの情報抽出は strands-agents の LLM ネイティブビジョン機能（外部ツール不要）で実行する。AG-003 は TOOL-001（交通費計算ツール）を呼び出さない。

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 経費精算申請フロー（CF-003）全体の処理順序と判断ルールを定義する
- 経費情報の一括確認・追加収集の方法と領収書画像からの情報抽出方法を定義する
- 申請期限チェック（BRL-14）を申請日から遡って3ヶ月以内かどうかで判断するルールを定義する

### 7.2 プロンプト構成要素
- **役割定義**: 経費精算申請エージェントとして経費精算申請フロー全体を担当する
- **処理フロー**: CF-003 の会話ステップ（Step1〜12）に準じた処理順序；一括確認方式
- **判断プロトコル**: JD-03/JD-04（充足判断）；JD-07（申請期限超過）；JD-09（上長承認要否）；BRL-05/JD-06（申請書チェック）；BRL-12（経費区分自動判断）
- **主な責任**: 経費情報の収集；領収書LLMビジョン解析（BRL-11）；経費区分自動判断（BRL-12）；HitL承認後の申請書生成（BRL-04/TOOL-002）；申請書チェック（BRL-05）
- **動的情報（該当する場合）**: 申請日（application_date）・申請期限の基準日（申請日から遡って3ヶ月前：BRL-14）を実行時に埋め込む
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合は、呼び出し元エージェント（AG-001）に要約して返す

**プロンプト全文**: 詳細設計書を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-003 Step7 の申請期限チェック（BRL-14）で「経費発生日から3ヶ月以内」の判断をLLMが行うため、実行時の申請日（application_date）と3ヶ月前の基準日をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 申請日（application_date）・申請期限基準日（申請日 - 90日） |

### 7.4 主要な設計意図
- **LLMビジョンによる領収書情報抽出**: strands-agents の LLM ネイティブビジョン機能を使用し、外部ツールなしで領収書画像から購入日・店舗名・品目・金額を抽出する（BRL-11）
- **経費区分自動判断（BRL-12）**: 品目から経費区分を LLM が自動判断し、判断不能時は「その他経費」を適用する。自動判断結果は必ず社員に提示して確認を促す（COND-002）
- **ドラフト提示とツール実行の分離**: 収集済み申請情報のテキスト整理・提示（ドラフト提示ステップ）と TOOL-002 呼び出し（ツール実行ステップ）を明確に分離する。HumanApprovalHook は BeforeToolCallEvent でトリガーし、承認（OK）後にはじめて TOOL-002 を実行する。HitL承認は1回のみ発生させる（GRD-009準拠）
- **HitL承認前のツール呼び出し禁止**: GRD-009 に従い、TOOL-002 の呼び出しはHitL承認（OK）後のみ許可する

## 10. 入力データ設計

### 10.1 初期化時の入力
該当なし（invocation_state で申請者名・申請日を受け取る）

### 10.2 対話ループでの入力
- **申請情報（各経費）**: 購入日・店舗名・品目・経費区分・金額・業務目的（自然文）
- **領収書画像**: オプション。社員がアップロードした場合に LLMビジョンで情報を抽出する
- **HitL承認結果**: 「OK」「修正」「キャンセル」のいずれか
- **特殊コマンド**:
  - 対話回数30回到達時: 処理終了通知（FR-012）
  - HitL承認で「キャンセル」選択時: 申請取り消し通知・フロー終了
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る

### 10.3 invocation_stateの入力
AG-001 から渡される情報:
```python
{
    "applicant_name": str,      # 申請者名
    "application_date": str     # 申請日（システム日付：YYYY-MM-DD形式）
}
```

> ※ session_id はAG-001から渡されるが、専門エージェント内部の invocation_state には含めない（マルチエージェント連携設計 SYS-MA-001 6.1節）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: 社員（コンソール / CLI）
- **出力内容**: 申請書ドラフト（Excel）ファイルパス・申請書チェック結果・上長承認要否通知

### 11.2 出力メッセージ
- 一括確認質問: `"経費精算申請に必要な情報を確認します。[必須項目一覧]をお知らせください（領収書画像もご提供いただけます）"`
- 経費区分提示（Lv1根拠提示）: `"品目「[品目名]」は「[経費区分]」と判断しました（BRL-12：[判断根拠]）。よろしいでしょうか？"`
- 上長承認要否通知: `"経費合計が5,000円を超えるため、上長の事前承認が必要です（BRL-13）。合計：[金額]円"`
- HitL承認確認: `"申請情報を確認してください。[サマリー]。承認しますか？（OK / 修正 / キャンセル）"`
- 申請書チェック結果（合格）: `"申請書チェック：合格。申請書ドラフトが生成されました。ファイル：[パス]"`
- 申請書チェック結果（不備）: `"申請書チェック：不備あり。[不備箇所・理由・適用ルール名]"`

### 11.3 生成ファイル
- **ファイル名**: `outputs/経費精算申請書_YYYYMMDD_HHMMSS.xlsx`（YYYYMMDD_HHMMSS はツール内部で `datetime.now().strftime("%Y%m%d%H%M%S")` により自律生成する。LLMパラメータとして受け取らない）
- **保存先**: `outputs/`
- **内容**: 収集済み申請情報（D-007）をセルマッピングした Excel 申請書ドラフト（BRL-04準拠）

## 12. バリデーション設計

**システム設計書の方針**: artifacts/03_system-design/outputs/バリデーション方針.md（SYS-VALIDATION-001）を参照してください。

### 12.1 バリデーション方針
- **購入日**: `parse_date`（YYYY-MM-DD / YYYY/MM/DD / YYYY年MM月DD日 → `datetime.date` 変換）
- **店舗名・品目**: `validate_non_empty_string`（非空・500文字以内）
- **経費区分**: `normalize_expense_category`（`Literal["事務用品費", "宿泊費", "資格精算費", "その他経費"]`；判断不能は「その他経費」自動適用：BRL-12）
- **金額**: `parse_amount`（非負整数・カンマ/円除去）
- **業務目的**: `validate_non_empty_string`（非空・500文字以内・BRL-15）
- **HitL承認結果**: `normalize_hitl_result`（`Literal["OK", "修正", "キャンセル"]`）

**バリデーションの詳細**: バリデーション方針（SYS-VALIDATION-001）およびデータモデル基本設計書を参照

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | 対応 |
|----------|----------|------|------|
| BRL-03 | 不足情報の対話収集 | 必須項目が未収集の場合 | 不足項目を特定して追加収集する（CF-003 Step4） |
| BRL-04 | 収集情報のみ使用 | 申請書生成時 | 収集済み申請情報（D-007）の値のみを申請書に記載し、推測・補完しない |
| BRL-05 | 申請書チェックの実施 | 申請書ドラフト生成後 | 申請ルール（DATA-001）に基づいてチェックし結果を提示する（CF-003 Step11） |
| BRL-11 | 領収書画像からの情報自動抽出 | 社員が領収書画像を提供した場合 | LLMのネイティブビジョン機能で購入日・店舗名・品目・金額を抽出する |
| BRL-12 | 品目からの経費区分自動判断 | 品目が判明している場合 | 品目から経費区分を自動判断し、判断不能時は「その他経費」を適用する。結果を社員に提示して確認を促す（COND-002） |
| BRL-13 | 経費上長承認要否 | 経費合計 > 5,000円 | 上長の事前承認が必要な旨を通知する（CF-005） |
| BRL-14 | 申請期限 | 経費発生日から3ヶ月以内 | 申請期限を超過している場合は通知する（申請自体は継続可能） |
| BRL-15 | 業務目的記載必須 | 申請情報収集時 | 業務目的が空の場合は再入力を促す |

## 13. Agent as Tools設計
※AG-003 は AG-001 から Agent As Tools として呼び出される専門エージェント

### 13.1 ツールとしての公開
- **ツール名**: `handle_general_expense_application`
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`
- **出力**: `str`（エージェントからの応答：申請書チェック結果・ファイルパスを含む）

### 13.2 ToolContextの使用
- invocation_stateから `applicant_name`・`application_date` を取得してエージェント初期化に使用する
- `session_id` は ToolContext から取得し、`SessionManager` の初期化に使用する（エージェントへの invocation_state には含めない）
- 申請フロー内の状態（収集済み申請情報・HitL承認結果）は `SessionManager` で DATA-005 に永続化する

**詳細**: マルチエージェント連携設計（SYS-MA-001 6.1節）を参照

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 経費情報が正しく収集されること；領収書画像から情報が正しく抽出されること（BRL-11）；経費区分が正しく自動判断されること（BRL-12）；申請書ドラフトがテキストとして提示されること（ツール呼び出しなし）；HitL承認（BeforeToolCallEvent）が1回のみ発生すること；HitL承認後にのみTOOL-002が呼び出されること；申請書ドラフト（Excel）が生成されること；申請書チェック結果が正しく提示されること
- **異常系テスト**: 領収書画像解析失敗時に手動入力フォールバックが起動すること；HitL承認前にTOOL-002が呼び出されないこと；HitL承認で「修正」選択時に情報収集ステップへ戻ること；申請期限超過時に通知され継続可能なこと；30回対話でTERMINATEDになること；システム系エラー時に呼び出し元エージェントに要約して返すこと
- **性能テスト**: 複数経費件数（最大件数）の情報収集が正常に完了すること

**詳細なテストケース**: 詳細設計書を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- strands-agents v1.25.0 の `@tool(context=True)` デコレータで Agent As Tools として実装する
- SlidingWindowConversationManager のウィンドウサイズは15（共通設定方針 2.2節準拠）
- HumanApprovalHook を登録し TOOL-002 呼び出し前の承認を必須とする
- 領収書画像の解析は strands-agents の LLMネイティブビジョン機能を使用する（外部ツール不要）

### 16.2 業務的制約
- TOOL-002（申請書生成）は HitL承認（OK）を受けた後のみ呼び出し可能（GRD-009）
- 申請書への推測・補完は禁止（BRL-04）
- 申請書の自動送信・実際の提出は禁止（GRD-010）
- TOOL-001（交通費計算ツール）は呼び出し禁止（P-READ-003は付与されていない）

### 16.3 前提条件
- AG-001 から invocation_state（applicant_name, application_date, session_id）が渡されること
- DATA-001（社内申請ルール）・DATA-002（申請書テンプレート）が参照可能な状態であること
- Amazon Bedrock（Claude Sonnet 4.5）が正常に呼び出せること（LLMビジョン機能含む）
