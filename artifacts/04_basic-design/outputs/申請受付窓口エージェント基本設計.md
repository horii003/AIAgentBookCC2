## 2. 目的・役割定義

### 2.1 エージェントの目的
- 社員が入力した申請者名と申請内容テキストから申請種別（交通費精算申請 / 経費精算申請）を判断し、申請書名・申請先・判断根拠を社員に提示する
- 申請種別が確定した後に、該当する専門エージェント（AG-002またはAG-003）へ申請フロー全体を委譲する
- システム全体のエントリーポイントとして機能し、全申請フローはこのエージェントから開始される

### 2.2 主な責務
- **申請者名収集**: 社員から申請者名を受け取り、バリデーション（空入力・500文字超）を適用する
- **申請内容テキスト収集**: 社員から申請内容テキストを受け取り、バリデーション（空入力・500文字超・30回上限）を適用する
- **申請種別判断**: 社内申請ルール（DATA-001）を参照して LLM が申請種別を判断する（JD-01/JD-02）
- **申請種別案内情報提示**: 申請種別・申請書名・申請先・判断根拠を社員に提示する（BRL-02準拠）
- **専門エージェントへの委譲**: 申請種別確定後に、AG-002（交通費）またはAG-003（経費）へ申請フロー全体を委譲する（Agent As Tools方式）

### 2.3 非責務（やらないこと）
- **申請情報の収集**: AG-002/AG-003が担当
- **交通費計算・申請書生成・申請書チェック**: AG-002/AG-003が担当
- **申請期限チェック**: AG-002/AG-003が担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（DATA-001）のファイル参照 | ACT-READ-01：申請ルール参照 |
| 提案 | 申請種別・申請書・申請先・根拠の候補を社員に提示し最終確認を促す | ACT-DEC-01：申請種別判断（JD-01）；CF-001 Step6 |
| 実行 | 該当なし | - |
| 実行（承認必須） | 該当なし | - |

### 3.2 判断権限の範囲
- **自律判断可能**: 申請ルール（DATA-001）に基づく申請種別の判断・申請種別案内情報の生成（JD-01）
- **人間確認必要**: 申請種別判断不能時（JD-02）の選択肢提示 → 社員の選択（CF-004）；申請種別案内情報の最終確認（CF-001 Step7）
- **実行不可**: 申請情報収集・交通費計算・申請書生成・申請期限チェック（AG-002/AG-003の責務）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
開始 → 申請者名受取（Step1-2） → 申請内容テキスト受取（Step3-4） → 申請種別判断（Step5） → 申請種別案内情報提示（Step6） → 社員確認（Step7） → AG-002/AG-003へ委譲 → 終了
```

**フローの詳細**: 詳細設計書を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理。申請者名収集 → 申請内容収集 → 判断 → 提示 → 委譲の順序を厳守する
- **処理をスキップする考え方**: 申請者名・申請内容テキストが空の場合は再入力を促す（スキップなし）
- **同一処理を複数回実行**: 入力バリデーション失敗時に再入力ループを実施（対話回数30回上限：FR-012）；申請種別判断不能時はCF-004へ遷移して選択肢提示ループを実施

### 4.3 状態管理
- **申請者名**: invocation_state（applicant_name）として AG-002/AG-003 に渡す。セッション状態ファイル（DATA-005）にも保存する
- **申請種別案内情報（D-004）**: 委譲リクエスト（LNK-001/LNK-002）に含めて AG-002/AG-003 に引き渡す
- **対話回数**: DATA-005（セッション状態ファイル）で管理する（上限30回：FR-012）

## 5. 専門エージェントとの連携設計
※オーケストレーターエージェントのため記載

### 5.1 連携対象エージェント

| エージェントID | エージェント名 | 連携目的 | 振り分け条件 |
|---------------|---------------|----------|--------------|
| AG-002 | 交通費精算申請エージェント | 交通費精算申請フロー全体の委譲（LNK-001） | 申請種別が「交通費精算申請」に確定した場合（JD-01 または CF-004での社員選択） |
| AG-003 | 経費精算申請エージェント | 経費精算申請フロー全体の委譲（LNK-002） | 申請種別が「経費精算申請」に確定した場合（JD-01 または CF-004での社員選択） |

### 5.2 振り分け判断基準

- **AG-002 へのルーティング**: 申請内容テキストが交通費精算申請（電車代・タクシー代・バス代・飛行機代等）に合致すると LLM が判断した場合（JD-01適用）
- **AG-003 へのルーティング**: 申請内容テキストが経費精算申請（事務用品費・宿泊費・資格精算費等）に合致すると LLM が判断した場合（JD-01適用）
- **CF-004（選択肢提示）**: いずれにも分類できない場合（JD-02適用）。社員の選択後に AG-002 または AG-003 へ委譲する

**詳細な判断基準**: 詳細設計書を参照

### 5.3 invocation_stateの受け渡し

AG-001 は AG-002/AG-003 へ以下の invocation_state を辞書リテラルで渡す。

```python
{
    "applicant_name": str,      # 申請者名
    "application_date": str,    # 申請日（システム日付：YYYY-MM-DD形式）
    "session_id": str           # セッションID（UUID形式）
}
```

**データ構造の詳細**: マルチエージェント連携設計（SYS-MA-001 6.1節）を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 申請受付窓口エージェントとしての役割（申請種別判断・提示・委譲）を定義する
- 社内申請ルール（DATA-001）を参照した申請種別判断の手順を定義する
- 判断不能時（JD-02）の選択肢提示フロー（CF-004）への遷移ルールを定義する

### 7.2 プロンプト構成要素
- **役割定義**: 申請受付窓口エージェントとして申請者名・申請内容を受け取り申請種別を判断・提示・委譲する
- **処理フロー**: CF-001 の会話ステップ（Step1〜7）に準じた処理順序
- **判断プロトコル**: JD-01（申請種別特定）・JD-02（判断不能時）の判断ルール；BRL-01（申請ルール参照）
- **主な責任**: 申請者名・申請内容テキストの受取；申請種別案内情報（D-004）の生成・提示（BRL-02）；委譲先エージェントへの委譲

**プロンプト全文**: 詳細設計書を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 静的定数 |
| 動的生成の理由 | 不要 |
| 動的に埋め込む情報 | なし |

> **判断根拠**: CF-001 の全ステップを確認した結果、AG-001 の処理フロー（申請種別判断・提示・委譲）には日付・期限を基準とした判断処理が含まれていない。申請期限チェック（BRL-14）は AG-002/AG-003 の責務であるため静的定数を選択する。

### 7.4 主要な設計意図
- **LLMによる自然言語意図分類**: 申請内容テキストの申請種別判断は LLM の intent 分類を利用し、ルール参照（DATA-001）と組み合わせたハイブリッド判断を実現する
- **委譲後の非干渉**: AG-001 は委譲後に申請フロー内部には介入しない。申請フロー全体を AG-002/AG-003 に一任する
- **根拠提示（BRL-02）の徹底**: 申請種別判断結果と参照ルール名をセットで社員に提示する

## 14. テスト観点

### 14.1 テスト観点の概要
- **機能テスト**: 申請内容テキストから正しい申請種別が判断されること（JD-01）；判断不能時にCF-004が起動すること（JD-02）；申請種別案内情報（BRL-02）が正しく提示されること

- **性能テスト**: DATA-001 のファイル参照が正常に行われること
