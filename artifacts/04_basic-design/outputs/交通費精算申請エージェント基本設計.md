## 2. 目的・役割定義

### 2.1 エージェントの目的
- AG-001 から委譲された交通費精算申請フロー全体（情報収集・交通費計算・HitL承認・申請書生成・申請書チェック・申請期限チェック）を自己完結で実行する
- 社員との対話を通じて移動区間情報（移動日・出発地・目的地・交通手段・業務目的）を一括確認で収集する
- 収集した情報を申請書テンプレート（DATA-002）にマッピングして Excel 申請書ドラフトを生成し、社員が確認できる状態にする

### 2.2 主な責務
- **申請情報収集**: CF-002 の会話ステップに従って移動区間情報を一括確認・追加収集する（BRL-07: 一区間ずつ収集、BRL-03: 不足情報の追加収集）
- **交通費自動計算**: TOOL-001（交通費計算ツール）で各移動区間の交通費を計算する（BRL-08: 自動計算、BRL-10: 駅名正規化）
- **上長承認要否判断**: 交通費合計が10,000円を超える場合に上長承認が必要な旨を通知する（BRL-09・JD-08）
- **申請期限チェック**: 経費発生日から3ヶ月以内かどうかをチェックし、超過時は社員に通知する（BRL-14・JD-07）
- **HitL承認**: Excel生成前に申請情報サマリーを提示し社員の承認（OK/修正/キャンセル）を確認する（GRD-009）
- **申請書ドラフト生成**: HitL承認（OK）後に TOOL-002（申請書生成ツール）で Excel 申請書ドラフトを生成する（BRL-04: 収集情報のみ使用）
- **申請書チェック**: 申請ルール（DATA-001）に基づいて申請書の不備を確認し結果を社員に提示する（BRL-05・JD-06）

### 2.3 非責務（やらないこと）
- **申請種別判断**: AG-001が担当
- **申請書の実際の提出**: 社員がシステム外で実施（自律度Lv0：実行不可）
- **HitL承認制御（TOOL-002呼び出し制御）**: HumanApprovalHookがハンドラー設計で担当

## 3. 自律度・判断権限

### 3.1 自律度レベル

| レベル | 内容 | 適用範囲 |
|------|------|----------|
| 参照 | 社内申請ルール（DATA-001）・申請書テンプレート（DATA-002）・運賃データ（DATA-003/004）のファイル参照 | ACT-READ-01/02/04 |
| 提案 | 不足情報の特定・上長承認要否の判断・申請書チェック結果の提示 | ACT-DEC-02/05（COND-003）；ACT-DEC-04（COND-005） |
| 生成 | 申請書ドラフト（Excel）生成・確認質問生成 | ACT-GEN-01（HitL承認後）；ACT-GEN-02 |
| 実行（承認必須） | TOOL-002（申請書生成）の実行はHitL承認（OK）後のみ可能 | P-EXEC-001；APR-001（金額閾値超過時） |

### 3.2 判断権限の範囲
- **自律判断可能**: 移動区間情報の充足判断（JD-03/JD-04）；駅名正規化（BRL-10）；交通費計算（BRL-08：TOOL-001呼び出し）；申請期限チェック（BRL-14/JD-07）；申請ルール準拠チェック（BRL-05/JD-06）
- **人間確認必要**: HitL承認（OK/修正/キャンセル）の選択；上長承認要否通知の受領（COND-003：金額10,000円超）；申請書チェック不備時の修正
- **実行不可**: 申請書の自動送信・実際の提出操作（GRD-010）；HitL承認なしの TOOL-002 呼び出し（GRD-009）

## 4. 処理フロー設計

### 4.1 基本フロー（概要）

```text
委譲受付（invocation_state受取）
→ 必須項目一括確認（Step1）
→ 申請情報回答収集（Step2）
→ 充足確認・不足情報追加収集（Step3）
→ 交通費自動計算（TOOL-001）（Step4）
→ 上長承認要否通知（金額閾値超過時）（Step5）
→ 申請期限チェック（Step6）
→ HitL承認確認（Step7-8）
→ 申請書ドラフト生成（TOOL-002）（Step9）
→ 申請書チェック結果提示（Step10）
→ 終了
```

**フローの詳細**: 詳細設計書を参照

### 4.2 フロー制御の考え方
- **原則的な処理順序**: 逐次処理。情報収集完了 → 交通費計算 → 上長承認通知 → 期限チェック → 申請書ドラフト提示（テキスト整理） → HitL承認（BeforeToolCallEvent） → 申請書生成ツール実行 → チェック の順序を厳守する
- **申請書ドラフト提示とツール実行の分離**: 収集済み申請情報のテキスト整理・提示（ドラフト提示ステップ）とTOOL-002呼び出し（ツール実行ステップ）を明確に分離する。HitL承認（HumanApprovalHook）は BeforeToolCallEvent でトリガーし、社員の承認（OK）後にはじめて TOOL-002 を実行する。承認前に TOOL-002 を実行してはならない
- **処理をスキップする考え方**: 上長承認要否通知は金額が10,000円以下の場合スキップ（JD-08）；HitL承認で「キャンセル」を選択した場合はフロー終了
- **同一処理を複数回実行**: 不足情報がある場合に追加収集ループ（Step2-3を繰り返し）；HitL承認で「修正」を選択した場合に情報収集ループへ戻る（Step2へ）；対話回数30回上限あり（FR-012）

### 4.3 状態管理
- **収集済み申請情報（D-007）**: `CollectedTransportInfo` モデル（移動区間リスト・合計金額）としてセッション状態ファイル（DATA-005）に保存する
- **HitL承認結果**: DATA-005 の `hitl_result` フィールドに保存し、TOOL-002 呼び出し前の前提条件として参照する
- **対話回数**: DATA-005 の `iteration` フィールドで管理する（上限30回：FR-012）

## 6. ツール設計

### 6.1 利用ツール

各エージェントが使用するツールは業務機能（計算・生成など）に特化したツールのみとする。

- **TOOL-001（交通費計算ツール）**: CF-002 Step4（交通費自動計算ステップ）で各移動区間の交通費を計算するために呼び出す（1区間1呼び出し・最大1回再試行）
- **TOOL-002（申請書生成ツール：交通費用）**: CF-002 Step9（HitL承認OK後）で交通費精算申請書ドラフト（Excel）を生成するために呼び出す（再試行なし・HitL承認後のみ呼び出し可能）

全必須項目収集完了後に TOOL-001 を区間ごとに呼び出し、交通費計算が完了してから上長承認要否チェック・期限チェック・HitL承認を経て TOOL-002 を呼び出す。

**ツールの詳細**: 交通費計算ツール基本設計書および申請書生成ツール基本設計書を参照

## 7. プロンプト設計

### 7.1 システムプロンプトの目的
- 交通費精算申請フロー（CF-002）全体の処理順序と判断ルールを定義する
- 移動区間情報の一括確認・追加収集の方法を定義する
- 申請期限チェック（BRL-14）を申請日から遡って3ヶ月以内かどうかで判断するルールを定義する

### 7.2 プロンプト構成要素
- **役割定義**: 交通費精算申請エージェントとして交通費精算申請フロー全体を担当する
- **処理フロー**: CF-002 の会話ステップ（Step1〜11）に準じた処理順序；一括確認方式（BRL-07）
- **判断プロトコル**: JD-03/JD-04（充足判断）；JD-07（申請期限超過）；JD-08（上長承認要否）；BRL-05/JD-06（申請書チェック）
- **主な責任**: 移動区間情報の収集（BRL-07）；交通費計算（BRL-08/TOOL-001）；HitL承認後の申請書生成（BRL-04/TOOL-002）；申請書チェック（BRL-05）
- **動的情報（該当する場合）**: 申請日（application_date）・申請期限の基準日（申請日から遡って3ヶ月前：BRL-14）を実行時に埋め込む
- **エラー発生時の振る舞い**: システム系エラー（ツールエラー・システム障害等）が発生した場合は、呼び出し元エージェント（AG-001）に要約して返す

**プロンプト全文**: 詳細設計書を参照

### 7.3 システムプロンプトの生成方式

| 項目 | 内容 |
|------|------|
| 生成方式 | 動的生成関数 |
| 動的生成の理由 | CF-002 Step6 の申請期限チェック（BRL-14）で「経費発生日から3ヶ月以内」の判断をLLMが行うため、実行時の申請日（application_date）と3ヶ月前の基準日をプロンプトに埋め込む必要がある |
| 動的に埋め込む情報 | 申請日（application_date）・申請期限基準日（申請日 - 90日） |

### 7.4 主要な設計意図
- **一括確認方式（BRL-07）**: 区間ごとに1問ずつ質問するのではなく、全必須項目を1メッセージで確認することで対話回数を削減する
- **ドラフト提示とツール実行の分離**: 収集済み申請情報のテキスト整理・提示（ドラフト提示ステップ）と TOOL-002 呼び出し（ツール実行ステップ）を明確に分離する。HumanApprovalHook は BeforeToolCallEvent でトリガーし、承認（OK）後にはじめて TOOL-002 を実行する。HitL承認は1回のみ発生させる（GRD-009準拠）
- **HitL承認前のツール呼び出し禁止**: GRD-009 に従い、TOOL-002 の呼び出しはHitL承認（OK）後のみ許可する。これを HumanApprovalHook と組み合わせて二重に保証する
- **申請期限通知後の継続**: BRL-14 に従い、申請期限超過を通知するが申請フロー自体は継続可能とする（中断は社員の意思による）

## 10. 入力データ設計

### 10.1 初期化時の入力
該当なし（invocation_state で申請者名・申請日を受け取る）

### 10.2 対話ループでの入力
- **申請情報（各区間）**: 移動日・出発地・目的地・交通手段・業務目的（自然文）
- **HitL承認結果**: 「OK」「修正」「キャンセル」のいずれか
- **特殊コマンド**:
  - 対話回数30回到達時: 処理終了通知（FR-012）
  - HitL承認で「キャンセル」選択時: 申請取り消し通知・フロー終了
  - `reset` / `リセット` / `最初から`: 会話履歴と申請者情報をリセットして最初の状態に戻る

### 10.3 invocation_stateの入力
AG-001 から渡される情報:
```python
{
    "applicant_name": str,      # 申請者名
    "application_date": str     # 申請日（システム日付：YYYY-MM-DD形式）
}
```

> ※ session_id はAG-001から渡されるが、専門エージェント内部の invocation_state には含めない（マルチエージェント連携設計 SYS-MA-001 6.1節）

## 11. 出力データ設計

### 11.1 出力形式
- **出力先**: 社員（コンソール / CLI）
- **出力内容**: 申請書ドラフト（Excel）ファイルパス・申請書チェック結果・上長承認要否通知

### 11.2 出力メッセージ
- 一括確認質問: `"交通費精算申請に必要な情報を確認します。[必須項目一覧]をお知らせください"`
- 上長承認要否通知: `"交通費合計が10,000円を超えるため、上長の事前承認が必要です（BRL-09）。合計：[金額]円"`
- HitL承認確認: `"申請情報を確認してください。[サマリー]。承認しますか？（OK / 修正 / キャンセル）"`
- 申請書チェック結果（合格）: `"申請書チェック：合格。申請書ドラフトが生成されました。ファイル：[パス]"`
- 申請書チェック結果（不備）: `"申請書チェック：不備あり。[不備箇所・理由・適用ルール名]"`

### 11.3 生成ファイル
- **ファイル名**: `outputs/交通費精算申請書_YYYYMMDD_HHMMSS.xlsx`（YYYYMMDD_HHMMSS はツール内部で `datetime.now().strftime("%Y%m%d%H%M%S")` により自律生成する。LLMパラメータとして受け取らない）
- **保存先**: `outputs/`
- **内容**: 収集済み申請情報（D-007）をセルマッピングした Excel 申請書ドラフト（BRL-04準拠）

## 12. バリデーション設計

**システム設計書の方針**: artifacts/03_system-design/outputs/バリデーション方針.md（SYS-VALIDATION-001）を参照してください。

### 12.1 バリデーション方針
- **移動日**: `parse_date`（YYYY-MM-DD / YYYY/MM/DD / YYYY年MM月DD日 → `datetime.date` 変換）
- **出発地・目的地**: `normalize_station_name`（末尾「駅」除去・BRL-10準拠）；非空チェック
- **交通手段**: `normalize_transport_type`（`Literal["電車", "バス", "タクシー", "飛行機"]`）
- **費用（手動入力時）**: `parse_amount`（非負整数・カンマ/円除去）
- **業務目的**: `validate_non_empty_string`（非空・500文字以内・BRL-15）
- **HitL承認結果**: `normalize_hitl_result`（`Literal["OK", "修正", "キャンセル"]`）

**バリデーションの詳細**: バリデーション方針（SYS-VALIDATION-001）およびデータモデル基本設計書を参照

### 12.2 申請ルール一覧

| ルールID | ルール名 | 条件 | 対応 |
|----------|----------|------|------|
| BRL-03 | 不足情報の対話収集 | 必須項目が未収集の場合 | 不足項目を特定して追加収集する（CF-002 Step3） |
| BRL-04 | 収集情報のみ使用 | 申請書生成時 | 収集済み申請情報（D-007）の値のみを申請書に記載し、推測・補完しない |
| BRL-05 | 申請書チェックの実施 | 申請書ドラフト生成後 | 申請ルール（DATA-001）に基づいてチェックし結果を提示する（CF-002 Step10） |
| BRL-07 | 一区間ずつの必須項目収集 | 申請情報収集時 | 移動区間ごとに必須項目（移動日・出発地・目的地・交通手段・費用・業務目的）を収集する |
| BRL-08 | 運賃の自動計算 | 移動区間情報収集完了後 | TOOL-001を呼び出して電車は経路テーブル検索、バス/タクシー/飛行機は固定運賃で計算する |
| BRL-09 | 交通費上長承認要否 | 交通費合計 > 10,000円 | 上長の事前承認が必要な旨を通知する（CF-005） |
| BRL-10 | 駅名正規化 | 出発地・目的地入力時 | 末尾「駅」を除去し、経路テーブル（DATA-003）の検索キーと一致する形式に正規化する |
| BRL-14 | 申請期限 | 経費発生日から3ヶ月以内 | 申請期限を超過している場合は通知する（申請自体は継続可能） |
| BRL-15 | 業務目的記載必須 | 申請情報収集時 | 業務目的が空の場合は再入力を促す |

## 13. Agent as Tools設計
※AG-002 は AG-001 から Agent As Tools として呼び出される専門エージェント

### 13.1 ツールとしての公開
- **ツール名**: `handle_transport_expense_application`
- **デコレータ**: `@tool(context=True)`
- **入力**: `query (str)`, `tool_context (ToolContext)`
- **出力**: `str`（エージェントからの応答：申請書チェック結果・ファイルパスを含む）

### 13.2 ToolContextの使用
- invocation_stateから `applicant_name`・`application_date` を取得してエージェント初期化に使用する
- `session_id` は ToolContext から取得し、`SessionManager` の初期化に使用する（エージェントへの invocation_state には含めない）
- 申請フロー内の状態（収集済み申請情報・HitL承認結果）は `SessionManager` で DATA-005 に永続化する

**詳細**: マルチエージェント連携設計（SYS-MA-001 6.1節）を参照

## 16. 制約事項・前提条件

### 16.1 技術的制約
- strands-agents v1.25.0 の `@tool(context=True)` デコレータで Agent As Tools として実装する
- SlidingWindowConversationManager のウィンドウサイズは20（共通設定方針 2.2節準拠）
- HumanApprovalHook を登録し TOOL-002 呼び出し前の承認を必須とする

### 16.2 業務的制約
- TOOL-002（申請書生成）は HitL承認（OK）を受けた後のみ呼び出し可能（GRD-009）
- 申請書への推測・補完は禁止（BRL-04）
- 申請書の自動送信・実際の提出は禁止（GRD-010）

### 16.3 前提条件
- AG-001 から invocation_state（applicant_name, application_date, session_id）が渡されること
- DATA-001（社内申請ルール）・DATA-002（申請書テンプレート）・DATA-003/004（運賃データ）が参照可能な状態であること
- Amazon Bedrock（Claude Sonnet 4.5）が正常に呼び出せること
