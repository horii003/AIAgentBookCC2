## 2. 目的・役割定義

### 2.1 ツールの目的
- 社員が申告した移動経路情報（出発地・目的地・交通手段）をもとに、運賃データファイルを参照して正確な交通費を計算する
- 電車経路の場合は経路テーブル（DATA-003: `data/fares/train_fares.json`）を検索して運賃を取得し、バス・タクシー・飛行機の場合は固定運賃データ（DATA-004: `data/fares/fixed_fares.json`）を参照する
- エージェントが運賃データを直接計算することなく、確実に正しい運賃を取得できるようにする

### 2.2 主な責務
- **電車経路テーブル検索**: DATA-003 の JSON ファイルから出発地・目的地をキーに運賃を検索する
- **固定運賃取得**: DATA-004 の JSON ファイルから交通手段（バス・タクシー・飛行機）をキーに固定金額を取得する
- **計算根拠の返却**: 使用した経路テーブルまたは固定運賃の情報を出力に含め、エージェントが根拠を社員に提示できるようにする

### 2.3 非責務（やらないこと）
- **交通手段バリデーション**: Pydanticモデルによる入力バリデーション（バリデーション方針 SYS-VALIDATION-001）
- **駅名正規化**: 「〇〇駅」→「〇〇」の正規化処理はハンドラー設計で対応（BRL-10）
- **上長承認要否判定**: 交通費合計10,000円超の閾値チェックはハンドラー設計で対応（BRL-09）
- **合計金額計算**: 複数区間の合計金額の集計はエージェント側で実施

### 2.4 エージェント単独では実現できない理由
- 電車の経路テーブル（DATA-003）は外部 JSON ファイルとして管理されており、エージェントのシステムプロンプトに埋め込むことが現実的でない（数百〜数千件の経路データ）
- 固定運賃データ（DATA-004）もファイル管理されており、ファイル読み込みとキー検索のロジックが必要
- 運賃データの改定（DATA-003/004 の更新）に対して、ツールを介することでシステムプロンプトの変更なしに対応できる

### 2.5 主要な設計意図
- **ファイル直接参照**: RAG を使用せず JSON ファイルを直接読み込み・検索することで、機微情報の外部送信リスクを排除する（外部API不使用方針準拠）
- **起動時ロード**: DATA-003/004 はアプリケーション起動時またはツール初回呼び出し時にメモリにロードし、都度ファイル I/O を行わないことでパフォーマンスを確保する（バリデーション方針 SYS-VALIDATION-001 9b節準拠）
- **1区間1呼び出し**: 1ツール呼び出しで1移動区間の運賃を計算する設計とし、複数区間の場合はエージェントがループで呼び出す

## 5. 入力設計（エージェント視点）

### 5.1 入力の意味と取得方法
- **移動区間情報**: CF-002 Step2〜3 の対話収集で社員から取得した1区間分の移動情報
- **取得元**: AG-002 が収集した社員回答（D-006）から各フィールドを抽出する
- **不足時の対応**: CF-002 Step3 で充足確認（JD-04）を行い、不足がある場合は追加収集してから呼び出す

### 5.2 入力項目一覧

| 入力ID | 内容 | 必須 | 取得元 | 不足時の対応 |
|-------|-----|------|-------|------------|
| IN-001 | 出発地（駅名またはバス停名等） | ○ | 社員回答（D-006）・BRL-10正規化済み | CF-002 Step3 で再質問 |
| IN-002 | 目的地（駅名またはバス停名等） | ○ | 社員回答（D-006）・BRL-10正規化済み | CF-002 Step3 で再質問 |

## 6. 出力設計（エージェント視点）

### 6.1 出力の意味と利用方法
- **交通費金額**: 計算された運賃（円）。エージェントが収集済み申請情報（D-007）の当該区間の費用フィールドに格納する
- **計算根拠**: エージェントが BRL-02（根拠提示）に従い、使用した経路テーブルまたは固定運賃の情報を社員に提示するために使用する

### 6.2 出力項目一覧

| 出力ID | 内容 | 型 | 利用目的 |
|-------|-----|---|---------|
| OUT-001 | 交通費金額（円） | 非負整数 | 収集済み申請情報（D-007）の当該区間費用フィールドに格納 |
| OUT-002 | 計算根拠（使用データの説明） | 文字列 | 社員への根拠提示（BRL-02準拠） |

### 6.3 出力の信頼性
- DATA-003（電車経路・運賃データ）または DATA-004（固定運賃データ）から直接取得した値であるため、データファイルの正確性に依存する

## 8. フォールバック・代替設計

### 8.1 ツール未使用時の代替手段
- 交通費計算ツールが失敗した場合、社員が費用を手動入力する（AG-002 がフォールバックとして手動入力を促す）
- 手動入力値は Pydantic モデルによるバリデーション（非負整数）を通過した場合に収集済み申請情報に格納する

### 8.2 再試行条件
- TOOL-001 のファイル参照失敗は一時的な I/O エラーの可能性があるため、1回のみ再試行する（実行制御方針 SYS-EXEC-CTRL-001 7.1節準拠）
- 再試行は即時（クールダウン: 要件上未定義）
- 2回失敗した場合は即座に手動入力代替処理へ遷移する（無限リトライ禁止原則準拠）

### 8.3 人間への引き渡し条件
- 電車経路テーブル（DATA-003）に対象経路が存在しない場合：社員に手動入力を促す
- 2回連続でファイルI/Oエラーが発生した場合：EX-06（データ不整合）として社員に手動入力を促す
- EX-07（システム障害）が発生した場合：社員に担当部門への問い合わせを案内してセッション終了

## 11. エラー・例外（ツール起因）

**システム設計書の方針**: artifacts/03_system-design/outputs/例外処理方針.md（SYS-EXCEPTION-001）を参照してください。

### 11.1 エラー対応の基本方針
- **EX-06（データ不整合）**: 電車経路テーブル検索失敗は1回再試行後に手動入力代替へ。固定運賃取得失敗も同様
- **EX-07（システム障害）**: ファイルロード失敗はエスカレーション案内・TERMINATED
- **EX-08（想定外）**: 上記に分類できない例外はエスカレーション案内・TERMINATED

### 11.2 エラー一覧

| エラーID | 想定原因 | エージェントの対応 |
|---------|---------|------------------|
| E-001 | 電車経路テーブル（DATA-003）に対象経路が存在しない | EX-06：社員に通知し費用の手動入力を促す（1回再試行後） |
| E-002 | 電車経路・運賃データ（DATA-003）のファイルI/Oエラー | EX-06：1回再試行後に手動入力代替へ |
| E-003 | 固定運賃データ（DATA-004）のファイルI/Oエラー | EX-06：1回再試行後に手動入力代替へ |
| E-004 | 入力バリデーションエラー（Pydanticモデル検証失敗） | EX-01：社員に日本語メッセージで再入力を促す |

## 13. 補助機能

### 13.1 load_fare_data（運賃データロード）
- **目的**: アプリケーション起動時またはツール初回呼び出し時に DATA-003/DATA-004 を JSON ファイルから読み込みメモリに保持する
- **エージェントへの影響**: ツール初回呼び出しの遅延を最小化する

### 13.2 normalize_station_name（駅名正規化補助）
- **目的**: 入力された駅名から末尾の「駅」を除去するなど、DATA-003 のキーと一致する形式に正規化する（BRL-10準拠）
- **エージェントへの影響**: 「渋谷駅→渋谷」等の表記揺れを吸収し、経路テーブル検索の成功率を高める

### 13.3 実装詳細

## 14. 詳細設計への引き渡し事項

### 14.0 ツール分割方針（必須）

| ツール関数名 | 役割 | 呼び出しエージェント |
|------------|------|-------------------|
| calculate_transport_fare | 1移動区間の交通費運賃を計算する（電車：DATA-003検索、バス/タクシー/飛行機：DATA-004参照） | AG-002 |

呼び出しエージェントが AG-002 のみのため、単一ツール構成とする。

### 14.1 実装時の識別子
- **ツール名**: `calculate_transport_fare`
- **デコレータ**: `@tool`
- **実装ファイル**: `tools/transport_calculator.py`

### 14.2 入出力マッピング注意点
- `departure`・`destination` は Pydantic `field_validator` による駅名正規化（`normalize_station_name`）を `mode="before"` で適用する
- `transport_type` は `Literal["電車", "バス", "タクシー", "飛行機"]` とし、別表記マッピングは Pydantic `normalize_transport_type` バリデータで処理する（バリデーション方針 7.1節準拠）
- `travel_date` は `datetime.date` 型として受け取る（日付パースは Pydantic `parse_date` バリデータで処理）

### 14.3 SDK実装時の注意事項
- DATA-003/DATA-004 はアプリケーション起動時またはツール初回呼び出し時にロードし、モジュールレベルの変数または `functools.lru_cache` でキャッシュする（都度ファイル I/O 禁止）
- ロード時に Pydantic モデルでデータ形式の整合性を検証する（バリデーション方針 9b節準拠）
- strands-agents v1.25.0 の `@tool` デコレータを使用する

### 14.4 モック／スタブ要否
- テスト時は DATA-003/DATA-004 の代わりにテスト用の小規模 JSON データ（フィクスチャファイル）を使用する（ファイルI/Oのモック化）

## 16. 制約事項・前提条件

### 16.1 技術的制約
- DATA-003（電車経路・運賃データ）に登録されていない経路は計算不可（手動入力フォールバック）
- ファイル I/O はアプリケーション起動時またはツール初回呼び出し時のみ実施し、都度読み込みは行わない
- 1ツール呼び出しで処理できるのは1移動区間のみ（複数区間はエージェントがループで呼び出す）

### 16.2 業務的制約
- 電車の対象経路は DATA-003 に登録されているもののみ（BRL-08準拠）
- バス：230円・タクシー：10,000円・飛行機：50,000円の固定運賃を適用する（DATA-004準拠）
- 運賃データの更新（DATA-003/004 の差し替え）は管理者が手動で行う

### 16.3 前提条件
- DATA-003（`data/fares/train_fares.json`）および DATA-004（`data/fares/fixed_fares.json`）がファイルとして存在し、参照可能な状態であること
- 呼び出しエージェントが AG-002 であること（GRD-009：ツール呼び出し権限制御）
- 入力の `departure`・`destination` は BRL-10 に従い正規化済みであること
