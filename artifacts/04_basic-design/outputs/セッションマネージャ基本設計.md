## 2. 目的・役割定義

### 2.1 コンポーネントの目的
- AG-001/AG-002/AG-003 が申請フロー全体を通じて一貫した状態を参照・更新できるよう、セッション状態（DATA-005）を永続化・管理するコンポーネント
- セッション再開（resume）・中断・終了のライフサイクルを管理し、社員が申請フローを途中から再開できる基盤を提供する
- セッション全体の対話回数（最大30回：FR-012）を管理し、上限到達時の TERMINATED 遷移を制御する

### 2.2 主な責務
- **セッション生成**: AG-001 からの呼び出しで新規セッションを生成し、session_id（UUID形式）を発行する。セッション状態ファイル（DATA-005）を `data/sessions/` に作成する
- **セッション状態の読み書き**: 全エージェント（AG-001/AG-002/AG-003）からセッション状態（ステータス・対話回数・業務コンテキスト・HitL承認結果等）の読み込みと書き込みのAPIを提供する
- **対話回数管理**: 対話ターンごとに `iteration` カウンタをインクリメントし、30回到達時に TERMINATED 遷移を実施する
- **セッションライフサイクル管理**: CREATED → ACTIVE → WAITING → CLOSED/TERMINATED の状態遷移を制御する（セッション管理方針 SYS-SESSION-001 3節準拠）
- **セッション再開（resume）**: セッション状態ファイルが存在し、状態が ACTIVE または WAITING の場合に前回の中断箇所から再開できる情報を復元する

### 2.3 非責務（やらないこと）
- **会話履歴の管理**: `SlidingWindowConversationManager`（strands-agents提供）が担当
- **業務ロジックの実行**: 各エージェント（AG-001/AG-002/AG-003）が担当
- **ファイルI/Oエラーの例外処理**: `ErrorHandler`（HD-001）が担当（EX-07として処理）
- **ReActループのカウント管理**: `LoopControlHook`（HD-003）が担当（1ターン内のループ制御）

### 2.4 必要性
- セッション状態を永続化することで、申請フロー途中での中断・再開が可能になる（セッション管理方針 10節準拠）
- AG-001 → AG-002/AG-003 のマルチエージェント連携で共有の session_id を持ち、エージェント間の状態整合性を保証する（マルチエージェント連携設計 SYS-MA-001 6.1節準拠）
- 対話回数（最大30回：FR-012）の一元管理により、上限超過時の安全な TERMINATED 遷移を保証する

### 2.5 主要な設計意図
- **ファイル永続化の採用**: インメモリではなく `data/sessions/` へのファイル永続化（JSON形式）を採用することで、プロセス再起動後のセッション再開を可能にする（セッション管理方針 9.1節準拠）
- **セッション状態の唯一性**: 1セッション = 1ファイル（DATA-005）の設計とし、全エージェントが同一ファイルを読み書きすることで状態の整合性を保つ
- **アクセス制御の明確化**: 各エージェントが書き込める情報の範囲をセッション管理方針 6.2節に従い制限する（申請者名・申請種別案内情報は AG-001 確定後は上書き禁止）

## 3. SessionManagerクラス

### 3.1 クラス概要
- **クラス名**: `SessionManager`
- **モジュール**: `session/session_manager.py`
- **説明**: セッション状態ファイル（DATA-005）の読み書きと、セッションライフサイクル管理を担当するクラス。AG-001/AG-002/AG-003 の各エージェントインスタンス生成時に `session_id` と `storage_path` を受け取ってインスタンス化される

### 3.2 クラス変数
- **`session_id`**: UUID形式のセッション識別子（形式: `sess_xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`）（初期値: コンストラクタ引数から設定）
- **`storage_path`**: セッション状態ファイルの保存ディレクトリパス（初期値: `"data/sessions/"`）
- **`session_file_path`**: セッション状態ファイルの絶対パス（`{storage_path}/{session_id}.json`）（初期値: コンストラクタで自動設定）

**詳細**: 詳細設計書を参照

## 4. 主要メソッド

### 4.1 `create_session`
- **目的**: 新規セッションを生成し、セッション状態ファイル（DATA-005）を作成する
- **引数**: `applicant_name: str`（申請者名）、`application_date: str`（申請日 YYYY-MM-DD形式）
- **戻り値**: `str`（発行した session_id）
- **フォーマット**: DATA-005 を JSON 形式で `data/sessions/{session_id}.json` に作成する（初期ステータス: CREATED）

### 4.2 `load_session`
- **目的**: 既存のセッション状態ファイルを読み込み、SessionState オブジェクトを返す
- **引数**: なし（インスタンスの `session_file_path` を使用）
- **戻り値**: `SessionState`（データモデル基本設計書 DM-001）または `None`（ファイルが存在しない場合）
- **デフォルトパス**: `{storage_path}/{session_id}.json`

### 4.3 `update_session`
- **目的**: セッション状態の特定フィールドを更新し、セッション状態ファイルに書き込む
- **引数**: `field: str`（更新フィールド名）、`value: Any`（更新値）、`agent_id: str`（更新元エージェントID。書き込み権限の検証に使用）
- **戻り値**: `None`

### 4.4 `transition_status`
- **目的**: セッション状態を指定の遷移先ステータスに変更する（遷移可能な状態遷移かどうかを検証する）
- **引数**: `new_status: str`（`Literal["CREATED", "ACTIVE", "WAITING", "CLOSED", "TERMINATED"]`）
- **戻り値**: `None`
- **フォーマット**: 遷移不可能なステータス変更（例: CLOSED → ACTIVE）はエラーとして `ErrorHandler` に委譲する

### 4.5 `increment_iteration`
- **目的**: 対話回数（`iteration`）をインクリメントし、30回到達時に TERMINATED 遷移を実施する
- **引数**: なし
- **戻り値**: `bool`（`True`: 継続可能、`False`: 上限到達で TERMINATED に遷移済み）

### 4.6 `resume_session`
- **目的**: セッション状態ファイルからセッション情報を復元し、再開可否を判定する
- **引数**: なし
- **戻り値**: `SessionState`（復元成功）または `None`（再開不可: CLOSED/TERMINATED またはファイル破損）
- **フォーマット**: セッション状態が ACTIVE または WAITING の場合のみ復元成功を返す

**詳細**: 詳細設計書を参照

## 5. セッション管理の仕組み

**システム設計書の方針**: artifacts/03_system-design/outputs/セッション管理方針.md（SYS-SESSION-001）を参照してください。

### 5.1 セッション生成フロー
1. AG-001 が `SessionManager.create_session(applicant_name, application_date)` を呼び出す
2. `SessionManager` が `sess_` + UUID を生成し `session_id` を確定する
3. `data/sessions/` ディレクトリが存在しない場合は自動作成する
4. セッション状態ファイル（`data/sessions/{session_id}.json`）を初期状態（CREATED）で作成する
5. `session_id` を AG-001 に返却し、AG-001 は invocation_state に含めて AG-002/AG-003 に渡す

### 5.2 セッション状態更新フロー
1. 各エージェント（AG-001/AG-002/AG-003）が `SessionManager.update_session(field, value, agent_id)` を呼び出す
2. `SessionManager` がエージェントIDに基づく書き込み権限を検証する（セッション管理方針 6.2節準拠）
3. 検証通過後、セッション状態ファイルを読み込んで指定フィールドを更新し、書き込む

### 5.3 対話回数管理フロー
1. 各エージェントが社員から入力を受け取るターンで `SessionManager.increment_iteration()` を呼び出す
2. `iteration` カウンタをインクリメントし、ファイルに保存する
3. `iteration` が30（FR-012上限）に達した場合: `transition_status("TERMINATED")` を実行し、`False` を返す
4. 呼び出し元エージェントは `False` を受け取ったらフローを終了し、ErrorHandler（EX-01）を通じて社員に通知する

### 5.4 セッション再開フロー
1. セッション開始時に `SessionManager.resume_session()` を呼び出す
2. `data/sessions/{session_id}.json` の存在を確認する（存在しない場合は `None` を返す）
3. ファイルが存在する場合: `SessionState` に読み込み、状態が ACTIVE または WAITING かどうかを確認する
4. ACTIVE または WAITING の場合: `SessionState` オブジェクトを返し、前回の中断箇所から再開する
5. CLOSED または TERMINATED の場合: 再開不可として `None` を返し、新規セッションを開始する

**詳細**: 詳細設計書を参照

## 6. マルチエージェント連携

### 6.1 セッション共有の仕組み

**AG-001（オーケストレーター）**:
1. 申請フロー開始時に `create_session` を呼び出して新規セッションを生成する
2. 発行した `session_id` を invocation_state（`{applicant_name, application_date, session_id}`）に含めて AG-002/AG-003 へ渡す
3. AG-001 は `applicant_name`・`application_type_info` をセッション状態ファイルに書き込む
4. AG-002/AG-003 への委譲後は、セッション状態の読み書きを AG-002/AG-003 に委ねる

**AG-002/AG-003（専門エージェント）**:
1. ToolContext から `session_id` を取得し（invocation_state には含めない：マルチエージェント連携設計 SYS-MA-001 6.1節）、`SessionManager(session_id, storage_path)` でインスタンス化する
2. `load_session` で既存のセッション状態を読み込み、申請者名・申請日等の業務コンテキストを復元する
3. 収集済み申請情報・HitL承認結果・申請書ドラフトパスを `update_session` でセッション状態ファイルに書き込む
4. AG-001 が確定した申請者名・申請種別案内情報（AG-001 の書き込み領域）を上書きしない

### 6.2 会話履歴の共有範囲
- **共有される情報**: セッション状態ファイル（DATA-005）の業務コンテキスト全体（申請者名・申請種別案内情報・収集済み申請情報・HitL承認結果・対話回数・ステータス）
- **共有されない情報**: 各エージェントの `SlidingWindowConversationManager` が保持する直近N件の会話履歴（エージェントインスタンスごとに独立管理）

## 10. テスト観点

| 観点 | テスト内容 | 期待結果 |
|-----|----------|---------|
| 正常系 | create_session 呼び出しでセッション状態ファイルが作成されること | `data/sessions/{session_id}.json` が初期状態（CREATED）で作成される |
| 正常系 | load_session でセッション状態ファイルの内容が正しく読み込まれること | `SessionState` オブジェクトが返却される |
| 正常系 | update_session で収集済み申請情報が正しく保存されること | ファイルの該当フィールドが更新される |
| 正常系 | increment_iteration を29回呼び出した時点で True が返ること | iteration=29、ステータスは ACTIVE のまま |
| 正常系 | increment_iteration を30回呼び出した時点で TERMINATED に遷移し False が返ること | iteration=30、ステータスが TERMINATED |

## 11. 詳細設計への引き渡し事項

### 11.1 実装時の識別子
- **クラス名**: `SessionManager`（`session/session_manager.py`）

### 11.2 SDK実装時の注意事項
- セッション状態ファイルの読み書きは原子的に行う（ファイルの部分書き込みによる破損防止のため、書き込みはtemporaryファイルへの書き込み後にリネームする方式が望ましい）
- マルチプロセス環境での同時書き込みは発生しない（AG-002とAG-003は同一セッション内で排他実行：マルチエージェント連携設計 6.3節）ため、ファイルロックは任意実装とする
- `update_session` のエージェントID検証ロジックは、セッション管理方針 6.2節の「誰が更新できるか」ルールに厳密に従って実装する
- `session_file_path` の `data/sessions/` ディレクトリは、`create_session` 時に `os.makedirs(exist_ok=True)` で自動作成する

### 11.3 モック／スタブ要否
- ファイルI/O処理（`create_session`・`load_session`・`update_session`）は、テスト時にtemporaryディレクトリを `storage_path` として注入することでモック不要でテスト可能
- `transition_status` のエラーケース（不正な状態遷移）のテストは実ファイルを使用して検証する

## 13. 制約事項・前提条件

### 13.1 技術的制約
- `data/sessions/` ディレクトリへのファイル書き込み権限が必要
- セッション状態ファイルのフォーマットは JSON 形式に固定する（バイナリ形式・データベースへの変更は詳細設計での確認が必要）
- strands-agents の `SlidingWindowConversationManager` とは独立して動作する（会話履歴管理はフレームワーク任せ、業務コンテキスト管理は `SessionManager` が担当）

### 13.2 業務的制約
- AG-001 が確定した申請者名・申請種別案内情報は AG-002/AG-003 による上書きを禁止する（セッション管理方針 6.2節準拠）
- セッション状態ファイルの保存期間は要件上未定義（セッション管理方針 15節 U-001）
- 同一セッション内での AG-002 と AG-003 の同時実行は禁止されているため、ファイルの同時書き込み問題は発生しない

### 13.3 前提条件
- `data/sessions/` の親ディレクトリ（`data/`）が存在するか、またはアプリケーション起動時に自動作成されること
- `models/data_models.py` の `SessionState` モデルが利用可能であること（データモデル基本設計書 DM-001 参照）
- Amazon Bedrock（Claude Sonnet 4.5）の利用とは独立して動作する（SessionManager 自体は LLM を呼び出さない）
