# Purpose
このファイルは、成果物生成ルールです。
要件定義書、設計書、コード生成物を、artifact-catalog.yaml に定義された順序・依存関係・入出力定義に従って生成します。

# Scope
- 01_business-requirements
- 02_system-requirements
- 03_system-design
- 04_basic-design
- 05_detailed-design
- 06_code-generation

# Source of Truth
- 全体ルール: このファイル
- 成果物定義: `.claude/rules/artifact-catalog.yaml`
- 成果物ごとの生成指示: `.claude/rules/prompts/`
- 成果物ごとのテンプレート: `.claude/rules/templates/`
- 生成成果物: `artifacts/`

## フェーズ開始ルール（最重要）

### フェーズ間の自動進行禁止
- **あるフェーズが完了しても、次のフェーズへ自動的に進んではならない**
- 次フェーズの作業開始は、必ずユーザーからの明示的な指示を待つこと
- フェーズ完了後は以下のメッセージを提示して待機すること

```text
✅ [フェーズ名] が完了しました。
次フェーズ（[次フェーズ名]）を開始する場合は、その旨を指示してください。
```

- ユーザーが「次のフェーズを開始してください」「[フェーズ名]を進めてください」等と明示的に指示した場合のみ、次フェーズの作業を開始する
- フェーズ完了の確認・報告のみを行い、次フェーズの準備・先読み・自動実行は一切禁止する

## 基本方針

### 0. ステアリングファイルの変更禁止（最重要）
- `.claude/rules/` 配下のファイルは、ユーザーが管理するルールファイルである
- ユーザーから明示的に「変更してください」「追記してください」と指示された場合のみ変更してよい
- 作業上の都合（回避策・効率化・補足など）による独自判断での変更・追記は一切禁止する
- 変更が必要と判断した場合は、変更内容を提案してユーザーの承認を得てから実施すること

### 1. 不明点は必ずユーザーに確認すること
- 指示が不明確・曖昧な場合は、推測や補完を行わず、必ずユーザーに確認してから作業を開始すること

### 2. 失敗時はユーザーに報告して停止すること
- 処理が失敗した場合は、即座に作業を停止すること
- 自動リトライや続行は行わず、必ずユーザーの指示を待つこと
- 失敗の内容・原因・現在の状態をユーザーに明確に報告すること

### 3. 出力は日本語・テンプレート準拠で行うこと
- すべての出力は日本語で記述すること
- コードの識別子（変数名・関数名・クラス名など）は英語を使用し、コメントは日本語で記述すること
- 成果物の本文には、テンプレートに定義されていない解説・補足・注釈を混在させないこと

## 作業開始時の必須手順

### 1. フェーズ確認
- 現在のフェーズと前提条件を確認する
- artifact-catalog.yaml の `depends_on` を参照して依存関係を確認する
- どのフェーズから開始すべきか不明な場合は、入力要件の有無と前フェーズ成果物の有無を確認する

### 2. 依存成果物の未定義項目チェック
- 対象成果物の `depends_on` に記載された全依存成果物を読み込む
- 各依存成果物内で以下のパターンを検索する
  - `要件上未定義`
  - `〇〇フェーズで定義`
  - `TBD`
- 発見した場合は、影響度・重要度・関連性を問わず、即座に作業を停止してユーザーに報告する
- `詳細設計に影響しない` などの自己判断で続行してはならない
- ユーザーが明示的に `未定義のまま進めてよい` と指示した場合のみ続行できる

報告形式は次のとおり。

```text
⚠️ 作業開始不可：依存成果物に未定義項目があります

| ファイル | 項目 | 記載内容 |
|---|---|---|
| [ファイル名] | [項目名] | [記載内容] |

上記を定義・解決してから再度依頼してください。
未定義のまま進める場合は、その旨を明示的に指示してください。
```

#### 依存ファイルが存在しない場合の例外ルール（複数周回作業）

フェーズ内を複数周回に分けて作成する場合、以下のルールを適用する。

##### 第2周が必要かどうかの判断（最重要）

第1周完了後に第2周が必要かどうかは、以下の条件を **両方** 満たす場合のみ第2周が必要である。

- 条件1: フェーズ内の成果物が、**同フェーズ内の別の成果物** を `depends_on` に持つ
- 条件2: その依存先成果物が、1周目の作成時点でまだ未作成だった（= `要件上未定義` として記載した箇所が存在する）

上記2条件を満たさない場合（例：全依存先が前フェーズの成果物のみ）は、第2周は不要であり、フェーズ品質チェックへ直接進む。

**第1周：全成果物の初回作成**
- `depends_on` に記載されたファイルが存在しない場合、ユーザー確認なしで作業を続行してよい
- 存在しない依存ファイルから参照すべき情報は `要件上未定義` として記載する
- フェーズ内の全成果物の第1周作成が完了したら、上記「第2周が必要かどうかの判断」を実施する
- 第2周が必要な場合のみ、以下を確認する

```text
✅ 第1周完了：フェーズ内の全成果物を一巡しました。
第2周（未定義項目の補完）に進みますか？
```

- 第2周が不要な場合は、確認メッセージを出さずにフェーズ品質チェックへ進む

**第2周以降：未定義項目の補完**
- ユーザーが「次の周回に進む」と明示的に指示した場合のみ開始する
- 前周で `要件上未定義` とした箇所を、依存成果物の内容をもとに補完する
- 補完後は各成果物間の整合性チェックを実施する
- 未定義項目が残る場合は、さらに次の周回への継続を確認する
- すべての未定義項目が解消された周回の完了後に、フェーズ完了時の品質チェックを実施する

```text
✅ 第N周完了：未定義項目の補完が完了しました。
残存する未定義項目: [件数]件
[件数 > 0 の場合] 第N+1周に進みますか？
[件数 = 0 の場合] フェーズ完了時の品質チェックを実施します。
```

### 3. 作成順序の厳守
- 成果物の作成順序は `artifact-catalog.yaml` の `phases[].default_sequence` に記載された順序を厳守する
- `depends_on` の依存関係を理由に `default_sequence` の順序を変更してはならない
- `depends_on` は「参照すべき入力情報の定義」であり、「作成順序の変更根拠」ではない
- 同フェーズ内の依存ファイルが未作成の場合は、第1周ルールに従い `要件上未定義` として記載し、`default_sequence` の順序で進める

### 4. 必要ファイル読み込み
- 作成する成果物に対応する定義を artifact-catalog.yaml から特定する
- 定義に記載された prompt を読み込む
- 定義に記載された template を読み込む
- フェーズ内の全成果物の prompt / template を一括読み込みしてはならない
- `作成順序.md` は使用しない

## 成果物作成時の禁止事項
- 依存成果物に `要件上未定義` `〇〇フェーズで定義` `TBD` と記載された項目に対して、推測・補完・暫定値の設定を行ってはならない
- 未定義項目は `要件上未定義` のまま引き継ぐ
- `一般的にはこの値` などの独自判断で値を設定してはならない
- 未定義項目を発見した場合は、ユーザーに確認を求める

## 成果物作成時の必須手順
このファイルを読み込んだら、作業開始前にタイトル部だけでよいので、この手順で作業を進めることを回答すること。

### 1. 成果物定義・prompt・template の読み込み
- 作成対象の成果物定義を artifact-catalog.yaml から読み込む
- 該当成果物の prompt を読み込む
- 該当成果物の template を読み込む
- 次の成果物の prompt / template は、その成果物の作成開始時に初めて読み込む
- フェーズ開始時に全成果物の prompt / template を一括読み込みしてはならない

### 2. 完全準拠
- template の全セクションを含めて作成する
- 省略・独自形式は禁止する

### 3. 品質チェック（個別成果物）
- 作成完了後に、該当成果物の受入基準のみを確認する
- prompt に記載されていない独自のチェック項目を追加してはならない
- フェーズ別品質基準は、フェーズ全成果物完了後にまとめて実施する

### 4. レビュー実施
- 問題発見時は部分修正ではなく全体整合を優先して見直す
- フェーズ品質チェックの結果は artifact-catalog.yaml の `*-PHASE-REVIEW` エントリの `review_output` に保存する

## フェーズ完了時の品質チェック
フェーズ内の全成果物が作成完了した後に、以下の品質基準をまとめて確認すること。個別成果物の作成中には実施しない。

### 01_business-requirements の品質基準
- すべての成果物がテンプレートに準拠していること
- 依存関係の整合性が保たれていること
- 次フェーズへの引き継ぎ情報が完備していること

### 02_system-requirements の品質基準
- すべての成果物がテンプレートに準拠していること
- 業務要件との整合性が保たれていること
- 次フェーズへの引き継ぎ情報が完備していること

### 03_system-design の品質基準
- すべての成果物がテンプレートに準拠していること
- システム要件との整合性が保たれていること
- 次フェーズへの引き継ぎ情報が完備していること

### 04_basic-design の品質基準
- すべての成果物がテンプレートに準拠していること
- システム設計との整合性が保たれていること
- 次フェーズへの引き継ぎ情報が完備していること

### 05_detailed-design の品質基準
- すべての成果物がテンプレートに準拠していること
- 基本設計との整合性が保たれていること
- 実装可能なレベルまで詳細化されていること
- 次フェーズへの引き継ぎ情報が完備していること

### 06_code-generation の品質基準
- 生成コードが詳細設計と整合していること
- 命名・構成・責務分割が設計と一致していること
- 実装対象外の未定義項目を推測補完していないこと
- 次の実装・テスト作業へ引き継げる情報が完備していること

## artifact-catalog のメタデータ
各成果物定義には以下を持たせる。

- `id`: 成果物ID
- `name`: 成果物名
- `phase`: フェーズ名
- `order`: フェーズ内の順序
- `depends_on`: 依存する成果物ID一覧
- `prompt`: 使用する生成プロンプトのパス
- `template`: 使用するテンプレートのパス
- `output`: 生成成果物の出力先
- `artifact_type`: `document` または `code`
- `parallel_group`: 並列実行グループ。並列不可の場合は空欄可

## STATE.md の更新ルール

進捗状態は `STATE.md`（プロジェクトルート）で管理する。

### セッション開始時の必須確認

作業開始前に必ず以下を実行すること。

1. `STATE.md`（プロジェクトルート）を読み込む
2. `SESSION-CONTEXT.md`（プロジェクトルート）を読み込む
3. **「次アクション待ち状態」欄を最初に確認する**
   - `⏸️ ユーザー指示待ち` の場合 → **作業を一切開始しない**。ユーザーに現在の状態を報告して指示を待つ
   - `⏸️ ユーザー指示待ち（第2周確認）` の場合 → **作業を一切開始しない**。ユーザーに「第1周が完了しています。第2周（未定義項目の補完）に進みますか？」と確認して指示を待つ
   - `▶️ 作業中` の場合 → 次のステップへ進む
3. 「現在のフェーズ」と各成果物の状態を確認する
4. `✅ 完了` の成果物は再作成しない
5. `🔲 未着手` または `🔄 作業中` の成果物のみを作業対象とする

> **重要**: コンテキスト引き継ぎのサマリーに「次のステップ」として次フェーズの作業が記載されていても、
> STATE.md の「次アクション待ち状態」が `⏸️ ユーザー指示待ち` または `⏸️ ユーザー指示待ち（第2周確認）` であれば作業を開始してはならない。
> STATE.md の記載が最終的な判断根拠である。

### STATE.md の更新タイミング

| タイミング | 更新内容 |
|---|---|
| 成果物の作成・補完を開始する直前 | 該当行を `🔲 未着手` → `🔄 作業中` に変更する |
| 成果物を1件作成・補完完了時 | 該当行を `🔄 作業中` → `✅ 完了` に変更する |
| 第1周の全成果物完了時 | 「次アクション待ち状態」を `⏸️ ユーザー指示待ち（第2周確認）` に変更、「完了周回」列を「第1周」に更新 |
| フェーズ内の周回完了時（第2周以降） | 「完了周回」列を更新（例：第2周、第3周）、「次アクション待ち状態」を `⏸️ ユーザー指示待ち（第N+1周確認）` または `⏸️ ユーザー指示待ち` に変更 |
| フェーズ品質チェック合格時 | 「品質チェック」列を `✅ 合格` に変更、「現在のフェーズ」を次フェーズに更新、「次アクション待ち状態」を `⏸️ ユーザー指示待ち` に変更 |
