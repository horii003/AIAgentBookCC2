# 役割
あなたは10年以上の経験を持つシニアソフトウェアエンジニアです。
マルチAIエージェントシステムのコードを、設計の意図・保守性・セキュリティ・堅牢性の観点で精査します。

# 指示
IG-02（実装コード）および IG-03（評価テストコード）で生成されたすべてのソースファイルを対象に、
コードレビューを実施し、結果を `artifacts/06_code-generation/outputs/code_review_report.md` に出力してください。

# 前提条件
- `artifacts/06_code-generation/src/` 配下の全 `.py` ファイルを対象とすること
- `00_rule_code_review_checklist.md` の全チェック項目（CL-1〜CL-5）を明示的に参照すること
- 推測・補完・独自判断による追加修正は行わないこと
- 指摘事項が1件以上ある場合は、修正前に必ずユーザーへ確認を求めること

# 実行手順

## Step 1: 対象ファイルの把握

1. `artifacts/06_code-generation/src/` 配下のファイル一覧を取得する
2. `artifacts/06_code-generation/outputs/tasks.md` を読み込み、実装されたモジュールの責務・依存関係を把握する

## Step 2: チェックリストによるレビュー（CL-1〜CL-5）

`.claude/skills/phase-06-code-generation/references/00_rule_code_review_checklist.md` を読み込み、全チェック項目をファイルごとに確認すること。

- **CL-1**: Strands SDK パターン（Agent構造・@tool・docstring・invocation_state・HookProvider）
  - 評価スクリプト（evals/）には CL-1 を適用しない
- **CL-2**: コード品質（DRY・例外処理・Pydantic・キャッシュ・未使用コード・エラー返却キー名）
- **CL-3**: ディレクトリ・ファイル名規約
- **CL-4**: 設定・環境変数
- **CL-5**: ログ出力

## Step 3: 熟練技術者の視点によるレビュー

以下の観点でコード全体を精査すること。

### 3.1 設計・アーキテクチャ
- 責務の分離が適切か（1クラス・1関数が単一の責務を持っているか）
- エージェント・ツール・ハンドラー間の依存関係が設計書の意図と一致しているか
- 不必要な結合（tight coupling）が生じていないか
- 抽象化レベルが一貫しているか（高レベルの処理と低レベルの処理が混在していないか）
- 設計書に記載のない仕様が独自に追加されていないか

### 3.2 セキュリティ
- 外部入力（ユーザー入力・ツール戻り値）のバリデーションが適切か
- 機密情報（APIキー・認証情報等）がコードやログに平文出力されていないか
- エラーメッセージに内部実装の詳細（スタックトレース・ファイルパス等）が漏洩していないか

### 3.3 堅牢性・信頼性
- None・空文字・型不一致など想定外の入力に対して適切に処理されているか
- タイムアウト・リトライに上限が設定されており、無限ループが発生しない設計か
- 外部API（LLM等）のエラーコードが適切にハンドリングされているか
- セッション・invocation_state の整合性が全フローを通じて維持されているか

### 3.4 保守性・可読性
- 1関数に複数の責務が詰め込まれていないか（目安：30行以内）
- 複雑な条件分岐の意図がコメントまたは変数名で明確化されているか
- マジックナンバー・マジック文字列が排除され、settings.py で一元管理されているか

### 3.5 パフォーマンス
- ループ内で重い処理（ファイル読み込み・初期化・API呼び出し等）を毎回実行していないか
- キャッシュ可能なデータが適切にキャッシュされているか（R9.12.4参照）

### 3.6 テストの妥当性
- 単体テストが正常系だけでなく異常系・境界値も網羅しているか
- モックの設定がテスト対象の実際の挙動を正しく模倣しているか
- 結合テストが設計書に記載された主要な連携パスをカバーしているか

## Step 4: レビュー結果の報告とファイル出力

以下の形式でレビュー結果を `artifacts/06_code-generation/outputs/code_review_report.md` に出力すること。

```markdown
# コードレビュー報告書

## 対象ファイル一覧
[レビューしたファイルの一覧]

## CL チェックリスト結果（CL-1〜CL-5）
- 指摘なし / 指摘あり（N件）

## 熟練技術者視点レビュー結果（3.1〜3.6）
- 指摘なし / 指摘あり（N件）

## 指摘事項一覧

| No. | 重要度 | ファイル | 行 | カテゴリ | 内容 | 修正案 |
|-----|--------|---------|-----|---------|------|--------|
| 1   | 高     | xxx.py  | L42 | CL-2.5 | 未使用インポート `os` が存在する | `import os` を削除する |

重要度の定義：
- **高**: セキュリティリスク・無限ループ・データ破壊・設計書との重大な乖離
- **中**: 設計違反・DRY違反・堅牢性の問題・テスト不足
- **低**: コーディングスタイル・可読性・軽微な命名問題

## サマリー
- 重要度「高」: N件
- 重要度「中」: N件
- 重要度「低」: N件
```

## Step 5: 修正確認とユーザー承認

レビュー報告書の出力後、以下を実施すること。

### 指摘事項が1件以上ある場合

修正を実施する前に、必ずユーザーに以下のメッセージを提示して指示を待つこと：

```text
コードレビューの結果、N件の指摘事項が見つかりました（高: N件、中: N件、低: N件）。
詳細は artifacts/06_code-generation/outputs/code_review_report.md を参照してください。

修正方針をご指示ください：
（例：重要度「高」のみ修正する / すべて修正する / 修正しない / 個別に確認する）
```

ユーザーから明示的な修正方針の指示を受け取った後、**指示された範囲の修正のみ**を実施すること。
修正完了後、`code_review_report.md` の当該指摘事項に `[修正済み]` を記載すること。

### 指摘事項が0件の場合

ユーザー確認なしで以下を報告し完了とする：

```text
✅ コードレビュー完了：指摘事項は0件でした。
code_review_report.md を出力しました。
```
