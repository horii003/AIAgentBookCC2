# 役割
あなたは社内で稼働するマルチAIエージェントシステムの実装を担当する開発者です。

# 指示
規約と詳細設計書、スケルトンコードを基に、実装の計画を立てるためのtasks.mdを作成してください。

# 前提条件
- 規約（`.claude/skills/phase-06-code-generation/references/`配下の規約ファイル）の内容を遵守すること
- 詳細設計書の仕様に厳密に従うこと
- スケルトンコードの構造・パターンを踏襲すること
- 解説や補足説明は記載しないこと
- タスクの順序はスケルトンコードの番号順（依存関係順）に従うこと

# 入力データ

以下の3つのソースからすべてのファイルを読み込み、各タスクに必要な入力ファイルをAI自身が判断・選定すること。

## 規約
- ファイル：.claude/skills/phase-06-code-generation/references/00_rule_directory_structure.md、.claude/skills/phase-06-code-generation/references/00_rule_project_conventions.md
- 規約ファイルを読み込むこと
- コーディング規約・命名規則・ディレクトリ構造等、実装に影響するルールを把握すること

## 設計書
- 詳細設計書：artifacts/05_detailed-design/outputs/ 配下の全ファイル
- 上記ディレクトリ配下のすべてのファイルを読み込むこと

## スケルトンコード
- ディレクトリ：.claude/skills/phase-06-code-generation/assets/
- このディレクトリ配下のすべてのファイル（01～17）を読み込むこと

## 入力ファイル選定ルール
- 各スケルトンコードの内容（概要・参照先・ファイル配置先）を読み取り、実装に必要な設計書を特定すること
- スケルトンコードが参照するモデル・方針・設計に対応する設計書ファイルを紐付けること
- 規約は全タスク共通の参照元として扱うこと
- 選定結果は各タスクの「参照する設計書」欄に具体的なファイルパスで記載すること

## ディレクトリ構造検証タスク

全実装タスクの後、結合テストの前に、以下の形式でディレクトリ構造検証タスクを1つ記載すること：

````markdown
## タスク[番号]: ディレクトリ構造検証

- **ステータス**: [ ] 未着手
- **参照する規約**: `.claude/skills/phase-06-code-generation/references/00_rule_directory_structure.md`
- **検証対象ディレクトリ**: `artifacts/06_code-generation/src/`
- **検証内容**:
  - R1準拠確認: `src/` 直下のディレクトリが許可リスト（config/, interfaces/, models/, agents/, handlers/, tools/, prompt/, knowledge/, session/, storage/, data/, template/, sample/, output/, logs/, evals/, docs/, tests/）のみであること
  - R2準拠確認: 各ファイルが正しいディレクトリに配置され、命名規則に従っていること（例: `*_agent.py` → `agents/`、`*_tools.py` または `*_generator.py` → `tools/`、`*_policies.py` → `knowledge/`、`prompt_*.py` → `prompt/`）
- **違反時の報告形式**:
  - `⚠️ [違反種別]: [ファイルパス] → [修正方法]`
- **違反時の対応**: 検出された違反をすべて修正してからステータスを完了にすること
````

### 結合テストの観点
- 複数モジュールを組み合わせた連携動作を検証するテストファイルを作成する

## タスク分割の方針
- 1つのスケルトンコードに対して1タスクを基本とすること
- エージェントやツールなど複数ファイルが生成されるものは、個別のサブタスクとして記載すること
- 依存関係がある場合はタスクの説明に明記すること
- 全実装タスクの後に結合テストタスクを必ず1つ以上配置すること
- 全実装タスクの後、結合テストタスクの前に、ディレクトリ構造検証タスクを必ず1つ配置すること
- スケルトン15・16・17の依存関係と推奨実装順：
  - スケルトン15（settings.py）はスケルトン02（model_config）の直後に実装する（スケルトン17の依存元）
  - スケルトン16（console_approval_adapter.py）はスケルトン05（human_approval_hook）の前に実装する（human_approval_hookがimportするため）
  - スケルトン17（base_agent.py）はスケルトン04（loop_control_hook）・スケルトン05（human_approval_hook）の後、スケルトン11・12（orchestrator/specialist_agent）の前に実装する

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- すべてのスケルトンコード（01～17）に対応するタスクが存在すること
- 各タスクに参照する設計書ファイル名が記載されていること
- 各タスクに参照するスケルトンコードファイル名が記載されていること
- 各タスクに成果物のファイルパスが記載されていること
- 各タスクに実装内容が具体的に記載されていること
- 各タスクにテスト内容が記載されていること
- タスクの順序が依存関係を考慮した順序になっていること
- 設計書のファイルパスがartifacts/配下の実際のパスであること（テンプレートパスを転記していないこと）
- 各タスクの「参照する設計書」が、スケルトンコードの内容と整合していること（無関係な設計書を参照していないこと）
- 規約ファイル（.claude/rules/配下）が全タスク共通の参照元として記載されていること
- ディレクトリ構造検証タスクが、全実装タスクの後・結合テストタスクの前に1つ存在すること
