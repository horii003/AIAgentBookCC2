# 役割
あなたは社内で稼働するマルチAIエージェントシステムの実装を担当する開発者です。

# 指示
tasks.mdに記載された各タスクを上から順に実行し、実装コードを生成してください。

# 前提条件
- tasks.mdのタスク順序に従い、上から順に1タスクずつ処理すること
- 各タスクの「参照する設計書」「参照するスケルトンコード」を必ず読み込んでから実装すること
- 規約（`.claude/skills/phase-06-code-generation/references/`配下の規約ファイル）を遵守すること
- スケルトンコードの構造・パターンを踏襲すること
- 設計書の仕様に厳密に従うこと
- 設計書に記載のない仕様を独自に追加しないこと
- 判断できない内容はコメントで `# TODO: 要件上未定義` と記載すること
- タスク完了ごとにtasks.mdのステータスを `[x] 完了` に更新すること

- artifacts/06_code-generation/outputs/tasks.md

## 規約
- `.claude/skills/phase-06-code-generation/references/` 配下の規約ファイル（00_rule_*.md）

- tasks.mdの各タスクに記載された「参照する設計書」および「参照するスケルトンコード」のファイルを読み込むこと
- 各タスクで必要なファイルのみを読み込めばよい（全ファイル一括読み込みは不要）

# 実行手順

**テストを実行せずにファイル生成のみで完了としてはならない。**
**各タスクは、単体テストが全件パスするまで完了にしてはならない。**

各タスクに対して以下の手順を繰り返すこと：

1. tasks.mdから次の未着手タスクを確認する
2. タスクに記載された「参照するスケルトンコード」を読み込む
3. タスクに記載された「参照する設計書」を読み込む
4. スケルトンコードの構造に従い、設計書の仕様を反映した実装コードを生成する
5. 「成果物のファイルパス」に指定されたパスにファイルを出力する
6. タスクの「単体テスト内容」に基づき単体テストコードを生成する
7. 「単体テストコードのファイルパス」に指定されたパスにテストコードを出力する
8. 単体テストをコマンドで実行する（例: `pytest tests/unit/test_xxx.py -v`）
9. テスト結果を確認する:
   - 全件パス → 手順11へ進む
   - 失敗あり → 手順10へ進む
10. 失敗したテストの原因を特定し、実装コードまたはテストコードを修正する。手順8に戻り再実行する。全件パスするまで繰り返すこと。
11. 全テスト通過後、tasks.mdの当該タスクのステータスを `[x] 完了` に更新する
12. 次のタスクへ進む

## 全実装タスク完了後のディレクトリ構造検証

ディレクトリ構造検証も実行せずに完了としてはならない。

すべての実装タスクが完了した後、結合テストの前に以下を実施すること：

1. tasks.mdのディレクトリ構造検証タスクを確認する
2. `artifacts/06_code-generation/src/` 直下のディレクトリ一覧を取得する
3. `.claude/skills/phase-06-code-generation/references/00_rule_directory_structure.md` のR1で定義された許可ディレクトリ（config/, interfaces/, models/, agents/, handlers/, tools/, prompt/, knowledge/, session/, storage/, data/, template/, sample/, output/, logs/, evals/, docs/, tests/）と照合し、許可外ディレクトリが存在しないか確認する
4. `src/` 配下の全ファイルを走査し、R2の命名規則・配置ルールに従って正しいディレクトリに配置されているか確認する（例: `*_agent.py` が `agents/` 以外にある場合は違反）
5. 違反を検出した場合:
   - 以下の形式で報告する: `⚠️ [違反種別]: [ファイルパス] → [修正方法]`
   - 報告されたすべての違反を修正する（ファイル移動・リネーム・import文の更新を含む）
   - 手順2に戻り、違反がなくなるまで繰り返す
6. 違反がない場合、tasks.mdのディレクトリ構造検証タスクのステータスを `[x] 完了` に更新する

## 全タスク完了後の結合テスト

結合テストも実行せずにファイル生成のみで完了としてはならない。

すべての実装タスクが完了した後に以下を実施すること：

1. tasks.mdの結合テストタスクを確認する
2. 結合テストコードを生成し、指定されたパスに出力する
3. 結合テストをコマンドで実行する（例: `pytest tests/integration/ -v`）
4. テスト結果を確認する:
   - 全件パス → 手順6へ進む
   - 失敗あり → 手順5へ進む
5. 失敗したテストの原因を特定し、実装コードまたはテストコードを修正する。手順3に戻り再実行する。全件パスするまで繰り返すこと。
6. 全テスト通過後、tasks.mdの結合テストタスクのステータスを `[x] 完了` に更新する

## テストに関するルール
- 単体テストは artifacts/06_code-generation/src/tests/unit/ に配置すること
- 結合テストは artifacts/06_code-generation/src/tests/integration/ に配置すること
- テストコードも規約（命名規則・コーディング規約）に準拠すること
- テスト対象の設計書に記載されたテスト観点を網羅すること
- 外部サービス（LLM API等）への実際の呼び出しはモックを使用すること

# 出力形式
- 各タスクの「成果物のファイルパス」に指定されたパスにPythonファイル（またはデータファイル）を出力すること
- テストコードは tests/unit/ および tests/integration/ に出力すること
- ファイルの文字コードはUTF-8とすること

# 受入基準
作成後に以下の受入基準を満たしているか確認すること。
- tasks.mdのすべてのタスクが `[x] 完了` になっていること
- ディレクトリ構造検証タスクが完了しており、`src/` 配下にR1違反ディレクトリおよびR2違反ファイルが存在しないこと
- 各タスクの「成果物のファイルパス」にファイルが存在すること
- 各タスクに対応する単体テストコードが「単体テストコードのファイルパス」に存在すること
- すべての単体テストをコマンドで実行し、全件パスしていること（ファイル生成のみは不可）
- 結合テストコードが「結合テストコードのファイルパス」に存在すること
- すべての結合テストをコマンドで実行し、全件パスしていること（ファイル生成のみは不可）
- 生成されたコードがスケルトンコードの構造・パターンに準拠していること
- 生成されたコードが設計書の仕様に準拠していること
- 規約（命名規則・ディレクトリ構造・コーディング規約）に準拠していること
- 設計書に記載のない仕様が独自に追加されていないこと
- `# TODO: 要件上未定義` のコメントが、設計書で未定義の箇所にのみ付与されていること
