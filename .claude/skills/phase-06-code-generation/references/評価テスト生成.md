# 役割
あなたは Strands Agents SDK で構築したエージェントの品質を自動評価するスクリプトの実装を担当する開発者です。

# 指示
評価設計書（`artifacts/05_detailed-design/outputs/評価テスト詳細設計.md`）を読み込み、テンプレート（`18_eval_test.md`）の操作手順に従って `evals/eval_{evaluation_name}.py` を生成してください。

# 前提条件
- 設計書2件を必ず読み込んでから生成すること
- テンプレートの操作手順（Step 1〜4）を順番に適用すること
- 設計書に記載のない仕様を独自に追加しないこと
- 初期設定の順序（[1]〜[7]）は変更禁止
- `memory_exporter.clear()` は各ケース実行前に必ず記述すること

# 実行手順

## Step 1: プレースホルダーの置き換え

設計書§1（ファイル概要）・§2（モジュール構成）を読み、以下を確定させる。

| プレースホルダー | 読み取り元 | 確定する値 |
|---|---|---|
| `{evaluation_name}` | 設計書タイトル・対象ファイル名 | スネークケースの評価名 |
| `{EvaluatorClass}` | 設計書§1.2 評価方式テーブル「評価器」 | 評価器クラス名 |
| `{evaluation_title}` | 設計書§7.1 main 処理フロー [1] | コンソール表示タイトル文字列 |
| `{evaluation_description}` | 設計書§1.1 目的 + §1.2 評価レベル説明 | docstring 本文 |
| `{agent_factory}` | 設計書§2.1 インポート一覧「helpers.*」 | エージェント生成関数名 |
| `{metadata_judge_key}` | 設計書§5.1 Case オブジェクトの metadata キー | `"goal"` または `"expected_tool"` |

マルチターン評価（GoalSuccessRateEvaluator）の場合は `{metadata_judge_key}` = `"goal"`、シングルターン評価（ToolSelectionAccuracyEvaluator）の場合は `"expected_tool"` を使用する。

## Step 2: 評価タイプの判定と差分適用

設計書§1.2「実行方式」を確認して評価タイプを判定する。

### シングルターン評価の場合（例: ToolSelectionAccuracyEvaluator）

テンプレートの「シングルターン評価の差分」セクションを適用する：

**変更1（import）**: `run_actor_conversation` を除去する
```python
# run_actor_conversation をインポートリストから削除する
from helpers import (
    patch_human_approval_hook, {agent_factory},
    memory_exporter, get_model, create_invocation_state,
)
```

**変更2（run_eval_task）**: マルチターン版の `run_actor_conversation` 呼び出しブロックを以下に差し替える
```python
result = agent(case.input, invocation_state=state.model_dump())
response = str(result)
logger.info("ケース '%s': response='%s'", case.name, response[:100])
```
そして戻り値を `{"output": response, "trajectory": session}` に変更する。

**変更3（metadata）**: EVAL_CASES の metadata キーを `"goal"` → `"expected_tool"` に変更する。

### マルチターン評価の場合（例: GoalSuccessRateEvaluator）

差分適用なし。マルチターン版スケルトンをそのまま使用する。

## Step 3: テストケースの記入

設計書§5.2「ケース一覧」を読み、各ケースを `EVAL_CASES` に追記する。

各ケースについて以下のフィールドを設計書から読み取る：

| フィールド | 設計書の読み取り元 |
|---|---|
| `name` | ケーステーブル `name` 欄 |
| `input` | ケーステーブル `input` 欄 |
| `task_description` | ケーステーブル `task_description` 欄 |
| `{metadata_judge_key}` の値 | ケーステーブル `goal` または `expected_tool` 欄 |

テンプレートの1ケーススロット（`{case_name_1}` 等）を1件目で埋め、2件目以降は同じ形式で `# TODO:` の前に追記する。

## Step 4: ログファイル名・レポートファイル名の確定

設計書§4（ログ設定）・§8（出力ファイル仕様）を読み、以下を確認する：

| 箇所 | 確定する値の例 |
|---|---|
| FileHandler のファイル名 | `eval_{evaluation_name}.log` |
| レポート保存パス | `eval_{evaluation_name}_report.json` |

スケルトン内の `{evaluation_name}` プレースホルダーが Step 1 の値に置き換わっていることを確認する。

# 受入基準

生成後に以下を満たしていることを確認すること：

- `evals/eval_{evaluation_name}.py` が出力されていること
- 初期設定が `eval_common_system_design.md` §6.1 の必須順序（[1]〜[7]）に準拠していること
- `memory_exporter.clear()` が `run_eval_task` 内の先頭付近（エージェント生成より前）に記述されていること
- `EVAL_CASES` に設計書§5.2 記載のすべてのテストケースが含まれていること
- 評価タイプ（シングルターン / マルチターン）が設計書§1.2 と一致していること
- `ensure_ascii=False, indent=2` でレポートが JSON 保存されていること
- プレースホルダー（`{...}` 形式）がコード内に残っていないこと
