# フェーズ01: 業務要件定義 実行スキル

このフェーズのスキルが担当する成果物: **BR-01〜BR-07**（業務要件一覧、業務一覧、役割分担定義、業務ルール定義_判断基準定義、業務プロセス定義、業務データ定義、ユースケース定義）

---

## Section 1: 起動時必須確認

### 1-1. 状態ファイルのコピーと読み込み

フェーズと状態管理を目的として、以下の2つのファイルをまずプロジェクトルートへコピーする。

1. `.claude/skills/phase-01-business-requirements/assets/STATE.md`をプロジェクトルートへコピーする。


以下のファイルを必ず読み込む。

1. `STATE.md`（プロジェクトルート）

### 1-2. 「次アクション待ち状態」の確認

- `⏸️ ユーザー指示待ち` または `⏸️ ユーザー指示待ち（第2周確認）` の場合 → **即停止**。現在の状態をユーザーに報告して作業を開始しない。
- `▶️ 作業中` の場合 → 次ステップへ進む。

### 1-3. フェーズ確認

`STATE.md` の「現在のフェーズ」が `01_business-requirements` 以外の場合は、以下を報告して停止する。

```
⚠️ フェーズ不一致
このスキルは 01_business-requirements 用です。
現在のフェーズは [現在のフェーズ] です。
正しいフェーズスキルを使用してください。
```

### 1-4. artifact-catalog.yaml の読み込み

`.claude/rules/artifact-catalog.yaml` を読み込み、`phase: 01_business-requirements` の成果物定義を確認する。

### 1-5. 作業対象の特定

`STATE.md` の成果物別進捗から `🔲 未着手` または `🔄 作業中` の成果物のみを作業対象とする。`✅ 完了` の成果物は再作成しない。

## Section 2: 依存チェック

各成果物の `depends_on` に記載されたファイルを確認する。

依存ファイルに以下のパターンが含まれる場合は **即座に停止** して報告する。

- `要件上未定義`
- `〇〇フェーズで定義`
- `TBD`

報告形式:
```
⚠️ 作業開始不可：依存成果物に未定義項目があります

| ファイル | 項目 | 記載内容 |
|---|---|---|
| [ファイル名] | [項目名] | [記載内容] |

上記を定義・解決してから再度依頼してください。
未定義のまま進める場合は、その旨を明示的に指示してください。
```

**第1周ルール（例外）**: `depends_on` に記載されたファイルが同フェーズ内の成果物であり、かつその成果物がまだ未作成の場合は、ユーザー確認なしで作業を続行してよい。存在しない依存ファイルから参照すべき情報は `要件上未定義` として記載する。

---

## Section 3: 成果物作成手順

`artifact-catalog.yaml` の `order` 順（BR-01, BR-02, BR-03, BR-04, BR-05, BR-06, BR-07）に従って処理する。

**各成果物について以下の手順を実施する（一括読み込み禁止）**:

1. `STATE.md` の該当行を `🔄 作業中` に更新する
2. `artifact-catalog.yaml` の該当成果物の `prompt` パスを確認し、そのファイルを Read する（遅延ロード）
3. `artifact-catalog.yaml` の該当成果物の `template` パスを確認し、そのファイルを Read する（遅延ロード）
4. プロンプトの指示に厳密に従い、テンプレートの全セクション構造を維持して成果物を生成する
5. `artifact-catalog.yaml` の `output` パスに成果物ファイルを Write する
6. プロンプト内の「受入基準」に記載されたチェック項目のみで個別品質チェックを実施する（独自チェック項目の追加禁止）
7. `STATE.md` の該当行を `✅ 完了` に更新する

**禁止事項**:
- フェーズ内全成果物のプロンプト/テンプレートを一括で読み込んではならない
- テンプレートに定義されていない解説・補足・注釈を本文に混在させてはならない

## Section 4: 複数周回ルール

フェーズ内の全成果物（BR-01〜BR-07）の第1周作成が完了したら、以下の2条件を確認する。

**条件1**: フェーズ内の成果物が、同フェーズ内の別の成果物を `depends_on` に持つか
**条件2**: その依存先成果物が、1周目の作成時点でまだ未作成だった（`要件上未定義` として記載した箇所が存在する）か

- **両条件を満たす場合** → `STATE.md` の「次アクション待ち状態」を `⏸️ ユーザー指示待ち（第2周確認）` に更新し、以下を表示して停止する:
  ```
  ✅ 第1周完了：フェーズ内の全成果物を一巡しました。
  第2周（未定義項目の補完）に進みますか？
  ```
  「完了周回」列を「第1周」に更新する。

- **両条件を満たさない場合** → 確認メッセージを出さずに Section 5（フェーズ品質チェック）へ直接進む。

**第2周以降**: ユーザーが「第2周に進む」と明示的に指示した場合のみ開始する。`要件上未定義` とした箇所を依存成果物の内容をもとに補完し、成果物間の整合性チェックを実施する。

---

## Section 5: フェーズ品質チェック

フェーズ内の全成果物が完了した後に、以下の品質基準を確認する（個別成果物作成中には実施しない）。

**01_business-requirements の品質基準**:
- すべての成果物がテンプレートに準拠していること
- 依存関係の整合性が保たれていること
- 次フェーズへの引き継ぎ情報が完備していること

品質チェック結果を `artifact-catalog.yaml` の `phase: 01_business-requirements` かつ `artifact_type: review` のエントリの `review_output` パスに Write する。

**合格時**: `STATE.md` を以下のように更新する:
- 「品質チェック」列: `✅ 合格`
- 「現在のフェーズ」: `02_system-requirements`
- 「次アクション待ち状態」: `⏸️ ユーザー指示待ち`

以下のメッセージを出力して**停止**する:
```
✅ 01_business-requirements が完了しました。
次フェーズ（02_system-requirements）を開始する場合は、その旨を指示してください。
```

## Section 6: 禁止事項

- 次フェーズ（02_system-requirements）への自動進行禁止
- `.claude/rules/` 配下ファイルの変更禁止（ユーザー明示指示がある場合のみ可）
- `要件上未定義` 項目の推測・補完・暫定値設定禁止
- フェーズ内全成果物のプロンプト/テンプレートの一括読み込み禁止
- すべての出力は日本語で記述する（コード識別子のみ英語）
- `一般的にはこの値` などの独自判断での値設定禁止
- 不明点はユーザーに確認する。推測や補完を行わない
